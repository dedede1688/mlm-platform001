## 一、项目全景扫描

### 1.1 业务模型

本项目为**多级分销电商平台**，采用"三三复制 + 双轨关系"模式。

**会员体系（8 级）**

| 等级 | 名称 | 代码 | 升级条件 |
|------|------|------|---------|
| 0 | 游客 | VISITOR | — |
| 1 | 会员 | MEMBER | 购买升级产品 |
| 2 | 经销商 | DISTRIBUTOR | 直推 1 个会员 |
| 3 | 主任 | DIRECTOR | 直推 2 个经销商 |
| 4 | 经理 | MANAGER | 团队销售额达标 |
| 5 | 总监 | SUPERVISOR | 团队销售额达标 |
| 6 | 总裁 | PRESIDENT | 团队销售额达标 |
| 7 | 董事 | BOARD | 团队销售额达标 |

**双轨关系**
- **直推关系（referrerId）**：永久绑定，谁推荐的就是谁的，永不改变
- **安置关系（parentId）**：三三复制滑落，决定品牌管理奖层级

**三大奖励系统**

| 奖励 | 比例 | 触发条件 | 发放方式 |
|------|------|---------|---------|
| 直推奖 | 20% | 被推荐人下单 | 即时（需推荐人已购升级品） |
| 品牌管理奖 | 20% | 安置链上对应层级有经销商 | 按层级轮流（第1-10层循环） |
| 分红奖 | 5%×5级 | 每日快照，周结 | 主任→董事 5 级独立池 |

**积分系统**
- 购买升级产品获积分
- 每日解锁 1%，100 天释放完毕
- 可转赠、购物抵扣
- 积分记录：earn → unlock → use / transfer_in / transfer_out / void

**资金底座（v43-7 重构后）**

| 字段 | 含义 | 来源 |
|------|------|------|
| balance | 可用余额 | 充值、退款 |
| frozenBalance | 冻结余额 | 提现申请时冻结 |
| consumeBalance | 消费余额 | 充值+退款（可消费不可提现） |
| earningsPending | 待结算收益 | 奖励/分红（未到结算周期） |
| earningsAvailable | 可提现收益 | 奖励/分红（已结算） |
| earningsVoided | 累计作废 | 退款扣回不足部分 |
| earningsFrozen | 收益冻结 | 提现申请时冻结 |

**订单状态机**

```
pending → paid → shipped → completed
    ↓        ↓
 cancelled  refunded（经 refund_requests 审核）
```

### 1.2 技术架构

| 层 | 技术 | 说明 |
|---|------|------|
| 前端 | Next.js 15 + React 19 + Tailwind CSS | App Router，47 个页面 |
| 后端 | Next.js API Routes | 99 个路由文件 |
| 数据库 | PostgreSQL + Prisma | 26 个模型，Supabase 托管 |

| 部署 | Vercel | 自动从 main 分支部署 |

### 1.3 安全体系

```
请求 → middleware（JWT 签名验证 + 角色检查）
  ├── /api/admin/* → 完整角色校验（super_admin / goods_admin / support_admin / finance_admin / auditor / points_admin）
  ├── 用户侧受保护路径 → 仅 JWT 登录校验
  └── 公开路径 → 直接放行

API 路由内部 → verifyPermission 再次验证（最终防线）
```

**已配置安全措施**
- middleware 真正验证 JWT 签名（v55.2 修复）
- Supabase Data API 已封锁生产业务表直访
- 支付密码 bcryptjs 哈希
- 提现需支付密码验证
- rate-limit 防暴力支付
- cron secret 防恶意触发
