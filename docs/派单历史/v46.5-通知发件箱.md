## 决策记录

胡子哥 v46.4 反馈："通知可以发送，但发完后后台有没有留存？发到哪里去了？有哪些人接到了？有哪些人看了？"

**现状**（v46.4）：
- 手动发送完就完事，**没**批次记录
- 业务触发（订单/提现）的通知**也**没区分
- `prisma.notification` 表只有 `userId`，要 JOIN user 才知道发给谁
- `isRead` 字段有，但**没**总数/已读率统计

**v46.5 方案**：
- 新增 `NotificationBatch` 表（发件箱）：每次发送 = 1 个 batch
- `Notification` 表加 `batchId` + `senderId` 2 个字段
- 业务触发的通知**也**入 batch（`sourceType='business'` 区分）
- 新页面 `/admin/notification-history`：发件箱列表 + 收件人明细
- admin 菜单加「发件箱」入口

**已知 trade-off**：
1. 历史通知（v46.4 已写入的 5+ 条）**不**补 batchId（没法补）——前端**不**显示无 batch 的通知
2. 系统公告全员发（`prisma.user.findMany`）在用户量 10w+ 时可能慢——胡子哥当前 < 1k 用户**不**是问题
3. `sendInApp` 加 `batchId` 参数，**不**影响现有 email/sms 通道（铁律 5：不破坏已稳定 API）

model Notification {

### 1.2 新增 `NotificationBatch` model

```prisma
model NotificationBatch {
  id              String    @id @default(uuid())
  type            String    @map("type")              // general / announcement / business
  subject         String?   @map("subject")
  content         String    @map("content")
  senderId        String?   @map("sender_id")         // null = 系统自动（业务触发）
  sourceType      String?   @map("source_type")       // 业务触发的具体类型，如 order_paid / withdrawal_rejected
  sourceId        String?   @map("source_id")         // 业务触发的源 ID
  recipientCount  Int       @default(0) @map("recipient_count")
  readCount       Int       @default(0) @map("read_count")
  createdAt       DateTime  @default(now()) @map("created_at")

  sender          User?     @relation("BatchSender", fields: [senderId], references: [id], onDelete: SetNull)
  notifications   Notification[]

  @@index([type])
  @@index([senderId])
  @@index([createdAt])
  @@index([sourceType])                              // 业务触发查找
  @@map("notification_batches")
}
```

### 1.3 命名约束

- 表名：`notification_batches`（snake_case 复数）
- 字段名：`batch_id` / `sender_id` / `recipient_count` / `read_count` / `source_type` / `source_id`
- **不要**用驼峰

### 2.2 改 `src/lib/services/order.service.ts`

**位置**：4 处 `sendInApp` 调用，line 202 / 236 / 257 / 451

**改造**：每个调用前**先创建 batch**，传 batchId。

**4 个位置统一模式**：
```typescript
// 创建业务 batch
const batch = await tx.notificationBatch.create({
  data: {
    type: 'business',
    subject: '订单支付成功',         // 或对应业务标题
    content: '',                    // 业务通知内容由模板渲染，batch 不存完整 content
    sourceType: 'order_paid',       // 或对应业务类型
    sourceId: order.id,
    recipientCount: 1,
  },
}).catch((err) => {
  logger.error('创建通知批次失败', { error: String(err) })
  return null
})

// 然后调 sendInApp 时传 batchId
sendInApp({
  userId: paidOrder.userId,
  templateType: 'order_paid',
  variables: nv,
  batchId: batch?.id,
}).catch(...)
```

**4 处对应**：
| 位置 | templateType | subject | sourceType | sourceId |
|------|--------------|---------|------------|----------|
| line 202 (payOrder) | `order_paid` | 订单支付成功 | `order_paid` | `paidOrder.id` |
| line 236 (shipOrder) | `order_shipped` | 订单已发货 | `order_shipped` | `order.id` |
| line 257 (completeOrder) | `order_completed` | 订单已完成 | `order_completed` | `order.id` |
| line 451 (cancelOrder) | `order_cancelled` | 订单已取消 | `order_cancelled` | `cancelledOrder.id` |

**关键**：
- `tx` 是当前事务（如果业务在事务里）——用 `tx.notificationBatch.create` 而不是 `prisma`
- 如果不在事务里（普通调），用 `prisma.notificationBatch.create`
- `batch = null` 时 `sendInApp` 仍能正常发（不传 batchId）
- **不**改 `sendEmail.ts` / `sendSms.ts`（铁律：email/sms 没发件箱需求）

### 3.2 新增 `src/app/api/admin/notification-history/route.ts`

```typescript
import { NextRequest, NextResponse } from 'next/server'
import { verifyPermission } from '@/lib/utils/admin-auth'
import { prisma } from '@/lib/prisma'

// GET /api/admin/notification-history — 发件箱列表
export async function GET(request: NextRequest) {
  try {
    const { user: admin, error: authError } = await verifyPermission(request, ['admin', 'super_admin'])
    if (authError || !admin) return authError!

    const { searchParams } = new URL(request.url)
    const page = parseInt(searchParams.get('page') || '1')
    const limit = Math.min(parseInt(searchParams.get('limit') || '20'), 100)
    const type = searchParams.get('type') || undefined  // general / announcement / business

    const skip = (page - 1) * limit

    const where = type ? { type } : {}

    const [batches, total] = await Promise.all([
      prisma.notificationBatch.findMany({
        where,
        orderBy: { createdAt: 'desc' },
        skip,
        take: limit,
        include: {
          sender: { select: { id: true, phone: true, nickname: true } },
        },
      }),
      prisma.notificationBatch.count({ where }),
    ])

    return NextResponse.json({
      success: true,
      data: {
        batches,
        pagination: { page, limit, total, totalPages: Math.ceil(total / limit) },
      },
    })
  } catch (error) {
    console.error('获取发件箱列表失败:', error)
    return NextResponse.json({ success: false, error: '获取失败' }, { status: 500 })
  }
}
```

### 3.3 新增 `src/app/api/admin/notification-history/[id]/route.ts`

```typescript
import { NextRequest, NextResponse } from 'next/server'
import { verifyPermission } from '@/lib/utils/admin-auth'
import { prisma } from '@/lib/prisma'

// GET /api/admin/notification-history/[id] — 批次详情（收件人列表）
export async function GET(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  try {
    const { user: admin, error: authError } = await verifyPermission(request, ['admin', 'super_admin'])
    if (authError || !admin) return authError!

    const batch = await prisma.notificationBatch.findUnique({
      where: { id: params.id },
      include: {
        sender: { select: { id: true, phone: true, nickname: true } },
      },
    })

    if (!batch) {
      return NextResponse.json({ success: false, error: '批次不存在' }, { status: 404 })
    }

    const { searchParams } = new URL(request.url)
    const page = parseInt(searchParams.get('page') || '1')
    const limit = Math.min(parseInt(searchParams.get('limit') || '50'), 200)
    const skip = (page - 1) * limit

    const [recipients, total, readCount] = await Promise.all([
      prisma.notification.findMany({
        where: { batchId: params.id },
        orderBy: { createdAt: 'desc' },
        skip,
        take: limit,
        include: {
          user: { select: { id: true, phone: true, nickname: true } },
        },
      }),
      prisma.notification.count({ where: { batchId: params.id } }),
      prisma.notification.count({ where: { batchId: params.id, isRead: true } }),
    ])

    return NextResponse.json({
      success: true,
      data: {
        batch,
        recipients,
        stats: {
          total,
          readCount,
          unreadCount: total - readCount,
          readRate: total > 0 ? Math.round((readCount / total) * 100) : 0,
        },
        pagination: { page, limit, total, totalPages: Math.ceil(total / limit) },
      },
    })
  } catch (error) {
    console.error('获取批次详情失败:', error)
    return NextResponse.json({ success: false, error: '获取失败' }, { status: 500 })
  }
}
```

## 风险点

1. **schema migration 风险**——加字段 nullable 是安全的，加新表也安全。**不**改已有字段类型
2. **历史通知无 batch**——v46.4 写入的 5+ 条通知**不**在发件箱显示（设计如此），胡子哥需知情
3. **业务通知 batch 创建失败时**——用 `.catch` + `null`，**不**影响主业务（订单支付/发货等）
4. **系统公告 readCount 实时性**——`readCount` 字段冗余存，第一次计算后**不**实时更新。前端从 `prisma.notification.count` 现算更准。**建议**：batch 表**不**存 readCount，每次详情页现算
5. **`recipientCount` 必须准确**——`createMany` 返回的 `result.count` 是真实写入数。用这个更新
6. **`Inbox` 图标存在性**——v6 教训：先 `grep "Inbox" lucide-react` 确认存在
