## 0. 任务背景

Batch 2.2.b（commit 27971da）+ Batch 2.2.c（commit 94ab392）已上线，`consumeBalance` 在订单支付/退款能正常累加。

但**奖励相关的 4 字段还是 0**——`earningsAvailable` / `earningsVoided` 永远不变化。

**Batch 2.2.d 目标**：让所有"奖励发放"和"奖励扣回"自动同步到 4 字段。

**写入规则**（v43-7-design.md 章节 8.4 已定）：

| 场景 | 触发位置 | 4 字段变化 |
|------|---------|-----------|
| 直推奖发放 | `reward.service.ts:createReferralReward` | earningsAvailable += amount |
| 品牌管理奖发放 | `reward.service.ts:createBrandBonusReward` | earningsAvailable += amount |
| 分红奖发放（5 池）| `reward.service.ts:createDividendReward` | earningsAvailable += perUserAmount |
| 手动奖励 | `api/admin/manual-reward/route.ts` | earningsAvailable += amount |
| 每日分红结算 | `dividend.service.ts:settleDailyDividends` | earningsAvailable += dividendAmount |
| 退款扣回奖励 | `reward.service.ts:processRefund` (refund_reward) | earningsAvailable -= reward.amount |
| 退款扣回分红 | `reward.service.ts:processRefund` (refund_dividend) | earningsVoided += dividend.amount |

**总计 7 处 user.update 改动**，都在事务内，单行扩展。

**完整设计文档**：`docs/v43-7-design.md` 章节 8.4

## 5. 不要改的地方（防范围扩散）

| 位置 | 原因 |
|------|------|
| `select` 字段（如 `{ balance, frozenBalance }`） | BalanceRecord 不读 4 字段，**select 不加 4 字段** |
| BalanceRecord 流水 | type=referral_reward / brand_bonus / dividend_reward / manual_reward / refund_reward / refund_dividend / daily_dividend 含义**不变** |
| where 条件 | 4 字段不校验余额（路径 1 接受偶尔越扣） |
| logOperation 调用 | 手动奖励日志**不动**（module='finance'，action='CREATE'） |
| RewardService / DividendService 的其他方法 | 只动上面 5 + 1 + 1 处 |

**铁律 5（路径 1 限制）**：
- 4 字段是"叠加统计"——不强求严格 ≥ 0
- 退款扣回时如果 earningsAvailable < reward.amount，会扣成负数（路径 1 接受）
- 如果以后想严格，加 where 条件防负数——但**本次不做**

const p = new PrismaClient()

  // 1. 找测试用户

  await p.$disconnect()
})()
