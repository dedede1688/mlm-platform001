### 待排期-1：每日解锁功能不生效
- **发现日期**：2026-06-28
- **发现任务**：v56.2 积分转赠前端测试
- **严重等级**：🔴 P0（影响所有升级用户的可用积分）
- **现象描述**：
  - 胡子哥测试账号注册 5 天，总积分 1500（2 次升级发放）
  - 但「可用积分」= 0、「锁定积分」= 0、「每日解锁」= 0 分/天
  - 积分明细只有 2 条 `reward` 类型记录，没有 `unlock` 类型记录
  - 弹窗输入 100 积分转赠，提示「可用积分不足，当前可用 0 积分」
- **推断根因**（未查 DB 验证）：
  - 胡子哥账号升级时间在 v55.1 修复之前（commit `289ac3d`，2026-06-27）
  - v54 派单文档明确写："**唯独缺 schedule 创建入口**"
  - 老数据只有 `createPointsRecord` 加了 totalPoints，没建 `pointsUnlockSchedule` 也没动 lockedPoints
  - 所以 totalPoints=1500、lockedPoints=0、cron 跑也无 schedule 可处理
- **影响范围**：
  - 所有在 v55.1 修复之前升级的用户都受影响
  - 这些用户的积分**永远无法解锁**（lockedPoints 没加，schedule 没建）
- **可能的修复方向**（待胡子哥决策）：
  - 方向 A：写一次性补建脚本，给所有"总积分 > 0 但 lockedPoints = 0 且没 schedule"的账号补建 schedule（最精准）
  - 方向 B：手动触发 cron `/api/cron/daily-tasks` 跑 dailyUnlock（**但无 schedule 不会生效**）
  - 方向 C：业务规则调整为"升级直接发到 unlockedPoints，不做每日解锁"（彻底改规则）
- **下一步动作**：等胡子哥拍板派单

### 待排期-2：admin 调积分不写 pointsRecord 明细
- **发现日期**：2026-06-28
- **发现任务**：v56.2 积分转赠前端测试（胡子哥顺手测试了后台调积分功能）
- **严重等级**：🟡 P1（不影响功能，但用户看不到调账明细，会引起疑惑）
- **现象描述**：
  - 胡子哥从后台给测试账号「充值 500 积分」
  - 顶部统计正确：总积分 1500 → 2000，可用积分 0 → 500
  - 但「积分明细」全部里**只有 2 条 reward 记录**，**没有充值记录**
  - 用户无法看到 admin 调账历史
- **根因（100% 确认，已 grep 代码）**：
  - `src/app/api/admin/users/[id]/points/route.ts` line 46-105 事务里**只做了 4 件事**：
    1. 查询用户
    2. 联动计算 totalPoints/unlockedPoints/lockedPoints
    3. `tx.user.update(...)` 更新 user 表
    4. 返回结果
  - **没有调用 `tx.pointsRecord.create(...)`**
  - `logOperation`（管理员操作日志）✅ 已写
  - `console.log` ✅ 已写
  - **`pointsRecord` 表 ❌ 没写**
- **同类历史 bug 模式**：
  - v46.11：admin 调钱（balance）没调 `sendInApp` 通知用户 → 已修
  - v46.12：退款审核/完成没发通知 → 已修
  - **本次：积分调账没写历史表 → 同一类"写 DB 但不写记录"**
- **修复方向**（待胡子哥决策）：
  - 在事务里加 `tx.pointsRecord.create({ type: 'admin_adjust', ... })`
  - `pointsRecord.type` 是 String 类型，不需要改 schema
  - 同时考虑加 `OrderNotificationService.notifyPointsAdjust`（参考 v46.11 balance_change 模板）
  - 老的已经调过的积分**没记录**——只能事后补，无法追溯
- **影响范围**：
  - 所有历史通过 `/api/admin/users/[id]/points` 调过的积分明细都缺失
  - 估计有 10+ 条胡子哥测试账号的调账历史都看不到
- **下一步动作**：等胡子哥统一排期

## 待排期问题 - 历史已发现但未修（从 git log / 派单历史反查）

> 这部分是从历史派单文档、git log 中**反查**出来的"之前发现但没修或没追溯修复"的问题。胡子哥确认后再归入 pending 分类。

### 历史-1：手续费 system_config key 不一致
- **来源**：v56.2 派单文档「风险点」第 1 条
- **问题**：
  - `/api/points/transfer` 读 `system_config.transfer_fee_percent`
  - `system-parameters.ts` 注册的 key 是 `points.transfer_fee_percent`
- **影响**：如果通过 system-parameters 修改手续费，转账 API 读不到，永远是默认值
- **派单不修原因**：v56.2 只做前端，避免范围扩大
- **建议**：下一个优化任务中顺手处理

### 历史-2：lookup API 隐私问题
- **来源**：v56.2 派单文档「风险点」第 2 条
- **问题**：任何登录用户都能通过 `GET /api/users/lookup?phone=xxx` 查询任意手机号对应的昵称
- **影响**：低（业务接受开放性，但有隐私隐患）
- **建议**：等胡子哥决定是否收紧

### 历史-3：lookup API 限流缺失
- **来源**：v56.2 测试中发现的潜在风险（未被胡子哥直接报）
- **问题**：手机号实时校验是 debounce 500ms，但每次输入都会触发一次 API 调用；如果用户连续输入，可能瞬时打多次
- **影响**：低（debounce 已控制）；高并发场景可能对 DB 有压力
- **建议**：暂不处理，观察运行情况
