## 决策记录

**当前状态**（`src/middleware.ts:31-43`）：

```typescript
function parseJwtPayload(token: string): { userId?: string; phone?: string; role?: string } | null {
  try {
    const parts = token.split('.')
    if (parts.length !== 3) return null
    const payloadJson = Buffer.from(
      parts[1].replace(/-/g, '+').replace(/_/g, '/'),
      'base64'
    ).toString('utf-8')
    return JSON.parse(payloadJson)
  } catch {
    return null
  }
}
```

**根因分析**：
- middleware 当前**只解析 base64 payload，不验证签名**
- 任何人都能伪造 `Bearer xxx.yyy.zzz`（其中 yyy 是 base64 编码的 JSON `{userId: 'admin', role: 'super_admin'}`）通过 middleware 角色检查
- v51.3 commit `e1c5153 fix: middleware 不再验证 JWT 签名（Edge Runtime 与 Node Runtime env 不一致导致 401）` 当时为避免 401 问题改为不验证
- **但现在 middleware 已强制 `runtime = 'nodejs'`**（line 132）—— Edge 限制不存在，应该恢复签名验证

**真实风险评估**：
- API 路由内部 `verifyPermission` → `verifyToken`（`auth.ts:24`）会验证签名，**最终防线已覆盖**
- 但 middleware 的角色检查是"早期拦截"，**漏调 verifyPermission 的路由就是漏洞**
- v47 commit `dbbca22` 已修复 14 个 verifyPermission + 4 个 middleware pathRoleMap，但**新加的路由可能漏调**

**修复目标**：
1. middleware 真正验证 JWT 签名（用 `jwt.verify` 而非手写 base64 解析）
2. 审计所有 admin 路由，确保都调 `verifyPermission` / `verifyAdmin` / `verifyToken`
3. 双保险：middleware 验证 + 路由内验证

## Step 1：改造 middleware.ts 用 jwt.verify 验证签名

### 1.1 关键技术点

**`runtime = 'nodejs'` 已设置**（line 132）—— 可以用 Node.js 标准库
**`jsonwebtoken` 已在 `auth.ts` 使用**—— 直接 import 复用
**`process.env.JWT_SECRET` 在 Node.js Runtime 可读**—— 复用 `auth.ts` 的 `getJwtSecret` 模式

### 1.2 改造方案

**位置**：`src/middleware.ts:1-43`

**当前代码**：

```typescript
import { NextRequest, NextResponse } from 'next/server'

const pathRoleMap: Record<string, string[]> = { ... }

function parseJwtPayload(token: string): { userId?: string; phone?: string; role?: string } | null {
  try {
    const parts = token.split('.')
    if (parts.length !== 3) return null
    const payloadJson = Buffer.from(
      parts[1].replace(/-/g, '+').replace(/_/g, '/'),
      'base64'
    ).toString('utf-8')
    return JSON.parse(payloadJson)
  } catch {
    return null
  }
}
```

**改造方案**：

```typescript
import { NextRequest, NextResponse } from 'next/server'
import jwt from 'jsonwebtoken'

const pathRoleMap: Record<string, string[]> = { ... }

/**
 * v55.2: 真正验证 JWT 签名（之前 v51.3 e1c5153 因 Edge Runtime 问题不验证）
 * 现在 runtime='nodejs'，JWT_SECRET 可读，恢复签名验证
 */
function verifyJwtPayload(token: string): { userId?: string; phone?: string; role?: string } | null {
  try {
    const secret = process.env.JWT_SECRET
    if (!secret) {
      console.error('[v55.2 middleware] JWT_SECRET 未配置')
      return null
    }
    const decoded = jwt.verify(token, secret) as { userId?: string; phone?: string; role?: string }
    return {
      userId: decoded.userId,
      phone: decoded.phone,
      role: decoded.role,
    }
  } catch (err) {
    // 签名失败/过期/格式错误 → 返回 null，不抛错（避免 middleware crash）
    return null
  }
}
```

### 1.3 主逻辑改造

**位置**：`src/middleware.ts:90-100`（原 parseJwtPayload 调用处）

**当前代码**：

```typescript
const token = authHeader.replace('Bearer ', '')
let userRole = ''
let userId = ''
const payload = parseJwtPayload(token)
if (payload) {
  userRole = payload.role || ''
  userId = payload.userId || ''
}
```

**改造方案**：

```typescript
const token = authHeader.replace('Bearer ', '')
let userRole = ''
let userId = ''
const payload = verifyJwtPayload(token)
if (payload) {
  userRole = payload.role || ''
  userId = payload.userId || ''
} else {
  // v55.2: 签名验证失败，明确返回 401 而不是放过
  return NextResponse.json(
    { success: false, error: '认证令牌无效或已过期' },
    { status: 401, headers: { 'x-trace-id': traceId } }
  )
}
```

**关键设计决策**：

1. **签名验证失败 = 401**：之前 v51.3 时代因 Edge Runtime 问题，验证失败导致 500/崩溃。**现在必须 catch + 401**
2. **保留 traceId**：方便问题排查
3. **payload 解析成功但 role 缺失**：仍放行（路由内部 `verifyPermission` 会再次校验）

### 1.4 删除旧函数

`parseJwtPayload` 不再使用，必须删除以避免死代码。

### 2.2 审计标准

每个 admin 路由必须满足以下**之一**：
- 调 `verifyPermission(request, [...])`
- 调 `verifyAdmin(request)`
- 调 `verifyToken(request)` + 手动检查 role
- 调 `getCurrentUser(request)` 或类似封装

**例外**（中间件已鉴权 + 不修改用户数据）：
- 只读统计路由可豁免，但**必须读 middleware 注入的 `x-user-id` / `x-user-role`**
- 公开回调路由（webhook）应放 `/api/webhook/`

## 风险点

1. **JWT_SECRET 在 middleware 可读性**：
   - middleware 已 `runtime = 'nodejs'`（line 132），应该可读
   - 但**极端情况下**环境变量未加载，catch 返回 null → 401（安全降级，不放过）
   - **必须本地测试确认 JWT_SECRET 真的能读到**

2. **Vercel Edge Runtime 兼容**：
   - Vercel 默认 middleware 在 Edge Runtime
   - 我们已强制 `runtime = 'nodejs'`，但**部署到 Vercel 时仍可能回退到 Edge**
   - **如果 Vercel 不支持 nodejs runtime for middleware**，需要回退方案（保留 parseJwtPayload 作为 fallback）

3. **性能开销**：
   - 每次 admin 请求都要 jwt.verify
   - jwt.verify 是 CPU bound 操作，但 token 解析 + HMAC 验签 < 1ms
   - 可接受

4. **审计发现的漏洞**：
   - 如果 Step 2 发现漏调 verifyPermission 的路由，**必须修复后才能 push**
   - 修复可能引入新的破坏性变更——需要 dev server 端到端测试

5. **生产环境 401 风险**：
   - v51.3 当时因为 401 崩溃而妥协
   - 现在 catch + 401 应该不会崩溃
   - 但**部署后必须立刻监控 5xx 错误率**
