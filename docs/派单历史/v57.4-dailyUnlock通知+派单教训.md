## 步骤 2：dailyUnlock 加 sendInApp 通知

**目的**：每天积分解锁后，用户铃铛收到通知，知道"为什么可用积分突然多了"。

**涉及文件**：
- `src/lib/services/points.service.ts`（改 `dailyUnlock` 函数）
- `src/lib/services/order-notification.service.ts`（参考 `notifyPointsAdjust`，加 `notifyPointsUnlock` 方法）

**修改 1：points.service.ts dailyUnlock 事务后加通知**

在 `dailyUnlock` 函数的事务外（事务提交后），对每个处理过的 schedule 调 `OrderNotificationService.notifyPointsUnlock`：

```ts
// 在 dailyUnlock 函数 for 循环结束、count++ 之后加：
await OrderNotificationService.notifyPointsUnlock({
  userId: schedule.userId,
  unlockAmount: dailyAmount,
  newUnlockedPoints: user.unlockedPoints + dailyAmount,
  newLockedPoints: Math.max(0, user.lockedPoints - dailyAmount),
  completedDays: newCompletedDays,
})
```

**注意**：参考 v46.11 模式，**事务外**通知（事务内发通知会被 GC 回收，参考 v46.7/v46.10 教训）。

**修改 2：order-notification.service.ts 加 notifyPointsUnlock 方法**

参考 `notifyPointsAdjust`（line 160-207）的实现，加 `notifyPointsUnlock`：

```ts
static async notifyPointsUnlock(params: {
  userId: string
  unlockAmount: number
  newUnlockedPoints: number
  newLockedPoints: number
  completedDays: number
}) {
  const variables = {
    unlockAmount: String(params.unlockAmount),
    newUnlockedPoints: String(params.newUnlockedPoints),
    newLockedPoints: String(params.newLockedPoints),
    completedDays: String(params.completedDays),
  }
  // 用异步 IIFE 模式（参考 notifyPointsAdjust），失败 catch 吞掉
  await (async () => {
    try {
      const b = await prisma.notificationBatch.create({
        data: {
          type: 'business',
          title: '每日积分解锁通知',
          content: `积分自动解锁 ${params.unlockAmount}，当前可用 ${params.newUnlockedPoints}`,
          templateType: 'points_unlock',
          recipientCount: 1,
        },
      })
      await sendInApp({
        userId: params.userId,
        templateType: 'points_unlock',
        variables,
        batchId: b.id,
      })
    } catch (err) {
      console.error('[v57.4 notifyPointsUnlock]', { error: String(err) })
      logger.error('积分解锁通知失败', { error: String(err) })
    }
  })()
}
```

## 派单方法论教训（v57 批次 3 个步骤总结）

按铁律 8 派单 6 步检查 + 铁律 14 业务决策 grep 验证，**v57 批次的派单存档漏了 3 个关键点**，导致 Mavis 自己在测试时连续修：

### 教训 1：改后端 key/配置时必须 grep 前端有没有硬编码

**v57 步骤 1 真实翻车**：
- commit `d0ede1b` 只改了后端 `route.ts:42` 的 key 一致性
- **前端** `src/app/dashboard/points/page.tsx:102-103` 硬编码 `const FEE_PERCENT = 10` 没读后端配置
- 胡子哥测试时发现"弹窗还显示 10% 但明细是 5%"
- 修复：commit `76c0658`（v57.1 补救）

**强制规则**（新增到派单 6 步检查）：
> 派单改后端配置/key 时，**必须 grep 前端有没有硬编码同名字符串**：
> ```bash
> # 找前端硬编码
> rg "FEE_PERCENT|transfer_fee_percent|DEFAULT_.*PERCENT" src/app src/components
> ```
> 如果有硬编码，**派单存档必须同时要求改前端**

### 教训 2：通知类派单必须在目标环境验证模板 seed

**v57 步骤 2 真实翻车**：
- 派单存档写了 `scripts/seed-points-adjust-template.cjs`
- 猫爪跑了 seed 脚本，typecheck/test/build 全过，commit + push
- **但 Supabase 远程 DB 没有 points_adjust 模板**（本地 seed 不会同步到 Vercel DB）
- 胡子哥测试：admin 调账，铃铛没新通知
- 修复：Mavis 手动跑 `node scripts/seed-points-adjust-template.cjs`（用 Supabase DATABASE_URL）

**强制规则**（新增到派单 6 步检查）：
> 通知类派单（涉及 `sendInApp` / `notificationTemplate`）：
> 1. 派单存档必须写"目标环境已 seed 模板"的验证步骤
> 2. 验收条件必须包含"DB 中模板 exists"的查询输出
> 3. **不能**只信任 build 通过 + 本地测试通过

### 教训 3：业务功能派单要 grep 前端/后端类型是否一致

**v57 步骤 3 真实翻车**：
- dailyUnlock 写 `pointsRecord.type: 'earn'`
- 前端 `FILTER_TABS` 有「解锁」tab 过滤 `type: 'unlock'`
- 胡子哥验收：「解锁」tab 永远是空的
- 修复：commit `f33976c`（v57.4 step1）

**强制规则**（新增到派单 6 步检查）：
> 写 `pointsRecord.type` / `order.status` / 任何 enum-like 字段前：
> ```bash
> # 找前端 type 过滤
> rg "type === 'unlock'|type: 'unlock'|filter.*type" src/app src/components
> ```
> 派单存档必须列出"前端期望的 type 值"和"后端实际写的 type 值"，**必须一致**
