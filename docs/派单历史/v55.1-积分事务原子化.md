## 决策记录

v54.1 commit `97aa215` 已完成 Group H/M/O2，但 **Group D 积分"暂存"bug 仍未修复**。

**当前代码**（`src/lib/services/points.service.ts:160-189`）：

```typescript
static async createPointsUnlockSchedule(data: {...}): Promise<{ id: string }> {
  // 第一步：增量 lockedPoints
  await prisma.user.update({
    where: { id: data.userId },
    data: { lockedPoints: { increment: data.totalPoints } },
  })
  // 第二步：创建 schedule 记录
  return prisma.pointsUnlockSchedule.create({ ... })
}
```

**根因**：两步操作**非原子**：
- 如果第一步成功 + 第二步失败 → `lockedPoints` 已增量 + schedule 不存在
- 结果：用户积分凭空多出，`totalPoints > unlockedPoints + lockedPoints` 不成立
- 资金风险：用户在升级时多收积分

**修复目标**：把 `createPointsRecord` + `createPointsUnlockSchedule` 放进一个 `prisma.$transaction`，全部成功或全部回滚。

## Step 1：改造 `createPointsRecord` 支持 tx 参数

**位置**：`src/lib/services/points.service.ts:27-59`

**当前代码**：

```typescript
static async createPointsRecord(data: {...}) {
  const user = await this.getUserPoints(data.userId)
  await prisma.user.update({
    where: { id: data.userId },
    data: { totalPoints: { increment: data.amount } },
  })
  return prisma.pointsRecord.create({ ... })
}
```

**改造方案**：

```typescript
static async createPointsRecord(data: {...}, tx?: Prisma.TransactionClient) {
  const client = tx ?? prisma
  const user = await this.getUserPoints(data.userId)
  await client.user.update({
    where: { id: data.userId },
    data: { totalPoints: { increment: data.amount } },
  })
  return client.pointsRecord.create({ ... })
}
```

**关键**：
- `tx` 参数可选——保留向后兼容（其他调用方不传 tx 时仍用 prisma）
- 内部 `prisma.user.update` 和 `prisma.pointsRecord.create` 都改为 `client.xxx`
- `getUserPoints(data.userId)` **不传 tx**——这是只读查询，不影响事务边界

**import**：`import { Prisma } from '@prisma/client'`

## Step 2：改造 `createPointsUnlockSchedule` 支持 tx 参数

**位置**：`src/lib/services/points.service.ts:160-189`

**改造方案**：

```typescript
static async createPointsUnlockSchedule(
  data: {...},
  tx?: Prisma.TransactionClient
): Promise<{ id: string }> {
  const client = tx ?? prisma
  await client.user.update({
    where: { id: data.userId },
    data: { lockedPoints: { increment: data.totalPoints } },
  })
  return client.pointsUnlockSchedule.create({
    data: {
      userId: data.userId,
      orderId: data.orderId || '',
      totalPoints: data.totalPoints,
      unlockedPoints: 0,
      remainingPoints: data.totalPoints,
      dailyUnlockRate: data.dailyUnlockRate,
      totalDays: data.totalDays,
      completedDays: 0,
      status: 'active',
      nextUnlockDate: data.nextUnlockDate,
    },
    select: { id: true },
  })
}
```

**关键**：
- `tx` 参数同样可选
- 两步操作（increment lockedPoints + create schedule）都在同一个 client（事务或 prisma）
- **不要**保留 `.catch` 兜底——必须在事务里失败就回滚

## Step 3：user.service.ts 用 `$transaction` 包住升级发积分逻辑

**位置**：`src/lib/services/user.service.ts:149-183`（`checkAndUpgradeLevel`）

**当前代码**：

```typescript
if (newLevel > user.level) {
  await prisma.user.update({
    where: { id: userId },
    data: { level: newLevel },
  })

  if (newLevel >= MEMBER_LEVELS.DISTRIBUTOR && user.level < MEMBER_LEVELS.DISTRIBUTOR) {
    const pointsPerBox = await getBusinessConfig<number>('upgrade.points_per_box', 500)
    const pointsAmount = user.upgradeProductCount * pointsPerBox
    if (pointsAmount > 0) {
      await PointsService.createPointsRecord({ userId, type: 'reward', amount: pointsAmount, description: '...' })
      const dailyUnlockRate = await getBusinessConfig<number>('upgrade.daily_unlock_rate', 0.01)
      const totalDays = Math.ceil(1 / dailyUnlockRate)
      const tomorrow = new Date()
      tomorrow.setHours(0, 0, 0, 0)
      tomorrow.setDate(tomorrow.getDate() + 1)
      await PointsService.createPointsUnlockSchedule({ ... }).catch((err: unknown) => {
        logger.error('[v54 D] 创建积分释放计划失败', { userId, pointsAmount, error: String(err) })
      })
    }
  }

  if (user.referrerId && newLevel === MEMBER_LEVELS.DISTRIBUTOR && user.level < MEMBER_LEVELS.DISTRIBUTOR) {
    await prisma.user.update({
      where: { id: user.referrerId },
      data: { directDistributorCount: { increment: 1 } },
    })
  }
}
```

**改造方案**：

```typescript
if (newLevel > user.level) {
  const pointsPerBox = await getBusinessConfig<number>('upgrade.points_per_box', 500)
  const pointsAmount = user.upgradeProductCount * pointsPerBox
  const dailyUnlockRate = pointsAmount > 0
    ? await getBusinessConfig<number>('upgrade.daily_unlock_rate', 0.01)
    : 0
  const totalDays = dailyUnlockRate > 0 ? Math.ceil(1 / dailyUnlockRate) : 0
  const tomorrow = new Date()
  tomorrow.setHours(0, 0, 0, 0)
  tomorrow.setDate(tomorrow.getDate() + 1)

  // v55.1: 用事务包住 level 更新 + 积分发放 + 释放计划创建
  // 任何一步失败整体回滚，避免积分凭空多出
  await prisma.$transaction(async (tx) => {
    await tx.user.update({
      where: { id: userId },
      data: { level: newLevel },
    })

    if (newLevel >= MEMBER_LEVELS.DISTRIBUTOR && user.level < MEMBER_LEVELS.DISTRIBUTOR) {
      if (pointsAmount > 0) {
        await PointsService.createPointsRecord({
          userId,
          type: 'reward',
          amount: pointsAmount,
          description: `升级为经销商发放积分（${user.upgradeProductCount}件升级产品 × ${pointsPerBox}）`,
        }, tx)

        await PointsService.createPointsUnlockSchedule({
          userId,
          orderId: '',
          totalPoints: pointsAmount,
          dailyUnlockRate,
          totalDays,
          nextUnlockDate: tomorrow,
        }, tx)
      }
    }

    if (user.referrerId && newLevel === MEMBER_LEVELS.DISTRIBUTOR && user.level < MEMBER_LEVELS.DISTRIBUTOR) {
      await tx.user.update({
        where: { id: user.referrerId },
        data: { directDistributorCount: { increment: 1 } },
      })
    }
  })
}
```

**关键设计决策**：

1. **事务范围包含 3 个操作**：
   - 升级 user.level
   - 发放积分（createPointsRecord + createPointsUnlockSchedule）
   - referrer 的 directDistributorCount +1

2. **不再使用 `.catch` 兜底**：
   - 之前用 `.catch` 是为了让 schedule 创建失败不影响主业务
   - 但这导致"积分凭空多出" bug
   - 现在事务化后，失败整体回滚，**用户不会升级成功**

3. **业务规则解读**：
   - 业务规则 §2.5：升级为经销商**一次性奖励积分 + 进入释放计划**
   - 如果释放计划创建失败（理论上不应该，但万一），**用户不应该升级成功**
   - 这是更严谨的业务逻辑

4. **常量提取到事务外**：
   - `pointsPerBox` / `dailyUnlockRate` / `totalDays` / `tomorrow` 在事务外算好
   - 事务里只做 DB 操作

## 风险点

1. **事务回滚影响 referrer.directDistributorCount**：
   - 当前事务包含 3 个操作
   - 如果 referrer update 失败，整个升级也回滚
   - 这是**预期行为**——升级必须全部成功

2. **`getBusinessConfig` 不传 tx**：
   - 这是缓存读取，不影响事务
   - 但要确保 `getBusinessConfig` 不做 DB 写操作（之前确认是只读）

3. **`getUserPoints` 不传 tx**：
   - 在 createPointsRecord 内部 `await this.getUserPoints(data.userId)`
   - 这是只读查询，不需要事务上下文
   - **但有 race condition 风险**——如果事务并发，user.totalPoints 可能读到旧值
   - 解决方案：把 `getUserPoints` 也接受 tx 参数（按需扩展，本次派单不做）

4. **business 逻辑变化**：
   - 之前：升级总是成功，schedule 失败时仅 log 不阻塞
   - 现在：升级 + 积分 + schedule 必须同时成功，否则全部回滚
   - **用户体验**：极少数情况（schedule 失败）下用户会觉得"我升级怎么没成功"——但这是正确的财务严谨性
