| 字段 | 内容 |

| **入参** | userId: string, amount: number, sourceType: 'reward'\|'dividend', sourceId: string |

### 写入点 7：reward_refund

**Happy path**：

### 修订 7：章节 6 风险点 1 统一模式说明

**位置**：章节 6 第 1 节「数据一致性」末尾

**改动**：在「v43-6 已验证的模式」代码块后追加：

```markdown
**v43-7 统一模式**：所有 BalanceRecord 写入都遵循"先查旧值 → 变更 → 用旧值+变动量计算新值"的模式：

\`\`\`typescript
// 步骤 1：查旧值
const before = await tx.user.findUnique({ where: { id }, select: { balance: true, frozenBalance: true } })

// 步骤 2：变更
await tx.user.update({ where: { id }, data: { balance: { increment: amount } } })

// 步骤 3：用旧值计算新值（不要用 findUnique 返回值，避免可能的缓存问题）
const newBalance = before.balance + amount
const newFrozen = before.frozenBalance // 如不变

await tx.balanceRecord.create({ data: { ..., balance: newBalance, frozenBalance: newFrozen, ... } })
\`\`\`

**理由**：v43-6 payOrder 用了这个模式（`nb = freshUser.balance - order.payAmount`），v43-7 保持一致，调试和审查更容易。即使 Prisma 6 事务内查询没问题，统一模式也避免了"看起来更优雅但风险不同"的代码风格。
```
