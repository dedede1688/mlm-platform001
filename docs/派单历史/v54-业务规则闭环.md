## Step 1：Group D 释放规则接通

### 1.1 背景（grep 验证后）

**真实状态**：
- ✅ `PointsUnlockSchedule` 表存在（`prisma/schema.prisma:290-309`）
- ✅ `PointsService.dailyUnlock()` 函数存在（`points.service.ts:159-214`）
- ✅ `vercel.json` cron 配置：`0 0 * * *` 每日 0 点跑 `/api/cron/daily-tasks`
- ✅ `cron.ts:17` 调用 `dailyUnlock()`
- ✅ `/api/cron/daily-tasks` 路由存在
- ❌ **`pointsUnlockSchedule.create()` 0 命中**——schedule 永远不会被创建

**根因**：`user.service.ts:159` 升级为经销商时只调 `PointsService.createPointsRecord`，**没有创建 schedule 记录**。

### 1.2 业务规则要求

`docs/业务规则需求文档.md` §2.5 line 280：
- 买 10 箱升级品 → 升级为经销商 → 系统一次性奖励 **10 × 500 = 5000 积分**
- 积分进入**释放计划**（不能立即使用，按日释放）

§6.1-6.3 line 438-457：
- 释放天数 = 100 / 每天释放比例（自动算）
- 1%/天 → 100 天
- 2%/天 → 50 天
- 0.5%/天 → 200 天

### 1.3 改造方案

**位置**：`src/lib/services/user.service.ts:155-166`

**当前代码**：

### 1.4 新增 `PointsService.createPointsUnlockSchedule` 公共方法

**位置**：`src/lib/services/points.service.ts`（加在 `createPointsRecord` 之后）

```typescript
// v54 D: 创建积分释放计划（升级为经销商时调用）
static async createPointsUnlockSchedule(data: {
  userId: string
  orderId: string | null
  totalPoints: number
  dailyUnlockRate: number
  totalDays: number
  nextUnlockDate: Date
}): Promise<{ id: string }> {
  return prisma.pointsUnlockSchedule.create({
    data: {
      userId: data.userId,
      orderId: data.orderId || '',
      totalPoints: data.totalPoints,
      unlockedPoints: 0,
      remainingPoints: data.totalPoints,
      dailyUnlockRate: data.dailyUnlockRate,
      totalDays: data.totalDays,
      completedDays: 0,
      status: 'active',
      nextUnlockDate: data.nextUnlockDate,
    },
    select: { id: true },
  })
}
```

**注意**：
- `orderId` schema 是 `String @map("order_id")`（无 nullable 标记）—— 升级积分无订单，传空字符串 `''`（不是 null）
- **必须先 grep 确认 `orderId` 是否 nullable**：
  ```bash
  rg -n "orderId\s+String" prisma/schema.prisma | grep -A1 PointsUnlockSchedule
  ```
- 如果 `orderId` 是 `String?`（nullable），传 `null`；否则传 `''`

### 1.5 关键校验

- `totalDays = Math.ceil(1 / dailyUnlockRate)`：
  - rate=0.01 → totalDays=100
  - rate=0.02 → totalDays=50
  - rate=0.005 → totalDays=200
  - rate=0.1 → totalDays=10
- `nextUnlockDate` = **明天 0:00**（cron 每日 0 点跑，提前 1 天）
- 事务安全：升级发积分 + 创建 schedule **应该在一个事务里**（避免发积分成功但 schedule 失败）

**事务改造**（推荐）：
```typescript
await prisma.$transaction(async (tx) => {
  await PointsService.createPointsRecord({ ... })  // 注意：这里需要改造支持 tx
  await tx.pointsUnlockSchedule.create({ ... })
})
```

**或者**：保持现有 `PointsService.createPointsRecord` 不变（已封装），在 schedule 创建失败时**用 `.catch` + logger.error**（不阻塞主业务）：
```typescript
await PointsService.createPointsRecord({ ... })
await PointsService.createPointsUnlockSchedule({ ... }).catch((err) => {
  logger.error('[v54 D] 创建积分释放计划失败', { userId, pointsAmount, error: String(err) })
})
```

**建议**：用 `.catch` 模式（v50 业务通知也是这个模式），简单可控。

### 2.2 问题分析

**H 路径（升级品订单）**：
- ✅ 推荐人 `directSalesAmount += order.payAmount`（line 343）
- ❌ 买家**自己**没加 `directSalesAmount`

**非 H 路径（普通订单）**：
- ✅ 推荐人 `directSalesAmount += order.payAmount`（line 358）
- ❌ 买家**自己**没加 `directSalesAmount`

**业务规则 §7.4 解读**：
- "升级品订单计入销售额"——指**谁的销售额**？买家还是推荐人？
- 根据 §5.1 line 385-426（双轨制 UI）推测：业务规则要求**买家自己**的销售额用于**自己升级**
- 当前实现：**只给推荐人加，**导致买家无法通过升级品订单升级

### 2.3 改造方案

**位置**：`src/lib/services/reward.service.ts:325-362`

**核心改动**：买家自己也加 directSalesAmount（升级品订单 + 普通订单都加）。
