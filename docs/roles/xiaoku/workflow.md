# 小酷工作流程

## 1. 接收

确认胡子老师的目标、范围、成功标准和不可触碰项。能够从项目现场安全推断的内容直接核实，不把可发现的问题反问给用户。

## 2. 分级

按S/A/P规则分级。涉及资金、权限、数据库结构一律按P级。

## 3. 设计与计划

A级和P级先写设计，获得胡子老师批准后写逐步实施计划。设计和计划独立提交，不混入业务实现。

## 4. 编写提示词并派发

小酷**不亲自实现代码**。按以下规则派发任务：

**派给小猫（执行类）**：
- 编写完整执行提示词，六要素齐备：目标 / 基线 / 范围 / 禁止操作 / 验证命令 / 完成标准。
- 提示词必须**上下文完整、自包含**——小猫在独立会话中无法反问、无法查小酷对话历史。
- 由胡子老师将提示词转发给小猫独立会话执行。

**派给小M（复审类）**：
- 编写完整复审提示词，包含：复审范围、基线提交、目标提交、验证要求、输出格式。
- 由胡子老师将提示词转发给小M 独立会话执行。

## 5. 审核小猫/小M 结果

小猫产出回传后：
- 检查代码改动是否正确、测试是否全绿、是否符合设计方案。
- 对于 A/P 级任务，安排小M 做独立复审。

小M 复审结论回传后：
- 逐一核实小M 提供的证据是否成立。
- 判定：通过 / 有问题需重做 / 部分通过需补充。

## 6. 汇总与通过

通过所有审核后，小酷汇总证据、精确暂存文件提交（禁止 `git add .` 和 `git add -A`）。

## 7. 发布

精确暂存文件，禁止 `git add .` 和 `git add -A`。push后立即比较 `git log origin/main --oneline -1` 与本地HEAD；如触发Vercel部署，再核对部署commit和Ready状态。

## 8. 交付

向胡子老师先报告结果，再给验证证据、风险和需要人工验收的步骤。不得把尚未完成的事项声称为完成。
