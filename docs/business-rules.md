# 二、奖励规则（v2 核心）

## 2.1 直推奖（referral_reward）

### 规则

| 项 | 值 |
|---|---|
| **比例** | 默认 **20%**（后台可调） |
| **资格门槛** | **推荐人 A 必须买过 ≥ 1 件升级品** |
| **触发场景** | B（被推荐人）下任何**已支付**订单（B 身份不限） |
| **接收人** | A = B 的直接推荐人 |
| **每单发放** | ✅ 每单都给 |
| **未解锁处理** | 推荐奖未解锁时，弹 Toast 提示用户 |

### 举例

> A 推荐 B，B 买 500 元产品
> - A 买过升级品 → **A 拿 100 元直推奖（20%）**
> - A 没买过升级品 → 系统不发奖，弹 Toast："您还未购买升级品，本次推荐奖 ¥100 未发放"

### 后台可调

- `reward.referral_rate` = 0.20（直推奖比例）

## 2.2 品牌管理奖（brand_bonus_reward）— v4 完整

### 规则

| 项 | 值 |
|---|---|
| **比例** | 默认 **20%**（后台可调） |
| **资格门槛** | A 必须 ≥ 经销商 |
| **触发商品** | **普通商品**（排除升级品） |
| **触发场景** | B 下任何已支付订单（B 身份不限） |
| **链** | **安置链（parentId）** |
| **轮换规则** | **B 买第 N 单 → 安置链上第 N 个经销商拿** |
| **"层"的定义** | **经销商的序号**（会员跳过但占位，序号仍递增） |
| **层数上限**（基于 A 直推经销商下线数） | 0 个 → 2 层；1 个 → 4 层；2 个 → 10 层；主任+ → 10 层 |
| **A 不是经销商** | 跳过 A，从安置链上**第 1 个经销商**开始编号 |
| **找不到经销商** | **沉淀到公司**（写 OperationLog，不发放） |
| **第 11 单** | 循环回第 1 个经销商 |

### "层"的口径

**"层"不是安置链物理位置，而是经销商的出现序号**。

```
安置链：B ← A（经销商）← M（会员）← X（经销商）← Y（经销商）

经销商序号：A 是第 1 个，X 是第 2 个，Y 是第 3 个
M 是会员，被跳过，但 M 仍然"占"一层（即下一个经销商是第 2 层而非第 3 层）
```

### 轮换规则举例

A 是经销商，A 推荐 B（A 直推 0 个下线经销商，A 最多 2 层）。

安置链（自下而上）：
```
         Y（经销商，第 3 个）
          ↑
         X（经销商，第 2 个）
          ↑
         A（经销商，第 1 个）—— A 是 B 的推荐人
          ↑
         B（买家）
```

B 买 11 单（每单 500 元）：

| 单数 | 直推奖 | 第 N 个经销商 | 品牌管理奖 | 备注 |
|------|--------|--------------|-----------|------|
| 1 | A 拿 100 | A（第 1 个） | **A** 拿 100 | A 拿 |
| 2 | A 拿 100 | X（第 2 个） | **X** 拿 100 | A 触发到第 2 层 |
| 3 | A 拿 100 | Y（第 3 个） | **沉淀** | ⚠️ A 最多 2 层 |
| 4-10 | A 拿 100 | - | **沉淀** | |
| 11 | A 拿 100 | A（第 1 个，循环） | **A** 拿 100 | 循环 |

### A 不是经销商的举例

A 是会员，A 上面是经销商 X：
```
         X（经销商）
          ↑
         A（会员，推荐人）
          ↑
         B（买家）
```

D 买：
- 第 1 单 → A 拿直推 → 跳过 A → 第 1 个经销商 = **X** → X 拿品牌管理奖
- 第 2 单 → A 拿直推 → 找第 2 个经销商（安置链没有）→ **沉淀**
- 第 11 单 → 循环回 X → X 拿

### 后台可调

- `reward.brand_bonus_rate` = 0.20（品牌管理奖比例）

### 沉淀实现

- 不写 `Reward` 表
- 不增加用户 `balance`
- 写 `OperationLog(action='BRAND_BONUS_SINK', module='reward')`，记录订单 ID / 买家 ID / 应发金额 / 应发位置 / 沉淀原因

# 四、安置规则

## 4.1 双链关系

用户表里同时有**两条链**：

| 字段 | 含义 | 业务用途 |
|------|------|----------|
| `referrerId` | **推荐链**（谁推荐我） | 直推奖、团队奖（已取消）、分红奖 |
| `parentId` + `position` | **安置链**（我在哪里） | 品牌管理奖（v2）、推荐树展示 |

## 4.2 安置算法（三三复制）

**核心规则**：
- 每个 parent 最多 **3** 个子节点（position = 1 / 2 / 3）
- **BFS** 找最近的空位
- 优先填满上一层 → 再往下一层

**算法伪代码**：

```
输入：referrerId（谁推荐了我）
1. 查所有 parentId = referrerId 的用户（最多 3 个子节点）
2. BFS 遍历：
   - 检查当前位置的空位（1/2/3）
   - 如果有空位 → 返回 (parentId=currentId, position=空位号)
   - 如果满了 → 把子节点加入队列
3. 队列空 → 抛错
```

**举例**：
```
                A（推荐人）
              / | \
            B   C   D     ← 第 1 层（3 个）
           /|\ 
          E F G            ← 第 2 层（B 下面）

B 满了（3 子），C 满了（3 子），D 还差 1 个
新用户 H：referrerId = X
   ↓
BFS：找最近的空位 → 优先填 D 的空位 → D.position=2
```

# 七、边界与异常

## 7.1 沉淀规则

### 定义

**沉淀** = 品牌管理奖在轮换过程中，如果目标位置（第 N 层经销商）找不到经销商，奖励金额**不发放**，**记入公司账**。

### 触发场景

1. **安置链上找不到第 N 个经销商**（经销商数量不足）
2. **超过 A 的层数上限**（如 A 直推 0 个下线经销商，最多 2 层；第 3 层 → 沉淀）

### 沉淀实现

- 不写 `Reward` 表
- 不增加用户 `balance`
- 写 `OperationLog(action='BRAND_BONUS_SINK', module='reward')`，记录：
  - 订单 ID
  - 买家 ID
  - 应发金额
  - 应发的"第 N 层"位置
  - 沉淀原因（如"安置链无第 N 个经销商"）

## 7.2 推荐奖未解锁

**触发条件**：A 推荐 B 下单 → A 没买过升级品

**行为**：
- ❌ 不发直推奖
- ✅ 弹 Toast："您还未购买升级品，本次推荐奖 ¥XX 未发放。购买升级品即可解锁。"
- ❌ 不发品牌管理奖（也不参与轮换）

## 7.3 找不到经销商

**触发条件**：品牌管理奖轮换时找不到对应位置的经销商

**行为**：沉淀（详见 7.1）

## 7.4 升级品订单计入销售额

**触发条件**：升级条件判断时

**行为**：**包括**升级品订单的实付金额（胡子哥确认）

## 7.5 升级条件 OR 逻辑去除

**触发条件**：升级为主任 / 经理 / 总监 / 总裁 / 董事时

**行为**：**只看销售额**，**不再看直推经销商数**（v2 变更）
