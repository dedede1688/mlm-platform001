## 一、6 大类结构(最终版)

### 1. 🏠 数据中台(首页)
**业务对象**:把"数据"汇总成可看的、可点穿的核心入口

- 核心 KPI(销售/订单/用户/库存)
- 异常清单(退款待审/提现待审/发货待处理/库存预警)
- 状态流转(订单进度可视化)
- 高级分析 → 运营报表(子页)

### 2. 📦 系统后台(基础建设)
**业务对象**:管后端基础

- **基础配置**:Logo / 轮播图 / 客服电话 / 用户协议 / 隐私政策
- **功能参数**:分红比例 / 提现门槛 / 升级规则 / 业务开关
- **通知管理**:通知模板 / 通知发件箱 / 通知详情
- **审计日志**:所有 admin 操作的记录
- **角色与权限**(🆕 新增):可视化配置每个角色能看哪些菜单

### 3. 📦 商品
**业务对象**:卖的东西

- 商品管理(列表 + 编辑)
- 分类管理(树形结构)

### 4. 💰 财务
**业务对象**:钱的事

- 奖励流水(推荐奖/品牌管理奖/分红奖)
- 提现审核
- 退款审核
- 拒绝理由模板

### 5. 🛒 订单
**业务对象**:交易过程

- 订单管理(列表 + 发货 + 物流)

### 6. 👤 会员
**业务对象**:人

- 会员管理(用户列表 + 详情 + 推荐树弹窗 + 余额流水弹窗)

## 五、🆕 角色与权限管理页面(详细方案)

### 现状
- 权限**写死**在 `src/lib/admin-menu.ts` 的 `ROLE_MENUS` 字典
- 改权限要:改代码 → build → push → 等部署,4 步才能动
- 财务管理员要加"看订单"权限?现在做不到(胡子哥昨天才发现这个问题)

### 目标
- super_admin 登录 → 进"系统后台-角色与权限" → 勾选每个角色能看哪些菜单 → 保存 → **立即生效**
- 不用动代码

### 实现方案:**轻量版**(推荐,1 天工作量)

#### 5.1 数据存储
利用项目里已有的 `system_config` 表(以前 v60 P2-A 修过):

```
key:        "role_menus"
value:      JSON.stringify({ super_admin: [...], goods_admin: [...], ... })
updatedAt:  时间戳
```

启动时:
- `admin-menu.ts` 改成"优先从 DB 读,读不到用 hardcoded 默认值"
- DB 里默认值 = 现在的 `ROLE_MENUIS` 字典

#### 5.2 页面设计
`/admin/system/roles` 页面:

```
┌─────────────────────────────────────┐
│  🛡️ 角色与权限                        │
│                                       │
│  超级管理员(super_admin)             │
│  ☑ 数据中台  ☑ 系统后台  ☑ 商品       │
│  ☑ 财务      ☑ 订单      ☑ 会员       │
│                                       │
│  商品管理员(goods_admin)             │
│  ☑ 数据中台  ☐ 系统后台  ☑ 商品       │
│  ☐ 财务      ☑ 订单      ☐ 会员       │
│                                       │
│  财务管理员(finance_admin)            │
│  ☑ 数据中台  ☐ 系统后台  ☐ 商品       │
│  ☑ 财务      ☐ 订单      ☐ 会员       │
│                                       │
│  客服管理员(support_admin)            │
│  ☑ 数据中台  ☐ 系统后台  ☐ 商品       │
│  ☐ 财务      ☐ 订单      ☑ 会员       │
│                                       │
│  审计员(auditor)                      │
│  ☑ 数据中台  ☐ 系统后台  ☐ 商品       │
│  ☐ 财务      ☐ 订单      ☐ 会员       │
│                                       │
│  [保存] [重置]                         │
└─────────────────────────────────────┘
```

每个角色一张卡(可折叠),里面有 6 个大类复选框,super_admin 勾选后保存。

#### 5.3 API
- `GET /api/admin/roles` 返回当前所有角色的菜单配置(从 DB 读)
- `PUT /api/admin/roles` 接收新配置,写 DB + 清缓存(类似 v60 P2-A 修的那个 `invalidateBusinessConfigCache`)

#### 5.4 缓存策略
- DB 改完 → 立即失效内存缓存
- 下次请求 → 重新读 DB → 拉新菜单
- 5 分钟缓存(参考现有 `referral-tree` 的 30 秒缓存)

### 工作量
- DB 改造 + 启动逻辑:2-3 小时
- API:1-2 小时
- 页面:2-3 小时
- 测试 + 集成:1 小时
- **总共 6-9 小时(约 1 天)**

### 不做的(避免范围蔓延)
- ❌ 自定义新建角色(只能改现有 5 个)
- ❌ 删除/重命名角色
- ❌ 用户级权限覆盖(只能按角色)
- ❌ 角色继承/复制

如果要这些高级功能,以后再加(2-3 天工作量)。

## 八、数据中台设计(核心)

不管 4 大类 / 6 大类,**数据中台是首页+核心**,这块不变。

```
┌─────────────────────────────────────┐
│  📊 数据中台                          │
│                                       │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐   │
│  │销售 │ │订单 │ │用户 │ │库存 │   │
│  │ 3000│ │  28 │ │  6人│ │  3个│   │
│  └─────┘ └─────┘ └─────┘ └─────┘   │
│                                       │
│  ⚠️ 异常清单(点进去直接处理)         │
│  · 退款待审 3 笔  [立即处理 →]       │
│  · 提现待审 1 笔  [立即处理 →]       │
│  · 库存预警 3 个  [立即处理 →]       │
│                                       │
│  📈 状态流转                          │
│  待支付(5)→已支付(3)→已发货(8)→完成  │
│                                       │
│  📋 高级分析 → 运营报表(子页)        │
└─────────────────────────────────────┘
```

**3 个关键 UX 改进**:
1. **异常清单**:退款/提现/库存预警都汇总在一处
2. **穿透链接**:点"3 笔退款待审"直接跳到退款管理页
3. **状态流转**:订单/提现的进度可视化
