# 充值设置闭环设计

> 日期：2026-07-11
> 状态：业务方案已确认，等待胡子老师审阅书面规格
> 执行分工：小M负责执行，猫爪负责独立审核与验证，小酷负责方案和风险控制

## 1. 目标

在不接入任何支付接口的前提下，完成“后台配置收款二维码 -> 用户线下扫码付款 -> 用户上传付款成功截图 -> 财务人工审核 -> 审核通过后增加购物余额”的充值闭环。

本项目不自动收款、不自动查询支付结果、不自动拉起微信或支付宝，也不根据支付渠道自动确认到账。

## 2. 已确认的业务规则

1. 后台只维护一张当前有效的充值二维码。
2. 二维码可以是聚合码，也可以是其他收款码，系统统一称为“平台充值二维码”。
3. 用户可以使用另一部设备直接扫码，也可以保存二维码后从付款软件相册识别。
4. 用户付款后上传的是“付款成功截图”，不是收款二维码。
5. 用户提交充值申请时：
   - 手机号由当前登录账号自动获取并只读展示；
   - 充值金额必填；
   - 付款成功截图必填；
   - 备注选填；
   - 不要求用户填写充值时间；
   - 不要求用户填写交易订单号。
6. 系统自动记录申请提交时间。
7. 后台人工核验付款截图和申报金额。
8. 审核通过后，增加用户购物余额并写资金流水、审核日志、操作日志和用户通知。
9. 审核拒绝时不改变用户资金。
10. 充值余额不能进入收益，也不能用于提现。
11. 关闭充值功能时，只阻止新申请；历史记录仍可查看，已有待审核申请仍可继续审核。

## 3. 技术方案

复用现有 `system_configs`（系统配置表），不新增数据库表，不执行数据库迁移。

新增或维护以下配置项：

| 配置项 | 含义 | 类型 |
|---|---|---|
| `recharge.enabled` | 充值功能是否启用 | 布尔值 |
| `recharge.qr_code_url` | 当前充值二维码图片地址 | 字符串 |
| `recharge.qr_code_label` | 二维码说明 | 字符串 |
| `recharge.payee_name` | 收款人名称 | 字符串 |
| `recharge.min_amount` | 最低充值金额 | 数字 |
| `recharge.max_amount` | 最高充值金额 | 数字 |
| `recharge.instruction` | 充值说明 | 字符串 |
| `recharge.contact_phone` | 客服电话 | 字符串 |
| `recharge.service_time` | 服务时间 | 字符串 |

旧的支付宝、微信和银行卡配置暂不删除，避免影响历史数据；用户端和新后台设置页面不再使用这些旧配置。

## 4. 后台充值设置接口

新增 `/api/admin/recharge-settings`（后台充值设置接口），提供读取和保存能力。

### 4.1 权限

- 超级管理员：允许读取和保存。
- 财务管理员：允许读取和保存。
- 审计员、商品管理员、客服管理员及其他角色：禁止访问。
- 接口内部必须使用项目现有的权限校验工具再次查库确认角色。
- 权限中间件增加精确角色映射，形成双重校验。

### 4.2 字段白名单

接口只允许读取和修改第 3 节列出的充值配置项。不得接受任意配置键，不得通过该接口修改其他系统配置。

### 4.3 保存校验

1. 启用充值时必须存在有效二维码图片地址。
2. 二维码图片地址必须以 `https://` 开头。
3. 不允许保存本地临时图片数据。
4. 最低金额必须是大于零的有限数字。
5. 最高金额必须是大于零的有限数字。
6. 最高金额必须大于或等于最低金额。
7. 所有文本字段保存前去除首尾空格。
8. 多个配置项在同一事务中更新，避免部分成功、部分失败。
9. 保存成功后立即清除业务配置缓存。
10. 保存行为写入操作日志，记录修改前后值，但不记录图片二进制数据。

## 5. 充值服务规则调整

### 5.1 配置读取

用户充值设置返回：

- 是否启用；
- 二维码图片地址；
- 二维码说明；
- 收款人名称；
- 最低金额；
- 最高金额；
- 充值说明；
- 客服电话；
- 服务时间。

不再向新用户页面返回支付宝、微信、银行卡支付方式选项。

### 5.2 创建充值申请

服务端在创建申请前必须检查：

1. 充值功能已启用；
2. 已配置有效二维码；
3. 充值金额有效且在限制范围内；
4. 付款成功截图存在且为有效网络图片地址；
5. 当前登录用户真实存在。

用户编号和手机号只从登录凭证获取，不接受前端传入。

数据库现有 `payment_method`（支付方式）字段继续保留。新充值申请由服务端统一写入 `qr_code`（二维码扫码充值），前端不能传入或覆盖该值。

历史记录中的 `alipay`（支付宝）、`wechat`（微信）和 `bank_card`（银行卡）保持不变，不做数据迁移。

## 6. 后台页面设计

在“财务管理”增加第四个标签页“充值设置”。

为避免继续扩大财务页面文件，充值设置表单使用独立组件，由财务页面引用。

### 6.1 表单内容

1. 充值功能启用或停用开关。
2. 二维码图片上传、预览、查看大图、替换和删除。
3. 二维码说明。
4. 收款人名称。
5. 最低充值金额。
6. 最高充值金额。
7. 充值说明。
8. 客服电话。
9. 服务时间。

### 6.2 交互规则

- 停用时提示：只停止新申请，已有申请仍可审核。
- 启用时没有二维码，不允许保存。
- 图片上传未获得有效网络地址时，不允许保存。
- 保存期间禁止重复提交。
- 保存失败时保留表单内容并显示具体错误。
- 保存成功后重新读取配置，确认数据库实际保存结果。

### 6.3 充值审核列表

- 删除新的支付方式筛选框。
- 新记录显示“二维码扫码充值”。
- 历史记录继续显示原支付方式。
- 查看凭证、审核通过、审核拒绝和审核日志功能保持不变。

## 7. 用户页面设计

### 7.1 充值启用状态

1. 展示当前二维码、收款人名称和二维码说明。
2. 提供“查看大图”和“保存二维码”。
3. 保存失败时提示用户打开大图后长按保存。
4. 展示两种付款方法：
   - 使用另一部设备直接扫码；
   - 保存二维码后，在付款软件扫一扫中从相册识别。
5. 明确提示付款完成后返回页面提交申请。

### 7.2 充值关闭状态

1. 隐藏二维码和充值申请表单。
2. 显示“充值服务暂时关闭，请联系客服”。
3. 客服电话存在时一并显示。
4. 历史充值记录继续展示。

### 7.3 申请表单

只保留：

- 当前手机号，只读；
- 充值金额，必填；
- 付款成功截图，必填；
- 备注，选填。

删除支付宝、微信和银行卡选择区域。不增加充值时间和交易订单号输入框。

### 7.4 历史记录

- 新记录显示“二维码扫码充值”。
- 旧记录继续显示原来的支付宝、微信或银行卡。
- 现有凭证弹窗继续使用。

## 8. 安全与错误处理

1. 用户端关闭表单不等于安全控制，服务端必须再次检查充值开关。
2. 后台接口必须使用配置字段白名单。
3. 二维码和付款截图均不允许本地临时图片数据。
4. 图片上传失败不能伪装成保存成功。
5. 后台保存失败不能留下半套配置。
6. 配置缓存必须在保存后立即失效。
7. 关闭充值不得自动拒绝已有待审核申请。
8. 不修改提现、奖励和收益业务。

## 9. 测试与验收

### 9.1 自动测试

至少覆盖：

- 充值设置读取与保存；
- 超级管理员和财务管理员权限；
- 其他角色拒绝访问；
- 字段白名单；
- 启用但无二维码时保存失败；
- 二维码地址校验；
- 金额范围校验；
- 配置缓存失效；
- 操作日志；
- 充值关闭时拒绝新申请；
- 关闭后历史待审核申请仍可审核；
- 新申请固定写入 `qr_code`；
- 用户不能伪造用户编号、手机号或支付方式；
- 历史支付方式不变。

### 9.2 页面验收

- 后台可以上传、预览、替换二维码；
- 后台可以启用和停用充值；
- 保存后刷新仍显示正确配置；
- 用户端可以查看大图和保存二维码；
- 用户端不再显示支付渠道选择；
- 用户手机号只读；
- 停用后申请表单消失，历史记录仍可查看；
- 后台原有审核功能无回归。

### 9.3 完整业务链路

1. 后台上传测试二维码并启用充值。
2. 用户刷新后看到新二维码。
3. 用户填写金额并上传测试付款截图。
4. 用户提交充值申请。
5. 后台看到待审核记录并查看付款凭证。
6. 后台审核通过。
7. 用户购物余额增加对应金额。
8. 余额流水、审核日志、操作日志和通知生成。
9. 后台停用充值。
10. 用户不能提交新申请，但历史记录仍可查看。
11. 后台仍可处理停用前已有的待审核申请。

## 10. 执行拆分

### 第一包：充值设置底座与业务规则

- 后台充值设置读取和保存接口；
- 配置项和严格校验；
- 权限、字段白名单、事务、缓存和操作日志；
- 用户设置返回结构调整；
- 创建充值申请规则调整；
- 新申请固定记录为二维码扫码充值；
- 自动测试；
- 不修改页面。

第一包由小M执行，猫爪独立审核验证。审核通过后才允许提交和部署。

### 第二包：后台与用户页面

- 财务管理增加充值设置标签页；
- 独立充值设置组件；
- 用户端二维码展示、查看和保存；
- 用户充值表单简化；
- 后台审核列表调整；
- 页面测试和完整业务链路验收。

第二包在第一包部署并验收通过后开始，同样由小M执行、猫爪审核验证。

## 11. 明确不做

- 不接入支付宝、微信、银行卡或聚合支付接口；
- 不自动确认到账；
- 不提供网页直接唤起付款软件；
- 不新增数据库表；
- 不执行数据库迁移；
- 不修改历史充值数据；
- 不让充值余额进入收益；
- 不让充值余额提现；
- 不在本轮实现多二维码、动态二维码或临时二维码。
