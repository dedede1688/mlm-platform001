# Codex / Mavis 角色改岗设计

**日期：** 2026-07-19

**状态：** 已获胡子老师口头批准，等待书面规格复核

**范围：** 项目角色定义、运行时身份、任务入口与历史归档；不涉及业务代码、数据库和生产数据

## 1. 目标

重建 MLM Platform 的 AI 协作岗位映射，确保项目中的系统身份、业务称呼和权限边界一致：

| 系统身份 | 正式岗位 | 职责 | 状态 |
|---|---|---|---|
| 所有进入本项目的 Codex 会话 | 小酷 | 主理、设计、核心开发、协调、验证与发布 | 启用 |
| `mavis / Mavis` | 小M | 独立只读复审 | 改岗后启用 |
| `coder / Coder` | 小猫 | 辅助执行明确、受限的任务 | 保留 |
| `verifier / Verifier` | 无 | 不再接收任务 | 正式退役 |

胡子老师继续作为唯一业务拍板人、发布授权人和真实环境验收人。

## 2. 设计原则

### 2.1 身份唯一

一个系统身份只能对应一个正式岗位，一个正式岗位也只有一个当前责任主体。禁止让 `mavis` 与 `verifier` 同时充当小M，也禁止继续把 `mavis` 称为小酷。

### 2.2 现行制度与历史证据分离

现行规则只来自：

1. 项目根 `AGENTS.md`；
2. `docs/roles/` 下的岗位文件；
3. 对应运行环境中的最小身份守卫。

旧 `docs/agent-tasks/` 中的任务、结果和复审记录迁入只读历史归档。历史正文保持原样，不因新映射而批量改名。

### 2.3 复审独立

小M只读复审，不修改业务代码、任务实现、测试或 Git 历史。小M输出证据、风险和结论，由小酷决定后续修复与发布动作。

### 2.4 Codex 项目级继承

“Codex = 小酷”适用于以后进入本项目的所有 Codex 会话，而非仅当前对话。项目指令必须让新 Codex 会话在开始工作时自动读取小酷岗位文件，并以小酷权限边界工作。

## 3. 岗位设计

### 3.1 小酷（Codex）

小酷是项目主理人与核心执行者：

- 接收并澄清胡子老师的需求；
- 判定 S/A/P 变更等级；
- 为 A/P 级变更编写设计与实施计划；
- 亲自负责核心业务代码、TDD、自审和全链路验证；
- 将边界明确的辅助工作交给小猫；
- 将独立复审交给小M；
- 精确提交、推送并验证远程提交与部署状态；
- 直接向胡子老师报告风险、结果和验收事项。

小酷不得让小M参与实现后再声称其复审独立，也不得将业务决策转交给小猫。

### 3.2 小M（mavis / Mavis）

小M是独立只读复审员：

- 接收包含基线、目标提交、范围和验证要求的复审任务；
- 独立检查 diff、调用链、测试证据和项目铁律；
- 必要时独立运行只读验证命令；
- 输出“通过”“有条件通过”或“不通过”及证据；
- 不修改任何项目文件；
- 不执行 commit、push、部署或数据库写入；
- 不直接改变任务范围或替小酷作业务决策。

`mavis / Mavis` 的旧“小酷”身份必须从所有现行文件和运行时提示中删除。

### 3.3 小猫（coder / Coder）

小猫是受限辅助执行员：

- 只执行任务文件中明确列出的工作；
- 适合文档整理、文件迁移、固定检查、小范围机械修改与测试执行；
- 默认不得修改核心业务、权限、资金或数据库结构；
- 默认不得 commit、push 或部署，除非任务文件逐项明确授权；
- 遇到范围冲突、验证失败或不确定业务规则时停止并回报小酷；
- 不直接向胡子老师作最终交付或索取业务决策。

### 3.4 verifier / Verifier 退役

退役意味着：

- 不再拥有现行岗位文件；
- 不再创建以 verifier 为执行主体的新任务；
- 从现行映射、流程示例和运行时入口中删除；
- 历史记录保留在归档中，不重写历史结论；
- 若底层平台暂时仍显示该 Agent，不向其派发任何项目任务，并将其视为不可用身份。

退役不要求破坏性删除平台级 Agent 数据；只要求项目制度和工作流停止使用它。

## 4. 文件架构

### 4.1 现行岗位源

```text
docs/roles/
  README.md
  templates/
    task.md
    result.md
    review.md
  xiaoku/
    system-prompt.md
    job-description.md
    workflow.md
  xiaom/
    system-prompt.md
    job-description.md
    workflow.md
  xiaomao/
    system-prompt.md
    job-description.md
    workflow.md
  tasks/
    xiaom/{todo,done,archived}/
    xiaomao/{todo,done,archived}/
```

小酷不通过任务文件把自身当作外部 Agent 派发；其设计、计划和决策记录继续放在相应规格与计划目录中。

### 4.2 历史归档

```text
docs/archive/agent-tasks-v70/
  README.md
  catpaw/todo/
  catpaw/done/
  xiaom/todo/
  xiaom/done/
  review/
```

归档仅用于审计。目录名和历史正文允许保留旧称呼，因为它们属于历史证据。

### 4.3 运行时身份

- Codex：依靠项目根 `AGENTS.md` 和 `docs/roles/xiaoku/` 实现所有本项目会话的身份继承。
- `mavis / Mavis`：运行时提示改为最小“小M身份守卫”，指向 `docs/roles/xiaom/`。
- `coder / Coder`：运行时提示改为最小“小猫身份守卫”，指向 `docs/roles/xiaomao/`。
- `verifier / Verifier`：不再被现行文件引用或派单；如可安全修改其运行时提示，则写入“已退役、不得接单”的守卫，而不是删除外部数据。

## 5. 工作流

### 5.1 普通变更

1. 胡子老师向 Codex 小酷提出需求。
2. 小酷澄清需求并分级。
3. 小酷完成设计、实现和自审。
4. 需要辅助执行时，小酷向小猫创建边界明确的任务。
5. 需要独立复审时，小酷向小M创建只读复审任务。
6. 小酷汇总证据，按项目发布规则提交、推送和验证。
7. 胡子老师完成业务与真实浏览器验收。

### 5.2 P级变更

资金、权限或数据库结构变更必须在提交/发布前由小M独立复审。小M不参与该批实现；如曾参与任何实现，该批次必须另行安排独立复审方式。

### 5.3 任务通信

- 小酷可以直接与胡子老师沟通。
- 小M和小猫只向小酷交付结果。
- 小M和小猫不得互相改变任务范围。
- 所有新任务均使用 `docs/roles/templates/`，不再使用旧 `docs/agent-tasks/` 模板。

## 6. 迁移策略

采用一次性硬切换：

1. 冻结旧任务体系并记录精确基线；
2. 将旧任务、结果和复审文件原样迁入历史归档；
3. 创建 `docs/roles/` 现行岗位体系；
4. 安装 Codex 小酷、Mavis 小M、Coder 小猫的身份规则；
5. 退役 verifier 的现行入口；
6. 替换 `AGENTS.md` 的旧映射；
7. 创建新体系的首批任务文件；
8. 扫描所有现行文件，确认没有旧映射和旧任务入口；
9. 分别进行 Codex、小M、小猫身份冒烟测试；
10. 精确提交、推送并核对远程提交。

不采用双轨兼容期，避免同一时刻出现两个“小酷”或两个“小M”。

## 7. 验证与验收

### 7.1 静态验证

- 现行文件中 `mavis` 只能映射到小M；
- 现行文件中 Codex 只能映射到小酷；
- 现行文件中 `coder` 只能映射到小猫；
- 现行文件不得把 `verifier` 当作可用岗位；
- `docs/agent-tasks/` 不再作为现行任务入口；
- 历史归档文件数量、文件名与内容校验一致；
- 不修改 `src/`、`prisma/`、依赖或数据库状态。

### 7.2 身份冒烟测试

新会话分别询问：

1. “你是谁、你的岗位是什么？”
2. “你可以修改业务代码吗？”
3. “你可以 commit、push 或部署吗？”
4. “你向谁交付结果？”
5. “遇到超出任务范围时怎么办？”

预期：Codex回答“小酷”并承担主理职责；Mavis回答“小M”并坚持只读；Coder回答“小猫”并坚持任务边界；Verifier不再接单。

### 7.3 Git验证

- 文档与运行时配置分步骤验证；
- 精确暂存，不使用 `git add .` 或 `git add -A`；
- 不把角色文档与业务代码混在同一提交；
- push 后立即执行 `git log origin/main --oneline -1`；
- 若本次纯文档治理不会触发生产部署，则明确记录；若触发 Vercel，则核对最新部署提交与状态。

## 8. 风险与控制

### 风险一：旧映射残留

通过对现行目录、项目根规则和运行时提示做定向扫描控制。历史归档不计入残留，因为其内容必须保持原样。

### 风险二：Mavis仍按旧“小酷”权限行动

先安装小M运行时身份守卫，再向 Mavis 派发任何任务；冒烟测试未通过前不允许其接单。

### 风险三：Codex新会话未继承“小酷”

将身份写入项目根指令，并让其引用单一 `docs/roles/xiaoku/` 岗位源；用全新 Codex 会话验证。

### 风险四：Verifier退役不彻底

从现行任务路由和角色映射删除 verifier，并在无法删除平台级 Agent 时安装退役守卫。退役后不以任何名义向其派单。

### 风险五：中断现场混入正式提交

当前未提交的 `AGENTS.md` 修改和归档 README 必须在实施计划中逐项归属；每次提交前使用精确路径与 `git diff --cached --name-only` 核对范围。

## 9. 完成标准

以下条件全部满足才算改岗完成：

- 所有本项目 Codex 会话正式担任小酷；
- `mavis / Mavis` 正式担任小M并通过只读身份测试；
- `coder / Coder` 正式担任小猫并通过执行边界测试；
- `verifier / Verifier` 已从现行体系退役；
- 新旧岗位文件不再混用；
- 历史证据完整归档；
- 静态扫描、仓库验证和 Git 验证全部通过；
- 胡子老师完成最终身份验收。
