# 商品详情余额不足引导与待支付订单恢复设计

**日期**：2026-07-14
**状态**：已完成业务决策，等待胡子老师审核书面规格
**范围**：商品详情“立即购买”支付链路、共用结算弹窗、订单详情接口安全白名单

## 1. 背景与目标

购物车已经支持余额不足时引导用户将可用收益转入购物余额，但商品详情页的“立即购买”仍只提示余额不足。用户前往充值后返回商品页，也可能因为页面状态丢失而再次创建订单。

本次目标：

1. 商品详情支付余额不足时，复用购物车已有的收益转余额能力。
2. 保留首次创建的待支付订单，禁止重复创建。
3. 用户前往充值再返回时，可以恢复并继续支付原订单。
4. 待支付订单已创建后锁定收货信息，避免页面允许修改但订单实际不更新。
5. 购物车和商品详情页共用相同的收货信息锁定规则。
6. 不自动支付、不绕过支付密码、不修改支付事务、库存或奖励逻辑。
7. 同时关闭现有订单详情接口可能返回完整用户对象的敏感字段泄露风险。

## 2. 已确认业务规则

### 2.1 支付与订单

- 首次确认时创建一次订单，并立即调用现有支付验证接口。
- 余额不足后保存原订单编号和余额缺口。
- 再次确认必须复用原订单，不再调用订单创建接口。
- 收益转入成功后只刷新资金信息和缺口，不自动重新支付。
- 用户必须再次输入支付密码并手动确认支付。
- 支付成功后清除该商品对应的待支付会话记录。

### 2.2 收益与充值

- 可用收益大于零时打开共用 `EarningsTransferModal（收益转余额弹窗）`。
- 默认转入金额为 `min(shortage, earningsAvailable)`，即余额缺口和可用收益中的较小值。
- 收益不足时允许部分转入，并准确显示剩余缺口。
- 没有可用收益时显示缺口和“去充值”入口。
- 前往充值前必须成功写入会话记录；写入失败时不跳转并提示错误。
- 从充值页返回后先验证原订单，验证完成前禁止创建新订单。

### 2.3 收货信息

- 待支付订单存在时，锁定已保存地址选择、收货人、手机号、地区和详细地址。
- 显示“订单已创建，收货信息以首次提交为准”。
- 支付密码输入框保持可用。
- 购物车和商品详情页使用同一锁定规则。

## 3. 安全前置修复

### 3.1 已发现问题

`OrderService.getOrderDetail()` 当前使用 `user: true` 查询完整用户对象，`GET /api/orders/[id]` 又直接返回订单数据。这可能把 `passwordHash（登录密码哈希）`、`paymentPasswordHash（支付密码哈希）` 等敏感字段带入用户端响应。

该问题属于 P0，必须在待支付订单恢复功能之前修复。

### 3.2 修复方式

将 `OrderService.getOrderDetail()` 改为 Prisma `select（字段白名单）`：

- 订单内部允许查询：`id`、`userId`、`orderNo`、金额字段、积分字段、状态、物流时间、创建时间和收货信息。
- 订单商品允许查询：商品项编号、商品编号、数量、单价、总价，以及商品的 `id`、`name`、`imageUrl`。
- 退款记录仅查询现有订单详情页真实使用的字段。
- 不查询 `user` 关系。
- 不查询密码哈希、余额、收益、角色、令牌或其他用户敏感字段。
- `userId` 仅供服务端鉴权使用，不进入客户端响应。

`GET /api/orders/[id]` 在确认所有权后，使用显式响应映射逐项构造公开订单对象。公开对象不包含 `userId`，未来服务层新增字段时也不能自动进入响应。

### 3.3 兼容性

需要核对并回归以下真实调用页面：

- `/dashboard/orders/[id]` 用户订单详情页；
- `/payment/order/[orderId]` 支付页；
- 新增的商品详情待支付订单恢复流程。

订单取消、确认收货等服务端调用仍可使用服务层返回的 `userId` 做权限校验。

## 4. 总体方案

采用渐进式共用方案：

1. 共用结算弹窗增加待支付订单字段锁定能力。
2. 商品详情页增加待支付订单状态、收益转余额弹窗和会话恢复。
3. 新增小型会话存储工具，统一校验、读写和清除。
4. 购物车继续使用现有 `pendingPayments（按商品区分的待支付字典）`，本包不增加购物车会话持久化。
5. 不新增数据库表，不修改 Prisma Schema（数据库结构），不修改支付事务。

- 服务端内部保留 `userId`，不查询完整 `user`。
- 保持商品项和退款记录满足现有页面需要。

- GET 路由继续使用现有令牌验证和所有权校验。
- 对外响应显式删除 `userId`。
- 响应中不得出现 `user`、`passwordHash`、`paymentPasswordHash` 或其他用户敏感字段。
- POST、PUT、DELETE 资金和状态逻辑不在本次修改范围。

### 5.3 `src/lib/utils/pending-payment-session.ts`（新增）

定义商品待支付会话结构：

```typescript
interface ProductPendingPaymentSession {
  version: 1
  userId: string
  productId: string
  orderId: string
  shortage: number
}
```

职责：

- 根据 `userId + productId` 生成隔离键。
- 写入前校验所有字符串非空、`shortage` 为有限且不小于零的数字。
- 读取时捕获 JSON 解析错误并清除损坏数据。
- 不存支付密码、登录令牌、完整收货地址或用户资金信息。
- 不在会话里复制 `payAmount` 或 `pointsUsed`；恢复时必须以服务端订单为准，页面内存只保存本次已验证的值。
- `sessionStorage（当前标签页会话存储）` 不可用或写入失败时返回明确失败结果，不吞错。

### 5.4 `src/components/checkout/CheckoutDialog.tsx`

- 复用现有 `hasPendingOrder` 参数，不增加重复状态源。
- 增加 `pendingOrderShipping` 可选参数，字段为 `recipientName`、`recipientPhone` 和 `shippingAddress`，仅用于恢复原订单的收货快照。

- `hasPendingOrder=true` 时隐藏地址簿和可编辑收货表单，改为显示收货人、手机号和完整地址的只读摘要；不解析或重组已保存地址，避免展示错误。
- 待支付订单清除后恢复正常可编辑收货表单。
- 支付密码输入和忘记支付密码帮助保持可用。
- 增加明确提示：“订单已创建，收货信息以首次提交为准”。
- 现有购物车已经传入 `hasPendingOrder`，因此自动获得一致锁定行为。

### 5.5 `src/app/products/[id]/page.tsx`

- 用户资金类型补充 `id` 和 `earningsAvailable`。
- 增加：
  - 当前待支付订单；
  - 原订单应付金额；
  - 当前缺口；
  - 收益转余额弹窗状态；
  - 恢复状态；
  - 恢复错误信息。
- 首次确认创建订单；已有恢复订单时跳过创建，直接支付原订单。
- 识别 `INSUFFICIENT_BALANCE（余额不足错误码）`，字符串仅作为兼容兜底。
- 余额不足时立即写入页面状态和会话记录。
- 收益转入成功后刷新用户资金，始终用原订单 `payAmount（应付金额）` 减最新购物余额重算缺口；若用户资金刷新失败，才按实际转入金额减少旧缺口。
- 恢复原订单时同步恢复 `pointsUsed（已使用积分）`，页面展示金额不得覆盖原订单应付金额。
- “去充值”前再次确认会话写入成功。
- 支付成功后清除会话，再跳转订单详情。
- 不在收益转入成功回调中调用支付接口。

## 6. 恢复状态机

商品详情页使用四种明确状态：

| 状态 | 含义 | 是否允许创建新订单 |
|---|---|---|
| `idle` | 没有待恢复会话 | 是 |
| `validating` | 正在向服务端验证原订单 | 否 |
| `restored` | 原订单验证成功并已恢复 | 否，只能复用原订单支付 |
| `validation_error` | 网络或服务端临时失败 | 否，必须重新验证 |

页面只在用户信息和商品信息都已加载后读取对应会话键。

## 7. 服务端恢复验证

恢复时调用修复后的 `GET /api/orders/[orderId]`，必须同时满足：

1. 接口鉴权成功；
2. 订单属于当前用户（由服务端校验）；
3. 订单状态为 `pending`；
4. 订单商品项包含当前 `productId`；
5. 订单商品数量和商品上下文符合“立即购买”的单商品订单约束。

验证成功后：

- 使用订单中的 `recipientName`、`recipientPhone` 和 `shippingAddress` 恢复展示；
- 打开结算弹窗时锁定收货信息；
- 恢复原订单编号、`pointsUsed` 和 `payAmount`，并按 `max(0, order.payAmount - latestUser.balance)` 重新计算当前缺口；
- 用重新计算后的缺口更新页面状态和会话记录，避免充值后继续显示旧缺口；
- 显示“已恢复待支付订单，不会重复下单”。

验证明确无效时清除会话：

- 404 订单不存在；
- 403 无权访问；
- 订单已经支付、取消或不是待支付；
- 商品不匹配；
- 会话格式损坏。

网络超时、接口 500 等临时错误：

- 不清除会话；
- 进入 `validation_error`；
- 显示“待支付订单验证失败，请重试”；
- 提供“重新验证”按钮；
- 验证成功前禁止新建订单。

## 8. 数据流

### 8.1 首次下单余额不足

```text
确认下单
  -> POST /api/orders 创建订单
  -> 保存 orderId，shortage 初始为 0
  -> POST /api/orders/{id}/verify-payment
  -> 返回 INSUFFICIENT_BALANCE
  -> 保存服务端 shortage
  -> 写入 sessionStorage
  -> 有收益：打开收益转余额弹窗
  -> 无收益：显示缺口和去充值
```

### 8.2 收益转入

```text
收益转入成功(amount)
  -> 重新读取 /api/users/me
  -> shortage = max(0, 原订单 payAmount - 最新购物余额)
  -> 若刷新失败，才使用 max(0, shortage - amount)
  -> 更新页面状态和 sessionStorage
  -> 关闭收益弹窗
  -> 提示用户手动重新确认支付
```

严禁在该回调中自动调用支付接口。

### 8.3 充值往返

```text
点击去充值
  -> 确认会话记录已成功写入
  -> 跳转 /dashboard/recharge
  -> 用户返回商品详情
  -> 读取会话
  -> GET /api/orders/{id} 验证
  -> 用订单应付金额减最新购物余额重新计算缺口
  -> 恢复原订单、原积分抵扣、原应付金额和收货信息
  -> 用户手动重新确认支付
```

### 8.4 支付成功

```text
verify-payment 成功
  -> 清除当前 userId + productId 的会话记录
  -> 关闭弹窗
  -> 跳转订单详情
```

## 9. 错误处理

- 会话写入失败：不前往充值，提示“待支付订单保存失败，请重试”。
- 会话解析失败：清除损坏记录，恢复普通下单。
- 订单明确失效：清除记录并提示订单状态变化。
- 订单验证临时失败：保留记录、禁止新建、允许重试。
- 收益转入失败：收益弹窗保持打开，显示后端错误。
- 支付密码错误：保持原订单和缺口，不清除会话。
- 支付成功但清除会话失败：记录错误但不回滚支付；下一次恢复时服务端状态会判定订单已支付并清理。
- 所有未知错误必须有用户可见提示，不允许静默失败。
