# 退款凭证校验与多次申请历史设计

> 日期：2026-07-15
> 状态：胡子老师已批准业务方案，等待设计文档确认
> 范围：用户退款申请、后台退款审核弹窗、前后端校验与自动化测试

## 一、问题背景

当前退款流程存在两个业务缺口：

1. 用户因“质量问题”或“商品损坏”申请退款时，即使没有上传图片也能提交；退款被拒后再次申请，同样可以不上传任何新凭证。
2. 后台审核弹窗只展示当前一条退款申请，审核员无法在同一弹窗中查看该订单第 1 次、第 2 次等历史申请的文字、图片、状态和管理员备注。

数据库已经按每次申请分别保存 `RefundRequest` 记录，并且每条记录已有 `images`、`description`、`adminComment`、`status` 和 `createdAt` 字段，因此本次不需要修改数据库结构。

## 二、目标

1. 对需要实物证据的退款原因强制要求图片。
2. 每次退款申请独立保存凭证，不自动继承上一次图片。
3. 后台审核时能够查看当前申请及该订单全部历史申请。
4. 后台拒绝退款时必须填写明确的拒绝原因。
5. 前端和后端执行一致校验，不能通过直接调用接口绕过。
6. 不改变退款金额、审核状态流转、退款完成、资金退回和通知链路。

## 三、业务规则

### 3.1 用户申请规则

| 退款原因 | 图片要求 | 补充说明要求 |
|---|---|---|
| 质量问题 | 至少 1 张 | 可选 |
| 商品损坏 | 至少 1 张 | 可选 |
| 未按约定时间发货 | 可选 | 可选 |
| 其他 | 可选 | 必填，去除首尾空格后不得为空 |

通用规则：

- 每次申请最多上传 5 张图片。
- 沿用现有图片格式和单张大小限制。
- 重新申请时表单从空状态开始，不自动带入任何历史图片。
- 当前申请上传的图片只属于当前这一次申请。
- 前端校验失败时不得发送退款申请请求。
- 后端校验失败时不得创建 `RefundRequest`、不得写操作日志、不得发送通知。

### 3.2 后台审核规则

- 审核通过时，管理员备注可选。
- 审核拒绝时，管理员备注作为拒绝原因，去除首尾空格后至少 5 个字符。
- 前端在拒绝原因不足 5 个字符时禁用或拦截提交，并显示明确提示。
- 后端审核接口执行相同校验，防止绕过前端。
- 已有审核权限和状态流转规则保持不变。

## 四、用户端设计

目标页面：

```text
src/app/dashboard/orders/[id]/refund/page.tsx
```

### 4.1 动态必填提示

- 当退款原因是“质量问题”或“商品损坏”时，“上传凭证”标题显示必填标记。
- 图片区域显示提示：“该退款原因至少需要上传 1 张凭证图片”。
- 没有图片时点击提交，显示红色错误提示，不发送接口请求。
- 当退款原因切换为无需图片的类型时，已上传图片保留，图片要求提示消失。

### 4.2 “其他”原因

- 选择“其他”后，补充说明显示必填标记。
- 补充说明去除首尾空格后为空时，不允许提交。

### 4.3 重新申请

- 从已拒绝状态点击“重新申请”进入现有退款申请页面。
- 不读取、不复制、不预填上一次申请的图片或文字。
- 按本次选择的退款原因重新执行完整校验。

## 五、退款申请接口设计

目标接口：

```text
POST /api/orders/[id]/refund
```

在现有登录、订单归属、订单状态和重复申请拦截之后，对请求体增加结构化校验：

1. `reason` 必须是非空字符串。
2. `images` 未提供时按空数组处理；提供时必须是字符串数组。
3. 图片数组最多 5 项，每一项必须是非空字符串。
4. `reason` 为“质量问题”或“商品损坏”时，`images.length >= 1`。
5. `reason` 为“其他”时，`description` 去除首尾空格后必须非空。

校验错误返回 400，并使用用户可理解的中文错误信息。校验通过后的创建、金额、通知和成功响应逻辑保持原样。

## 六、后台审核弹窗设计

目标页面：

```text
src/app/admin/refunds/page.tsx
```

### 6.1 当前申请

弹窗顶部完整展示当前待审核申请：

- 标题注明“第 N 次申请”。
- 订单号、用户、退款金额。
- 退款原因、补充说明、申请时间。
- 当前申请的全部凭证图片，图片可点击查看原图。
- 管理员备注输入框及审核按钮。

### 6.2 历史申请

当前申请下方增加原生折叠区域：

```text
历史申请记录（共 N-1 次）
```

- 当前申请之外没有历史记录时，不显示该区域。
- 默认折叠，点击后展开。
- 每次历史申请独立显示：申请序号、时间、状态、退款原因、补充说明、当次图片、管理员备注。
- 历史记录按申请时间从新到旧展示，但申请序号按真实时间顺序计算，例如最早记录为“第 1 次申请”。
- 当前申请与历史记录不得混合图片或文字。

### 6.3 加载状态和失败处理

- 打开审核弹窗时，使用当前登录令牌加载该订单全部退款申请。
- 加载期间显示“正在加载历史申请”。
- 历史加载完成前禁用审核提交按钮。
- 加载失败时显示“历史申请加载失败”和“重新获取”按钮。
- 加载失败时不得继续通过或拒绝，避免审核员在缺少历史证据时操作。
- 关闭弹窗或切换退款记录时，清空上一次历史记录、错误和加载状态，防止状态串台。

## 七、历史数据获取方案

复用现有接口：

```text
GET /api/orders/[orderId]/refund
```

理由：

- 该接口已返回同一订单全部退款申请。
- 已按 `createdAt desc` 排序。
- 已允许超级管理员和财务管理员读取。
- 不需要新增接口或数据库结构。

后台页面调用时必须携带：

```text
Authorization: Bearer <token>
```

不能使用当前退款列表分页数据代替历史查询，因为同一订单的旧申请可能不在当前分页中。

## 八、后台审核接口设计

目标接口：

```text
POST /api/admin/refunds/[id]/review
```

保持现有鉴权、状态校验、状态更新、日志和通知流程，只增加拒绝原因校验：

- `action === 'reject'` 时，`adminComment.trim().length >= 5`。
- 不满足时返回 400，不更新退款状态、不写日志、不发通知。
- `action === 'approve'` 时备注继续可选。

## 九、数据流

### 9.1 用户申请

```text
选择原因
  -> 前端判断图片或说明是否必填
  -> 上传当前申请图片
  -> 前端校验
  -> POST 退款申请
  -> 后端重复执行同一业务校验
  -> 创建独立 RefundRequest
  -> 发送现有退款申请通知
```

### 9.2 后台审核

```text
点击通过或拒绝
  -> 打开当前申请弹窗
  -> GET 同一订单全部退款申请
  -> 计算当前是第 N 次申请
  -> 展示当前申请 + 折叠历史
  -> 管理员填写备注
  -> 前端审核校验
  -> POST 审核接口
  -> 后端再次校验
  -> 执行现有审核、日志和通知流程
```

## 十、错误处理

| 场景 | 处理方式 |
|---|---|
| 质量问题或商品损坏没有图片 | 前端阻止；后端返回 400 |
| “其他”没有补充说明 | 前端阻止；后端返回 400 |
| 图片字段不是字符串数组 | 后端返回 400 |
| 图片超过 5 张 | 后端返回 400 |
| 历史记录加载失败 | 弹窗显示重试，审核按钮禁用 |
| 拒绝原因不足 5 个字符 | 前端阻止；后端返回 400 |
| 历史记录为空 | 正常审核当前申请，不显示历史区域 |

## 十一、测试设计

### 11.1 用户申请接口

- 质量问题无图片返回 400，且不创建、不通知。
- 商品损坏无图片返回 400，且不创建、不通知。
- 质量问题有 1 张图片可以创建。
- 未按约定时间发货无图片可以创建。
- 其他无补充说明返回 400。
- 图片不是数组、包含空字符串或超过 5 张时返回 400。
- 现有 pending、approved 防重复规则保持通过。

### 11.2 用户申请页面

- 两种证据型原因显示图片必填标记和提示。
- 无图片时不调用 `fetch`。
- “其他”无补充说明时不调用 `fetch`。
- 重新进入页面时不继承上一次图片。

### 11.3 后台审核页面

- 真实加载并展示两次申请的不同文字和图片。
- 当前申请显示“第 2 次申请”。
- 历史折叠区显示“第 1 次申请”。
- 历史加载期间和失败时审核按钮禁用。
- 切换订单时不残留上一订单历史。
- 图片属于各自申请，不交叉显示。

### 11.4 审核接口

- 拒绝备注不足 5 个字符返回 400，且不更新、不写日志、不通知。
- 拒绝备注满足要求保持原流程。
- 通过时备注可选，原流程不变。

### 11.5 回归测试

- 退款金额、审核通过、审核拒绝、退款完成、资金回退和通知测试全部通过。
- 类型检查、代码规范检查、完整测试、覆盖率和生产构建通过。
- 登录态浏览器验证用户申请页和后台多次申请历史弹窗。

## 十二、修改边界

允许修改范围将在实施计划中精确列出，原则上仅包含：

- 用户退款申请页面；
- 用户退款申请接口；
- 后台退款管理页面；
- 后台退款审核接口；
- 对应测试文件。

明确禁止：

- 修改 `prisma/schema.prisma` 或执行迁移；
- 修改退款金额计算；
- 修改退款完成和资金退回逻辑；
- 修改通知模板或通知链路；
- 修改订单支付逻辑；
- 新增依赖或修改测试配置；
- 顺带重构无关模块。

## 十三、验收标准

1. 质量问题和商品损坏没有图片时，前端和后端均拒绝。
2. 未按约定时间发货不上传图片也能提交。
3. 其他原因没有补充说明时，前端和后端均拒绝。
4. 重新申请不继承旧图片。
5. 后台审核弹窗能区分并查看第 1 次、第 2 次等全部申请的文字和图片。
6. 当前申请和历史申请不会串数据。
7. 拒绝原因少于 5 个字符不能提交。
8. 历史记录加载失败时不能审核。
9. 不新增数据库结构，不改变退款资金和通知链路。
10. 自动化验证和真实浏览器验收通过。
