# 小M与小猫上岗及协作流程设计

> 日期：2026-07-18  
> 负责人：小酷（`mavis`）  
> 拍板人：胡子哥  
> 状态：待胡子哥书面复核

## 1. 目标

建立不会串角色的四角色协作体系：

- 胡子哥：唯一业务拍板人和真实验收人；
- 小酷（Agent ID：`mavis`）：主理人、核心开发和调度者；
- 小M（Agent ID：`verifier`）：独立只读复审员；
- 小猫（Agent ID：`coder`）：辅助执行者。

同时把原业务角色名“猫爪/猫抓”和目录名 `catpaw` 全量迁移为“小猫”和 `xiaomao`，包括历史 001～039 号任务与结果。

## 2. 已确认问题

### 2.1 不能用聊天提示词覆盖运行时身份

小M岗位提示词被发送到 `mavis` 会话后，`mavis` 正确报告自己是小酷并拒绝冒充 `verifier`。这不是小M反对岗位说明，而是提示词发给了错误的 Agent。

完整身份必须由对应 Agent 的运行时 `project-prompt.md` 自动加载：

- `mavis` 只能是小酷；
- `verifier` 只能是小M；
- `coder` 只能是小猫。

聊天中的“上岗启动指令”只能核验身份、触发阅读和考核，不能重新指定身份。

### 2.2 小猫当前尚未安装完成

当前 `~/.mavis/agents/coder/project-prompt.md` 仍定义业务角色为“猫爪”，并引用 `catpaw/todo/猫爪_NNN号任务.md`。

因此现在直接要求 Coder 以“小猫”上岗，会出现两种保护性反馈：

1. 拒绝身份覆盖；
2. 把岗位提示词误判为研发需求，要求提供具体代码任务。

这两种反馈都说明必须先更新运行时提示词，再重启 Coder。

## 3. 采用方案

采用“运行时身份先安装、历史迁移后执行”的两阶段方案。

### 阶段一：小酷完成最小启动改造

由小酷亲自完成，不能派给尚未上岗的小猫：

1. 更新 `~/.mavis/agents/coder/project-prompt.md`，把业务角色改为“小猫”；
2. 更新 `mavis` 和 `verifier` 的运行时提示词，把协作对象“猫爪”改为“小猫”；
3. 创建 `docs/agent-tasks/xiaomao/` 基础目录；
4. 创建小猫岗位说明和小猫 System Prompt；
5. 创建 `小猫_040号任务.md`；
6. 更新 `AGENTS.md` 和当前协作 README 的现行角色映射；
7. 不在这个阶段批量改写 001～039 历史文件，控制启动改造范围。

完成后必须关闭旧 Coder 会话并重新启动，确保新的运行时提示词被加载。

### 阶段二：小猫上岗后执行全量历史迁移

小猫通过胡子哥五题考核后，执行小猫040号任务：

1. 把 `docs/agent-tasks/catpaw/{todo,done,archived}/` 中的历史文件逐个迁移到已经创建好的 `docs/agent-tasks/xiaomao/` 对应目录；
2. 把历史 001～039 号文件名中的“猫爪”改成“小猫”；
3. 把文件正文中的“猫爪、猫抓、catpaw”改成“小猫、xiaomao”；
4. 更新全仓库所有有效路径引用；
5. 将旧目录中的岗位说明和 System Prompt 与阶段一的新版本对照后合并，不允许用旧版覆盖新版；
6. 删除已经清空的旧 `catpaw` 目录；
7. 保持任务编号、业务结论和Git历史可追踪；
8. 输出小猫040号结果文件；
9. 默认不push，由小酷审核后处理提交与推送。

## 4. 上岗机制

### 4.1 身份核验原则

每个 Agent 启动后，第一步只报告系统实际加载的：

- Agent ID；
- DisplayName；
- 业务角色；
- 岗位定位。

发现不匹配必须停止，不能按聊天提示词冒充其他角色。

### 4.2 小M上岗

启动真正的 `verifier` 会话后：

1. 核验 `verifier / Verifier / 小M`；
2. 读取 `AGENTS.md`；
3. 读取协作 README；
4. 读取小M岗位说明和 System Prompt；
5. 阅读最近5份真实复审结果；
6. 等待胡子哥五题考核；
7. 胡子哥明确宣布“小M上岗通过”后才能接任务。

上岗阶段不执行todo任务，不修改代码，不commit、不push、不部署。

### 4.3 小猫上岗

只有阶段一完成并重启 `coder` 后才允许开始：

1. 核验 `coder / Coder / 小猫`；
2. 读取 `AGENTS.md`；
3. 读取协作 README；
4. 读取小猫岗位说明和 System Prompt；
5. 等待胡子哥五题考核；
6. 胡子哥明确宣布“小猫上岗通过”后，才能领取小猫040号任务。

上岗启动指令必须明确：这是项目治理和身份核验任务，不是代码需求，不进入需求规格设计，不使用功能开发流程。

## 5. 岗位权限

### 5.1 小M

职责：

- 独立查看源码和diff；
- 独立运行测试、类型检查和构建；
- 检查资金边界、权限绕过、测试可信度和文件范围；
- 输出“通过、有条件通过、不通过”三态结论。

禁止：

- 修改业务代码和测试实现；
- `git add`、commit、push和部署；
- 自己修复自己发现的问题；
- 给小猫派任务；
- 替胡子哥做业务拍板。

唯一项目写入例外是任务明确要求的复审结果文件。

### 5.2 小猫

职责：

- 文档整理；
- 文件迁移和归档；
- 文本替换；
- 清理临时文件；
- 按任务补测试；
- 纯CSS或格式调整；
- 执行明确授权的机械任务。

禁止：

- 做业务决策；
- 默认修改核心业务、资金、权限和数据库结构；
- 默认commit、push和部署；
- 扩大任务范围；
- 覆盖他人已有修改；
- 自动执行其他todo任务。

## 6. 下一步任务安排

级别：A（协作机制和大量文档路径变更）。

负责人：小猫。

独立复审：小M040号。

- `catpaw` 目录迁移为 `xiaomao`；
- 有效文档中不存在未说明的“猫爪、猫抓、catpaw”残留；

### 小M040号：小猫改名独立复审

- 文件和编号完整性；

- 越界修改；
- 历史业务结论是否被误改；
- 运行时提示词是否与仓库文档一致。

### 小M041号：后台操作权限安全审计

级别：P。

重点：

- `support_admin` 是否能直接调用写接口；
- 是否能修改等级、状态和角色；
- 动态操作权限是否只在前端生效；
- `points_admin` 是否为幽灵角色；
- 前端按钮权限与后端授权是否一致。

审计结果交给小酷，由小酷实施P级修复，再由小M提交前复审。

## 7. 提交顺序

1. 设计文档独立提交；
2. 小酷最小启动改造独立提交；
3. 小猫040号历史迁移独立提交；
4. 小M040号复审结果和必要修正独立提交；
5. 小M041号安全审计独立提交；
6. 后端权限修复另走P级设计、TDD和提交前复审。

任务文档和业务代码不能混在同一个提交。

## 8. 验证

角色改名阶段至少执行：

```powershell
rg -n "猫爪|猫抓|catpaw" AGENTS.md docs
git status --short
git diff --summary
git diff --name-only
```

运行时提示词单独扫描：

```powershell
rg -n "猫爪|猫抓|catpaw" `
  "$HOME\.mavis\agents\mavis\project-prompt.md" `
  "$HOME\.mavis\agents\verifier\project-prompt.md" `
  "$HOME\.mavis\agents\coder\project-prompt.md"
```

文档改名不等于代码功能变化；如果没有修改业务代码，测试和构建应标明“非本任务强制项”。若改名涉及测试路径或脚本引用，则必须运行受影响测试和类型检查。
