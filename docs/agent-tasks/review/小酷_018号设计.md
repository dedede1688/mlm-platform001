# 小酷_018号设计：忘记支付密码的后台人工重置流程

## 一、目标

为忘记支付密码的用户提供一条安全、可审计、无需真实短信接口即可落地的处理路径：

1. 用户在结算弹窗点击“忘记支付密码”。
2. 系统提示用户联系客服完成身份核验。
3. 超级管理员在会员详情的“资金账户”标签中清除旧支付密码。
4. 用户收到站内通知后，自行设置新的 6 位支付密码。
5. 用户重新发起支付，系统不得自动扣款。

## 二、已确认的业务规则

- 只允许超级管理员重置支付密码。
- 管理员只能清除旧支付密码，不能查看或代用户设置新支付密码。
- 重置前必须填写不少于 5 个字的原因。
- 重置前必须输入目标用户手机号后 4 位。
- 重置成功后必须写操作日志并发送站内通知。
- 通知或操作日志失败不得把已经成功的密码清除伪装成接口失败。
- 支付密码未设置时，用户通过现有页面自行设置新密码。
- 不修改登录密码、余额、收益、订单、支付记录或数据库结构。

## 三、后台流程

入口：`后台 -> 会员管理 -> 会员详情 -> 资金账户 -> 支付安全`

“支付安全”区域展示：

- 支付密码状态：已设置 / 未设置。
- 超级管理员可见“重置支付密码”按钮。
- 其他管理员只能查看状态，不显示执行按钮。

二次确认弹窗展示用户手机号、昵称和当前状态，并要求输入：

- 重置原因，去除首尾空格后不少于 5 个字。
- 用户手机号后 4 位，必须是 4 位数字并与目标用户匹配。

## 四、后端设计

新增接口：

```text
POST /api/admin/users/[id]/payment-password/reset
```

请求体：

```json
{
  "reason": "用户本人联系客服申请重置",
  "phoneSuffix": "8001"
}
```

处理顺序：

1. `verifyPermission(request, ['super_admin'])` 校验超级管理员权限。
2. 校验原因和手机号后 4 位。
3. 查询目标用户，拒绝不存在或已删除用户。
4. 拒绝手机号后 4 位不匹配。
5. 拒绝本来就未设置支付密码的用户。
6. 使用带条件的 `updateMany` 把 `paymentPasswordHash` 清空，防止并发重复重置。
7. 写操作日志，不记录任何明文密码。
8. 调用通知服务并等待其结束；通知内部捕获错误，不影响主响应。
9. 返回“支付密码已重置，请通知用户重新设置”。

## 五、数据展示与最小泄露

后台会员列表接口增加 `hasPaymentPassword` 布尔值。查询时可以读取 `paymentPasswordHash`，但响应映射必须删除该字段，只返回：

```json
{
  "hasPaymentPassword": true
}
```

测试必须证明响应中不存在 `paymentPasswordHash`。

## 六、用户端流程

- 已设置支付密码：显示“忘记支付密码？”按钮。
- 点击后打开当前页面内的说明弹窗，不跳到修改密码页面。
- 弹窗说明需要联系客服核验身份，提供“联系客服”和“取消”。
- “联系客服”进入 `/help`（帮助中心）查看客服信息。
- 未设置支付密码：显示“立即设置”，进入 `/dashboard/payment-password`。
- 设置完成后由用户手动重新支付，不得自动调用支付接口。

## 七、通知与审计

新增站内通知模板 `payment_password_reset`，建议文案：

```text
您的支付密码已由平台重置。为保障账户资金安全，请尽快前往个人中心重新设置支付密码。如非本人申请，请立即联系客服。
```

操作日志至少包含：管理员编号、目标用户编号、动作、原因、时间、网络地址和浏览器信息。禁止记录旧支付密码、新支付密码或密码加密值。

## 八、验收标准

- 非超级管理员不能调用重置接口。
- 手机号后 4 位错误、原因过短、用户不存在、用户未设置支付密码都不能重置。
- 成功后只有 `paymentPasswordHash` 变为空，其他资金和身份字段不变。
- 管理员接口只返回支付密码状态，不泄露加密值。
- 通知失败仍返回重置成功。
- 结算弹窗不再把忘记密码用户带到要求旧密码的修改页面。
- 用户重新设置后必须手动再次支付。
- 不修改数据库结构，不执行迁移。
