## 五、P1-A：提交按钮禁用状态

### 5.1 目标行为

以下任一情况，提交按钮必须处于 disabled（禁用）状态：

- 未选择退款原因。
- `质量问题`没有图片。
- `商品损坏`没有图片。
- `其他`没有有效补充说明。
- 正在提交。

以下情况按钮应允许点击：

- `未按约定时间发货`，无图片、无说明。
- `质量问题`或`商品损坏`且至少有 1 张合法图片。
- `其他`且补充说明 trim 后非空。

`handleSubmit` 内现有的二次共享校验必须保留，不能因为按钮禁用而删除。

### 5.2 推荐最小实现

共享校验很轻量，不需要新增依赖。可在组件渲染期间计算：

```typescript
const currentValidation = validateRefundApplication(form)
const formInvalid = !currentValidation.success
```

按钮使用：

```tsx
disabled={submitting || formInvalid}
```

`handleSubmit` 内仍然重新执行：

```typescript
const validation = validateRefundApplication(form)
```

不要把 handler 改成只复用渲染时对象，避免未来异步路径或重构时丢失程序性二次拦截。

## 六、P1-B：加强空表单测试证据

### 6.1 测试环境

项目已经具备：

- `@vitejs/plugin-react`
- `react-dom/server`
- Vitest 的 React 转换能力

不得新增依赖、不得修改 `vitest.config.ts`、不得修改 package 或锁文件。

测试文件保持 `.test.ts`，如需渲染 JSX，使用：

```typescript
React.createElement(...)
renderToStaticMarkup(...)
```

### 6.2 必须增加的证据

1. 真实导入 `RefundApplyPage`，不得 mock 被测组件。
2. 只 mock 外部边界，例如：
   - `next/navigation`
   - `next/image`
   - Supabase 浏览器客户端
3. 使用 `renderToStaticMarkup` 渲染组件初始状态。
4. 从完整按钮元素集合中按“提交申请”文字精确找到唯一按钮，断言它是 disabled。
5. 断言初始 textarea（补充说明）为空。
6. 断言初始页面没有退款凭证预览图片。
7. 断言初始原因是“请选择退款原因”。
8. 提取所有 `useEffect` 完整函数块，确认初始化 effect 不调用 `setForm` 写入历史数据。
9. 确认页面没有从 URL 查询参数、订单详情 GET 请求、localStorage 或 sessionStorage 读取退款说明和图片。
   - `localStorage.getItem('token')` 是合法的鉴权读取，不得误报。
10. 保留现有“切换原因不清空当前图片”测试。

### 6.3 测试可信度规则

- 禁止 `if (match)` 静默跳过。
- 禁止 `A || B` 宽松断言。
- 禁止固定 200/300/500 字符窗口。
- 按钮必须先枚举完整 `<button>...</button>`，再按文字筛选，并断言恰好 1 个。
- 函数与 effect 提取必须使用括号或花括号平衡逻辑。
- 不得只靠 `refundRequest`、`previousRefund` 两个变量名不存在证明业务结论。
