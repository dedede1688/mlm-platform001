# 猫爪_005号任务：后台收益作废功能

## 一、任务背景

胡子哥在后台会员详情的「资金账户」里测试可用收益调整时，输入 `-40` 被系统拦截。

这是当前设计的安全限制：上一包只允许「增加可用收益」，不允许用负数直接减少收益。

现在需要新增一个明确、安全、可追踪的功能：

```text
收益作废
```

业务含义：

```text
从用户可用收益中扣除一笔金额，并同时增加到作废收益中。
```

也就是：

```text
earningsAvailable 减少
earningsVoided 增加
```

不要做“负数扣减可用收益”，要做明确的「作废收益」操作。

## 三、任务目标

实现后台会员收益作废能力：

1. 后台会员详情 → 资金账户 → 资金调整中新增「作废收益」选项。
2. 管理员输入正数金额，例如 `40`。
3. 后端把用户 `earningsAvailable` 减少 40。

## 六、后端实现要求

### 1. 新增调账类型

在后台用户余额调整接口中新增类型：

```text
earnings_void
```

中文含义：

```text
作废收益
```

注意：项目后端可能已经有 `earnings_void` 的部分定义，请先搜索确认。如果已有，就补齐逻辑，不要重复造新类型。

### 2. 资金变化规则

当 `type = earnings_void` 时：

```text
earningsAvailable -= amount
earningsVoided += amount
```

要求：

- amount 必须是正数。
- amount 不能大于当前 `earningsAvailable`。
- 如果可用收益不足，返回失败。
- 不影响 `balance`。
- 不影响 `frozenBalance`。
- 不影响 `earningsFrozen`。
- 不影响 `earningsPending`。

### 3. 并发保护

必须使用数据库层面的条件更新，防止并发扣成负数。

推荐方式：

```typescript
updateMany({
  where: {
    id: userId,
    earningsAvailable: { gte: amount },
  },
  data: {
    earningsAvailable: { decrement: amount },
    earningsVoided: { increment: amount },
  },
})
```

如果 `count === 0`，说明可用收益不足或状态变化，应返回失败。

### 4. BalanceRecord（资金流水）

必须写入资金流水。

建议：

```text
type: earnings_void
amount: 正数金额
sourceType: admin_balance_adjust
description: 作废收益 ¥xx.xx
```

说明：amount 可以保持正数，靠 type 和 description 表示作废动作，避免负数语义混乱。

如果项目已有流水类型规范，请按现有规范处理，但结果文件必须说明。

### 5. OperationLog（操作日志）

必须写操作日志，记录管理员对用户执行了收益作废。

如果现有余额调整接口已有统一日志逻辑，要确认 `earnings_void` 也进入该逻辑。

### 6. 通知

必须通知用户。

建议新增通知模板：

```text
earnings_voided
```

中文标题：

```text
收益作废通知
```

通知内容示例：

```text
【敏维科技】{{userName}} 您好，您的可用收益 ¥{{amount}} 已被后台作废。剩余可用收益：¥{{earningsAvailable}}，累计作废收益：¥{{earningsVoided}}。如有疑问请联系客服。
```

通知失败不能影响主流程。

如果最终复用已有 `balance_change` 模板，也可以，但结果文件必须说明原因。
