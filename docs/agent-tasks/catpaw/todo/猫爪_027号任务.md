## 七、P0-A：严格校验服务端原始缺口类型

### 7.1 当前漏洞

当前实现类似：

```typescript
const apiShortage = Number(verifyErr.data?.shortage)

if (Number.isFinite(apiShortage) && apiShortage >= 0) {
  actualShortage = apiShortage
}
```

这会导致：

```text
Number(null)  = 0
Number('')    = 0
Number(false) = 0
```

三个非法类型全部被当作合法零缺口。

### 7.2 正确实现

必须先保留原始值，不得先 `Number(...)`：

```typescript
const rawApiShortage = verifyErr.data?.shortage

if (
  typeof rawApiShortage === 'number' &&
  Number.isFinite(rawApiShortage) &&
  rawApiShortage >= 0
) {
  actualShortage = rawApiShortage
} else {
  const calculatedShortage = calculatePendingShortage(paymentAmount, user.balance)
  // 显式处理 null
}
```

要求：

1. `null`、`''`、`'0'`、`false`、对象、数组、NaN、Infinity、负数均不得直接使用。
2. 非法接口值必须进入本地兜底计算。
3. 兜底计算返回 `null` 时进入 `validation_error`。
4. 该错误分支不得更新缺口、不得持久化、不得打开收益弹窗、不得自动支付。
5. 原订单编号和已有会话必须保留。

## 八、P0-B：新订单禁止硬编码持久化零缺口

### 8.1 当前漏洞

当前新订单创建后、调用支付验证前存在：

```typescript
shortage: 0
setPendingPayment(currentPendingPayment)
persistPendingPayment(orderId, 0)
```

这会在支付接口缺口异常时留下假的零缺口会话。

### 8.2 正确实现

在创建订单前，先确保首次下单所需资金数字有效；创建订单取得 `currentPayAmount` 后，使用：

```typescript
const initialShortage = calculatePendingShortage(currentPayAmount, user.balance)
```

只有 `initialShortage !== null` 时，才允许构造和持久化新的待支付记录：

```typescript
currentPendingPayment = {
  orderId,
  payAmount: currentPayAmount,
  shortage: initialShortage,
  shipping: ...,
}

setPendingPayment(currentPendingPayment)
persistPendingPayment(orderId, initialShortage)
```

要求：

1. 删除 `persistPendingPayment(orderId, 0)`。
2. 删除新订单记录中的硬编码 `shortage: 0`。
3. 若初始缺口计算返回 `null`，不得继续支付验证；进入 `validation_error` 并提示“资金数据异常，请重新验证”。
4. 不能因为修复零缺口而重新引入重复创建订单风险。
5. 新订单创建成功后若本地持久化失败，应保留当前内存中的订单编号并明确提示，不得再次自动创建订单。
6. 正常接口缺口返回后，仍使用接口的合法真实缺口覆盖初始缺口。
7. 支付成功后才清除会话。

### 8.3 设计约束

不允许修改 `pending-payment-session.ts` 的结构，不允许新增 nullable shortage。

因此必须在创建订单前后保证 `currentPayAmount` 和 `user.balance` 可被 `calculatePendingShortage` 正确验证。

## 九、P1-A：锁定收货快照空白值校验与标准化

### 9.1 当前漏洞

当前代码只检查：

```typescript
!lockedShipping.recipientName
!lockedShipping.recipientPhone
!lockedShipping.shippingAddress
```

`'   '` 属于 truthy，会绕过校验。

### 9.2 正确实现

建议先计算完整性：

```typescript
const hasCompleteLockedShipping = Boolean(
  lockedShipping?.recipientName.trim() &&
  lockedShipping?.recipientPhone.trim() &&
  lockedShipping?.shippingAddress.trim()
)
```

待支付分支必须：

```typescript
if (!lockedShipping || !hasCompleteLockedShipping) {
  toast.error('订单收货信息缺失，请联系客服')
  return
}

shippingSnapshot = {
  recipientName: lockedShipping.recipientName.trim(),
  recipientPhone: lockedShipping.recipientPhone.trim(),
  shippingAddress: lockedShipping.shippingAddress.trim(),
}
```

同时：

1. JSX 只读摘要必须以完整快照为条件；对象存在但字段空白时显示只读错误。
2. 确认按钮在快照缺失或字段空白时禁用。
3. `handleConfirm` 必须保留第二道 return 防护。
4. 完整快照提交给 `onConfirm` 前必须 trim。
5. 非待支付模式原地址校验与地址簿保存逻辑保持不变。

## 十二、完整业务链路推演

结果报告必须逐步推演：

1. 首次购买前资金数据有效性检查。
2. 无默认地址用户填写完整收货信息。
3. 创建订单并取得真实订单编号和应付金额。
4. 用应付金额和当前余额计算初始缺口。
5. 保存真实初始缺口，不保存硬编码 0。
6. 调用支付验证。
7. 接口返回合法余额不足缺口时更新会话。
8. 接口返回非法类型时进入本地兜底计算。
9. 兜底计算异常时保留订单编号、停止弹窗和自动支付。
10. 关闭再打开结算窗口时锁定首次收货快照。
11. 快照字段为空白时只读报错、禁用确认、不显示编辑表单。
12. 快照完整时只输入支付密码并复用原订单。
13. 收益转入成功但资金刷新失败时保留原订单并要求重新验证。
14. 全程不自动支付、不绕过支付密码、不创建第二个订单。
15. 支付成功后才清除会话并跳转订单详情。
