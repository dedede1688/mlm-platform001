# 猫爪_017号任务：购物车余额不足引导功能完整发布

## 一、任务性质

013-016 号开发和复审已经完成，所有 P0/P1 问题闭合。本任务执行唯一一次完整发布验证、精确提交、推送和部署确认。

本轮已获得胡子老师授权：

- 可以 commit（提交）。
- 可以 push（推送）。
- 可以触发 Vercel（部署平台）自动部署。

## 二、效率规则

为了避免终端重复运行和长时间卡住：

1. 每项验证只运行一次。
2. 不同时启动两个相同的测试或构建进程。
3. 命令进入后台后等待原命令完成，不要重复启动。
4. 不为读取输出反复创建多个临时文件。
5. 单个命令超过 10 分钟仍无完成状态，停止该命令并如实报告，不无限等待。
6. Vercel 状态最多检查 15 次，每次间隔约 20 秒；超过 5 分钟仍未完成就报告“仍在构建”，不要一直挂着。

## 五、唯一一次完整验证

严格按以下顺序运行。

### 5.1 代码规范检查

```powershell
npm run lint
```

要求：退出码 0，0 个 Error（阻断错误）。项目既有 Warning（警告）可记录但不阻塞。

### 5.2 类型检查

```powershell
npm run typecheck
```

要求：退出码 0。

### 5.3 全量测试 + 覆盖率门槛

只运行这一条，不再重复运行 `npm test`：

```powershell
npm run test:coverage:check
```

要求：

- 全部测试文件通过。
- 0 个失败测试。
- 语句覆盖率不低于 70%。
- 记录测试文件数、测试数和覆盖率。

### 5.4 项目构建

```powershell
npm run build
```

要求：退出码 0，编译成功，页面生成完成。

### 5.5 差异检查

```powershell
git diff --check
```

Windows 的 LF/CRLF 换行提示可以记录，不属于空白错误；真实空白错误必须停止。

任一关键验证失败：立即停止，不提交、不推送、不部署，并写结果报告。

## 六、提交前精确暂存

仅暂存 4 个目标文件：

```powershell
git add -- 'src/app/cart/page.tsx' 'src/components/checkout/CheckoutDialog.tsx' 'src/components/EarningsTransferModal.tsx' '__tests__/components/cart-earnings-transfer-flow.test.ts'
```

然后运行：

```powershell

git diff --cached --stat
git status --short
```

`git diff --cached --name-only` 必须恰好显示上述 4 个文件。多一个或少一个都必须停止，先纠正暂存区。

## 七、提交和推送

提交说明：

```text

```

执行：

```powershell
git commit -m 'feat(cart): guide earnings transfer on insufficient balance'
git push origin main
```

若 PowerShell（命令行）引号解析失败，使用单个临时提交说明文件配合 `git commit -F`，成功后立即删除；不要连续尝试多套 Shell（命令行环境）。

## 八、铁律 1：验证远程提交

推送后立即运行：

```powershell
git log origin/main --oneline -1
git rev-parse HEAD
git rev-parse origin/main
```

必须确认：

```text
本地 HEAD = origin/main = 本次新 commit hash
```

不一致则推送未闭环，不得报告部署成功。

## 九、持续集成与部署验证

### 9.1 GitHub Actions（远程自动检查）

确认本次提交对应的 GitHub Actions（远程自动检查）结果。至少记录：

- Lint（代码规范检查）。
- TypeScript type check（类型检查）。
- Tests（测试）。
- Coverage threshold（覆盖率门槛）。

若远程检查仍在运行，可与 Vercel 一起按下方次数限制轮询；若失败，记录真实失败步骤和日志摘要，不要把 Vercel Ready 当作远程检查通过。

### 9.2 Vercel（部署平台）

确认 Vercel 最新生产部署：

- commit hash 等于本次提交。
- 环境为 Production（生产环境）。
- 状态为 Ready（已就绪）。

优先使用已验证可用的 GitHub Commit Status API（提交状态接口）或 Vercel Dashboard（部署控制台）。

不要无限轮询：最多 15 次、每次约 20 秒。浏览器或网络不可用时如实报告“未独立确认”，不要伪造 Ready。

## 十、业务发布后人工验收清单

不要伪造登录态浏览器验证。结果报告中明确交给胡子老师完成：

1. 单商品余额不足：出现收益转余额弹窗。
2. 收益转入成功：不自动支付，提示再次确认。
3. 同一商品再次确认：复用原订单，不重复创建。
4. 双商品 A→B→A：分别保留订单和余额缺口。
5. A 支付成功：B 的待支付记录仍保留。
6. 可用收益为 0：不弹不可操作的收益弹窗，显示“去充值”。
7. 关闭结算弹窗：订单保留，可在“我的订单”继续支付。
