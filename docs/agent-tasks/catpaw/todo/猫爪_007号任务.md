## 三、本轮只做三个收尾点

### 收尾 1：可用收益不足返回 400

当前 `updateMany.count === 0` 时抛出：

```text
可用收益不足
```

但路由外层统一返回 500。

要求：

- 可用收益不足属于业务校验失败，返回 HTTP 400。
- 返回体保持：

```json
{
  "success": false,
  "message": "可用收益不足"
}
```

- 其他未知异常继续返回 500。
- 不要破坏事务原子性。
- 不要把条件更新改成普通 `update`。

允许采用最小改动：在路由外层 `catch` 中根据错误消息映射状态码。

必须更新接口测试：

```text
updateMany.count = 0
→ status = 400
→ success = false
→ message 包含“可用收益不足”
→ 不写流水、不写日志、不发通知
```

### 收尾 2：通知关联本次资金流水

收益作废事务已经创建 `BalanceRecord`（资金流水），该记录有唯一 `id`。

要求：

1. 保存 `tx.balanceRecord.create(...)` 的返回结果。
2. 在事务返回值中带出 `balanceRecordId`。
3. 路由调用 `notifyEarningsVoid` 时传入：

```typescript
balanceRecordId: result.balanceRecordId
```

4. `notifyEarningsVoid` 参数新增：

```typescript
balanceRecordId: string
```

5. 调用 `sendInApp` 时增加：

```typescript
sourceType: 'balance_record'
sourceId: params.balanceRecordId
```

这样用户通知可以追踪到本次唯一的资金流水，不要使用用户编号代替操作编号。

必须补测试：

- 路由传给通知的 `balanceRecordId` 等于资金流水创建结果的 `id`。
- `sendInApp` 收到 `sourceType = balance_record`。
- `sendInApp` 收到 `sourceId = balanceRecordId`。

### 收尾 3：补强原子资金更新断言

当前测试数量足够，但必须直接证明数据库更新参数正确。

在：

```text
__tests__/api/admin/users/balance.test.ts
```

增加或补强断言，精确检查：

```typescript
expect(tx.user.updateMany).toHaveBeenCalledWith({
  where: {
    id: 'u1',
    earningsAvailable: { gte: 40 },
  },
  data: {
    earningsAvailable: { decrement: 40 },
    earningsVoided: { increment: 40 },
  },
})
```

还必须直接确认 `updateMany.data` 不包含：

```text
balance
frozenBalance
consumeBalance
earningsFrozen
earningsPending
```

不要只检查 mock（模拟数据）返回的 `afterUser`，不要只检查 BalanceRecord（资金流水）里的余额快照。
