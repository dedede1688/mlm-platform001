## 六、P0-1：待支付状态无条件锁定地址区域

### 6.1 当前错误

当前代码类似：

```typescript
if (hasPendingOrder && lockedShipping) {
  // 使用锁定快照
} else {
  // 校验可编辑 state
}
```

当前 JSX 类似：

```tsx
hasPendingOrder && lockedShipping ? lockedSummary : editableForm
```

当 `hasPendingOrder=true`、`lockedShipping=null` 时，会错误回退到首次下单模式。

### 6.2 必须先写的红灯测试

修改 `__tests__/components/checkout-pending-order-lock.test.ts`，必须精确证明：

1. `handleConfirm` 第一层分支只依赖 `hasPendingOrder`，不得依赖 `hasPendingOrder && lockedShipping`。
2. 在 `hasPendingOrder` 分支内部，`lockedShipping` 缺失或三个字段任一为空时必须 return。
3. 快照缺失时不得进入 `recipientName`、`recipientPhone`、`addressPca`、`detailAddress` 的首次下单校验。
4. JSX 第一层分支只依赖 `hasPendingOrder`。
5. `hasPendingOrder=true` 且 `lockedShipping=null` 时显示只读错误提示。
6. 上述状态不得渲染：
   - 地址簿选择器；
   - 收件人姓名输入框；
   - 手机号输入框；
   - 省市区选择器；
   - 详细地址输入框。
7. 快照缺失时确认按钮应禁用，且 `handleConfirm` 仍有第二道 return 防护。
8. `lockedShipping` 完整时仍允许用户只输入支付密码重新确认。
9. 非待支付模式仍保留原地址校验和地址簿保存逻辑。

禁止把测试调整为接受：

```tsx
hasPendingOrder && lockedShipping ? ... : editableForm
```

测试必须提取真实 `handleConfirm` 函数块和地址 JSX 分支；找不到起点或终点必须失败。

### 6.3 最小实现要求

`handleConfirm` 应采用以下逻辑结构：

```typescript
let shippingSnapshot: CheckoutLockedShipping

if (hasPendingOrder) {
  if (
    !lockedShipping ||
    !lockedShipping.recipientName.trim() ||
    !lockedShipping.recipientPhone.trim() ||
    !lockedShipping.shippingAddress.trim()
  ) {
    toast.error('订单收货信息缺失，请联系客服')
    return
  }

  shippingSnapshot = {
    recipientName: lockedShipping.recipientName.trim(),
    recipientPhone: lockedShipping.recipientPhone.trim(),
    shippingAddress: lockedShipping.shippingAddress.trim(),
  }
} else {
  // 保留原有首次下单逐项校验
  // 从可编辑 state 构造 shippingSnapshot
}
```

JSX 应采用两层分支：

```tsx
{hasPendingOrder ? (
  lockedShipping ? (
    // 只读收货摘要
  ) : (
    // 只读错误提示，不显示可编辑表单
  )
) : (
  // 原可编辑地址表单
)}
```

确认按钮必须增加类似：

```typescript
disabled={submitting || (hasPendingOrder && !lockedShipping)}
```

原有支付密码校验、`onConfirm`、非待支付地址簿保存逻辑保持不变。

## 七、P0-2：三个缺口调用方显式处理 null

### 7.1 总规则

删除 `src/app/products/[id]/page.tsx` 中三个：

```typescript
calculatePendingShortage(...) ?? 0
```

任何非法数字都不得被转成零缺口、不得显示恢复成功、不得覆盖待支付会话。

### 7.2 调用方一：`restorePendingPayment`

正确行为：

```typescript
const shortage = calculatePendingShortage(orderPayAmount, latestUser.balance)
if (shortage === null) {
  setRestoreStatus('validation_error')
  setRestoreError('资金数据异常，请重新验证')
  return
}
```

该分支必须：

- 不调用 `setPendingPayment`；
- 不调用 `persistPendingPayment`；
- 不调用 `clearProductPendingPayment`；
- 不进入 `restored`；
- 保留原会话并允许重新验证。

### 7.3 调用方二：余额不足接口兜底计算

服务端 `data.shortage` 只有在：

```typescript
Number.isFinite(apiShortage) && apiShortage >= 0
```

时才可直接使用。

否则调用 `calculatePendingShortage(paymentAmount, user.balance)`。若返回 `null`：

- 设置 `validation_error`；
- 显示“资金数据异常，请重新验证”；
- 不更新 `pendingPayment.shortage`；
- 不持久化 shortage=0；
- 不打开收益转余额弹窗；
- 不自动支付；
- 返回 `null`，保留原订单编号。

### 7.4 调用方三：收益转入成功后的重新计算

收益转入已经成功，随后读取最新用户资金并重算缺口。

要求：

1. `fetchUser` 返回 null：
   - 不用 `pendingPayment.shortage - transferAmount` 猜测最终缺口；
   - 保留原待支付记录；
   - 关闭收益弹窗；
   - 进入 `validation_error`；
   - 提示“收益已转入，请重新验证资金信息”。
2. `calculatePendingShortage` 返回 null：同样进入验证失败，不更新和持久化缺口。
3. 合法结果才允许更新 `pendingPayment` 和会话缺口。
4. 任何分支都不得自动调用支付接口。

### 7.5 必须先写的红灯测试

修改 `__tests__/components/product-pending-payment-flow.test.ts`，必须提取三个真实函数块并证明：

1. `restorePendingPayment` 不含 `calculatePendingShortage(...) ?? 0`。
2. 恢复计算为 null 时进入 `validation_error`，不持久化、不 restored。
3. `handleCheckoutConfirm` 的余额不足分支不含 `?? 0`。
4. API 缺口为负数时不能直接 `Math.max(0, apiShortage)` 变成 0。
5. 兜底计算为 null 时不更新、不持久化、不打开收益弹窗。
6. `handleEarningsTransferSuccess` 不含 `?? 0`。
7. 最新资金读取失败或计算为 null 时不猜测缺口、不持久化。
8. 收益转入成功后仍不调用 `verify-payment` 或 `handleCheckoutConfirm`。
9. 三个错误分支均保留原订单编号和原会话。

禁止只检查全文件不存在 `?? 0`；必须分别检查三个真实函数块和对应错误分支。
