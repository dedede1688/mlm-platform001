## 五、已确认的问题

### 5.1 P0：订单创建成功后丢失订单编号

当前流程：

```text
POST /api/orders 创建订单成功
→ 取得 orderId
→ calculatePendingShortage(...) 返回 null
→ 直接 return null
→ pendingPayment 和 sessionStorage 都没有保存 orderId
→ 用户重试
→ 再次 POST /api/orders
→ 产生重复 pending 订单
```

根因不是订单接口，而是资金预校验发生在创建订单之后。

### 5.2 P1：空白锁定收货快照仍显示成功摘要

当前 `CheckoutDialog.tsx` 的 JSX 仍使用：

```tsx
lockedShipping ? (
  // 黄色“订单已创建”摘要
) : (
  // 红色错误提示
)
```

当对象存在，但姓名、手机号或地址是纯空格时，对象仍为 truthy（真值），页面会显示错误的黄色成功摘要。

### 5.3 P2：测试报告与真实代码不一致

当前测试仍存在：

- 固定 300/400 字符窗口；
- `A || B` 宽松断言；
- 未验证资金预校验发生在创建订单接口之前；
- 未通过真实 JSX 渲染验证空白快照显示红色错误。

## 六、确认采用的修复方案

胡子老师已选择 **方案 A：创建订单前校验资金数据**。

强制业务规则：

1. 在调用 `fetch('/api/orders')` 前，必须校验首次下单使用的应付金额和用户余额。
2. 预校验返回 `null` 时，必须进入 `validation_error`，显示明确错误并立即返回。
3. 预校验失败时，`POST /api/orders` 调用次数必须为 0。
4. 不允许使用 `shortage: 0` 或 `persistPendingPayment(orderId, 0)` 伪造缺口。
5. 不允许把 `shortage` 改成 nullable（可空类型）。
6. 不允许修改会话版本或会话数据结构。
7. 订单创建成功后，不得再存在任何会让订单编号丢失的 `return` 分支。
8. 正常支付、余额不足、收益转余额和支付成功清理会话的原有行为必须保持。

7. 订单创建成功后，不存在因初始缺口为 `null` 而直接丢弃 `orderId` 的分支。

测试必须使用完整函数体提取和索引顺序比较，禁止固定字符窗口。

建议增加一个括号平衡提取函数，例如：

```typescript
function extractBracedBlock(source: string, marker: string): string {
  const markerIndex = source.indexOf(marker)
  expect(markerIndex).toBeGreaterThanOrEqual(0)

  const openIndex = source.indexOf('{', markerIndex)
  expect(openIndex).toBeGreaterThan(markerIndex)

  let depth = 0
  for (let index = openIndex; index < source.length; index += 1) {
    if (source[index] === '{') depth += 1
    if (source[index] === '}') depth -= 1
    if (depth === 0) return source.slice(markerIndex, index + 1)
  }

  throw new Error(`未找到完整代码块：${marker}`)
}
```

顺序断言必须类似：

```typescript
const confirmBlock = extractBracedBlock(source, 'const handleCheckoutConfirm')
const precheckIndex = confirmBlock.indexOf('calculatePendingShortage(finalPrice, user.balance)')
const createOrderIndex = confirmBlock.indexOf("fetch('/api/orders'")

expect(precheckIndex).toBeGreaterThanOrEqual(0)
expect(createOrderIndex).toBeGreaterThan(precheckIndex)
```

不得使用：

```typescript
expect(A || B).toBe(true)
source.substring(index, index + 300)
source.substring(index, index + 400)
if (match) { ... }
```

### 7.2 第一组最小实现

```text
src/app/products/[id]/page.tsx
```

在 `if (!orderId)` 新订单分支中，必须在 `fetch('/api/orders')` 前完成预校验：

```typescript
const preCreateShortage = calculatePendingShortage(finalPrice, user.balance)

if (preCreateShortage === null) {
  setRestoreStatus('validation_error')
  setRestoreError('资金数据异常，无法创建订单')
  toast.error('资金数据异常，无法创建订单')
  return null
}
```

要求：

1. 上述校验必须真实位于创建订单接口之前。
2. 预校验只负责阻止脏数据创建订单，不得写 sessionStorage。
3. 创建订单接口返回的 `payAmount` 必须先检查原始类型，禁止 `Number(null)` 变成 0：

```typescript
const rawCreatedPayAmount = orderData.data?.payAmount
currentPayAmount = (
  typeof rawCreatedPayAmount === 'number' &&
  Number.isFinite(rawCreatedPayAmount) &&
  rawCreatedPayAmount >= 0
)
  ? rawCreatedPayAmount
  : finalPrice
```

4. 创建成功后使用接口的合法真实金额重新计算缺口。
5. 重新计算理论上不会返回 `null`，因为 `finalPrice` 和 `user.balance` 已通过预校验，接口金额也经过严格归一化。
6. 防御分支不得丢弃订单编号。允许在重新计算异常时回退到已经验证有效的 `preCreateShortage`，但禁止回退到 0：

```typescript
const recalculatedShortage = calculatePendingShortage(currentPayAmount, user.balance)
const initialShortage = recalculatedShortage === null
  ? preCreateShortage
  : recalculatedShortage
```

7. 随后正常构造并保存：

```typescript
currentPendingPayment = {
  orderId,
  payAmount: currentPayAmount,
  shortage: initialShortage,
  shipping: {
    recipientName: input.recipientName,
    recipientPhone: input.recipientPhone,
    shippingAddress: input.shippingAddress,
  },
}

setPendingPayment(currentPendingPayment)
persistPendingPayment(orderId, initialShortage)
```

8. 不能新增第二次创建订单调用。
9. 不能自动重新支付。
