## 四、业务逻辑专项核验

### 4.1 统一值计算

在 `src/app/admin/users/page.tsx` 中确认存在等价逻辑：

```typescript
const actualPhoneSuffix = detailUser?.phone ? detailUser.phone.slice(-4) : ''
const normalizedSuffix = payPwdResetSuffix.trim()
const suffixMatches = /^\d{4}$/.test(normalizedSuffix) && normalizedSuffix === actualPhoneSuffix
```

检查这些值是否基于当前 `detailUser` 计算，不得使用旧用户缓存、固定手机号或写死后四位。

### 4.2 第一层拦截：按钮状态

确认“重置支付密码”按钮只有同时满足以下条件才能启用：

1. 当前不在处理状态。
2. 重置原因去空格后不少于 5 个字符。
3. 输入值是 4 位数字。
4. 输入值等于当前会员真实手机号后四位。

错误后四位必须保持按钮禁用。

### 4.3 第二层拦截：打开确认弹窗前

检查 `handleResetPaymentPassword`：

1. 保留原有权限、原因长度和 4 位数字格式校验。
2. 在 `setShowPayPwdConfirm(true)` 前比较真实后四位。
3. 不匹配时显示：

```text
手机号后 4 位不匹配，请核对后重试
```

4. 不匹配分支必须立即 `return`。
5. 不得打开二次确认弹窗。

### 4.4 第三层拦截：真正调用接口前

检查 `doResetPaymentPassword`：

1. 在 `setSavingPayPwdReset(true)` 和 `fetch(...)` 之前再次比较真实后四位。
2. 不匹配时显示同一条错误提示。
3. 不匹配时关闭或保持关闭确认弹窗。
4. 不匹配时立即 `return`。
5. 不得进入“处理中”状态。
6. 不得调用重置接口。

必须检查语句顺序，不能只检查字符串是否在同一个文件里出现。

### 4.5 即时界面反馈

确认输入满 4 位但不匹配时：

- 输入框出现错误边框或等价错误状态。
- 输入框下方显示红色“不匹配”提示。
- 改为正确后四位后提示消失。
- 未输入满 4 位时不错误提示“不匹配”。

### 4.6 后端错误透传

确认接口失败分支优先读取：

```typescript
data.error || data.message || '重置失败'
```

后端现有 `errorResponse(...)` 使用 `error` 字段，因此不能继续只读 `data.message`。

### 4.7 后端安全边界保持

只读检查后端路由，确认仍然根据数据库中的目标用户手机号执行：

```typescript
const actualSuffix = targetUser.phone.slice(-4)
if (phoneSuffix !== actualSuffix) {
  return errorResponse('手机号后 4 位不匹配', 400)
}
```

并确认不匹配时不会：

- 清空支付密码哈希。
- 写操作日志。
- 发通知。

前端校验只能改善体验，后端必须继续作为最终安全边界。

## 五、测试可信度专项核验

### 5.1 禁止静默跳过

检查新增测试不得使用以下假通过模式：

```typescript
if (match) {
  expect(...)
}
```

关键提取失败时必须先：

```typescript
expect(match).not.toBeNull()
```

再执行后续断言。

### 5.2 检查断言是否被放宽

猫爪在红灯阶段后曾调整测试表达式。必须特别核验这些调整是否仍然真实约束业务，而不是为了绿灯放宽：

1. 允许变量名 `actualPhoneSuffix`、`normalizedSuffix`、`suffixMatches`，但必须确认这些变量在真实业务文件中具有正确含义。
2. 如果测试使用 `A || B`（或）断言，必须逐项确认 A、B 任一成立时都足以证明同一业务要求。
3. 如果某个“或”分支只检查变量名称存在、却不验证变量定义或使用位置，应列为 P1 测试缺口。
4. 正确后四位流程不能只检查 `setShowPayPwdConfirm(true)` 字符串存在；至少要结合前置拦截顺序，确认正确值不会被无条件阻断。
5. 红色提示测试必须限定在目标输入框附近或目标 JSX 范围，不得用全文件第三次出现等脆弱方式误匹配其他文案；如果当前实现仍依赖这种脆弱定位，按影响列为 P1 或 P2，并说明是否阻塞提交。

### 5.3 接口测试增强

确认“手机号后 4 位不匹配”接口测试真实断言：

```typescript
expect(data.error).toBe('手机号后 4 位不匹配')
expect(prisma.user.updateMany).not.toHaveBeenCalled()
expect(logOperation).not.toHaveBeenCalled()
expect(OrderNotificationService.notifyPaymentPasswordReset).not.toHaveBeenCalled()
```

不得只断言状态码或错误文本。

### 5.4 红灯证据评价

猫爪报告修复前有 9 个新增测试失败。请结合测试差异判断：

- 失败是否确实源于缺少真实后四位比较、缺少二次拦截、缺少错误字段读取。
- 是否存在先写过度绑定实现细节的测试，随后通过放宽断言才变绿。
- 即使红灯过程不完美，也要区分“业务实现正确”和“测试设计仍需加强”。

- 业务实现正确、安全边界完整。
