## 三、业务目标

后台管理员在：

```text
会员管理 → 会员详情 → 资金账户 → 资金调整
```

选择「作废收益」，输入正数金额后，系统必须执行：

```text
earningsAvailable -= amount
earningsVoided += amount
```

并满足：

- 可用收益不能扣成负数。
- 不改变余额、冻结余额、冻结收益、待结算收益。
- 写资金流水。
- 写操作日志。
- 通知用户。
- 通知失败不能让已经成功的资金操作表现为失败。

## 四、强制检查项

### 2. 输入校验

确认 `earnings_void`（收益作废）满足：

- `amount` 必须是数字。
- 必须使用 `Number.isFinite` 或等价方式拒绝 `NaN`、`Infinity`、`-Infinity`。
- `amount` 必须大于 0。
- 0、负数、字符串、空值均不能进入资金事务。
- 原因至少 5 个字的规则仍然有效。

如果当前代码只用 `isNaN` 而没有拒绝 `Infinity`，列为 P1。

### 3. 资金原子性和并发保护

确认后端使用数据库条件更新：

```typescript
updateMany({
  where: {
    id,
    earningsAvailable: { gte: amount },
  },
  data: {
    earningsAvailable: { decrement: amount },
    earningsVoided: { increment: amount },
  },
})
```

并确认：

- `count === 0` 时返回可用收益不足。
- `earningsAvailable` 和 `earningsVoided` 在同一事务中更新。
- 不修改 `balance`。
- 不修改 `frozenBalance`。
- 不修改 `consumeBalance`。
- 不修改 `earningsFrozen`。
- 不修改 `earningsPending`。
- 不使用 `$queryRaw` 或 `$queryRawUnsafe`。

### 4. 并发下流水和日志快照准确性

重点检查以下风险：

当前实现先读取 `before`，再执行条件 `updateMany`。两个请求并发时，第二个请求读取到的 `before` 可能已经过期。

请判断以下内容是否使用了数据库更新后的真实值，而不是简单用旧值计算：

- `BalanceRecord`（资金流水）的说明文字。
- `format4FieldDelta`（四字段变化说明）。
- `OperationLog.oldValue`（操作前值）。
- `OperationLog.newValue`（操作后值）。
- 通知中的剩余可用收益和累计作废收益。

如果数据库资金结果安全，但并发时流水或日志可能记录错误金额，列为 P1，并给出明确证据。

### 5. BalanceRecord（资金流水）

确认：

- `type = earnings_void`。
- `amount` 使用正数。
- `sourceType` 和 `sourceId` 能追踪管理员操作。
- 描述明确写出作废金额和原因。
- 资金流水与用户字段更新处于同一事务。

### 6. OperationLog（操作日志）

确认：

- 操作日志包含管理员、用户、作废前值和作废后值。
- `oldValue` 同时记录 `earningsAvailable` 和 `earningsVoided`。
- `newValue` 同时记录 `earningsAvailable` 和 `earningsVoided`。
- 日志失败不会造成资金事务回滚或接口误报失败。

### 7. 通知职责和失败隔离

这是本次最重要的检查项。

确认通知在资金事务提交后由路由层调用，并检查 `notifyEarningsVoid`（收益作废通知）整个方法是否都被异常保护覆盖。

特别检查：

```typescript
const user = await prisma.user.findUnique(...)
```

如果这次用户查询位于 `try/catch`（异常保护）之外，那么可能出现：

```text
资金事务已经成功提交
→ 通知方法查询用户失败
→ 接口返回失败
→ 管理员误以为没成功并重试
→ 同一笔收益被重复作废
```

如果确认存在该链路，必须列为 P0，并给出「不通过」结论；不能以“通知内部其他代码有 try/catch”判定安全。

同时确认：

- 通知调用被 `await`（等待完成）。
- 通知失败被记录，但不影响主流程成功响应。
- 不使用未等待的异步调用。
- 模板类型为 `earnings_voided`。

### 8. 通知模板和生产数据库

确认 `prisma/seed.ts` 已有：

```text
earnings_voided
```

但必须明确区分：

```text
写入 seed.ts ≠ 生产数据库已经存在模板
```

如果尚未对生产数据库执行单条模板 upsert（存在则更新，不存在则新增），应列为部署阻塞项。禁止建议直接运行全量 `npx prisma db seed`，因为全量 seed 可能写入其他测试或初始化数据。

### 9. 前端交互

确认：

- 下拉选项显示「作废收益」。
- 提交类型为 `earnings_void`。
- 输入正数，不使用负数表达扣减。
- 0 或负数前端拦截。
- 超过当前可用收益前端提示。
- 大额二次确认弹窗显示作废金额、当前可用收益、作废后可用收益、作废后累计作废收益。
- 原有「增加可用收益」仍提交 `earnings_add`，不能被本轮改坏。

### 10. 自动化测试

资金逻辑属于高风险范围，不能只依赖类型检查和构建。

检查是否有自动化测试覆盖至少以下场景：

1. 正常作废：可用收益减少、作废收益增加。
2. 余额、冻结余额、消费余额、冻结收益、待结算收益均不变。
3. amount 为 0 被拒绝。
4. amount 为负数被拒绝。
5. amount 为字符串被拒绝。
6. amount 为 `NaN` 被拒绝。
7. amount 为 `Infinity` 被拒绝。
8. 可用收益不足被拒绝。
9. 并发条件更新 `count = 0` 被拒绝。
10. 写入 BalanceRecord（资金流水）。
11. 写入 OperationLog（操作日志）。
12. 通知成功。
13. 通知失败不影响资金调整成功响应。

如果没有新增任何自动化测试，至少列为 P1，并且不建议直接提交推送；资金操作不能只凭源码检查、类型检查和构建判定安全。

## 七、结论口径

出现以下任一情况，必须判定「不通过」：

- 资金更新不是原子条件更新。
- 可用收益可能被扣成负数。
- 只减少可用收益而没有增加作废收益。
- 修改了余额、冻结余额、冻结收益或待结算收益。
- 资金已提交后，通知异常可能让接口返回失败并诱发管理员重复操作。
- 修改了数据库结构或使用原始 SQL。

出现以下情况，至少判定「有条件通过」，并且暂不建议提交推送：

- 没有资金逻辑自动化测试。
- 并发时流水或日志快照可能不准确。
- `Infinity` 等非有限数字没有被拒绝。
- 生产数据库尚未同步 `earnings_voided` 模板。

只有资金、流水、日志、通知失败隔离、测试和模板同步方案全部明确安全，才可以建议提交推送。
