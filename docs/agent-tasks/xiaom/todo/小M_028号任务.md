## 六、P0：创建订单前资金预校验

完整读取：

```text
src/app/products/[id]/page.tsx
__tests__/components/product-pending-payment-flow.test.ts
```

必须逐项确认：

1. `handleCheckoutConfirm` 的新订单分支先执行：

```typescript
calculatePendingShortage(finalPrice, user.balance)
```

2. 上述预校验真实位于 `fetch('/api/orders')` 之前。
3. 预校验返回 `null` 时：
   - 设置 `validation_error`；
   - 显示资金数据异常提示；
   - 立即返回；
   - 不写待支付会话；
   - 不调用创建订单接口。
4. 不存在 `shortage: 0` 或 `persistPendingPayment(orderId, 0)`。
5. 创建订单接口返回的 `payAmount` 先检查原始类型，不能让 `null`、空字符串或 `false` 被转换成 0。
6. 接口金额合法时使用接口真实金额重新计算缺口。
7. 重新计算异常时，只允许回退到创建前已验证有效的真实缺口，不允许回退到 0。
8. 取得有效 `orderId` 后，在 `setPendingPayment` 之前不得存在会丢失有效订单编号的返回分支。
9. `orderId` 为空时返回属于创建失败，不得误判为“丢失有效订单编号”。
10. 已有 `pendingPayment.orderId` 时必须跳过创建订单接口，复用原订单。
11. 不得自动重新支付。

必须完整推演：

```text
资金数据异常
→ 创建订单前预校验失败
→ POST /api/orders 调用次数为 0
→ 数据库不产生订单

资金数据正常
→ POST /api/orders 调用一次
→ 取得有效 orderId
→ 保存 pendingPayment 和 sessionStorage
→ verify-payment
→ 余额不足时更新真实缺口
→ 用户关闭并重开后复用原 orderId
```

### 6.1 测试可信度特别检查

猫爪报告称已经证明“预校验失败时创建订单接口调用次数为 0”。

必须如实判断测试实际属于哪一种：

- 真实运行 `handleCheckoutConfirm` 并 mock `fetch`，断言创建订单调用次数为 0；
- 还是仅通过读取源码和比较字符串索引证明代码顺序。

源码顺序契约测试可以作为证据，但不得在报告中写成“真实接口调用次数测试”。

## 八、最高优先级：猫爪报告准确性与测试合规性

猫爪执行记录明确显示：

1. 尝试真实 JSX 渲染测试失败；
2. 随后删除该测试，改用源码字符串断言；
3. 任务文件把“真实 JSX 渲染测试无法在现有依赖下完成”列为停止条件。

必须独立核验并如实回答：

### 8.1 是否存在真实 JSX 渲染测试

搜索两个测试文件是否真实使用：

```text
renderToStaticMarkup
React.createElement
真实导入 CheckoutDialog
```

如果不存在，则猫爪结果中“空白收货 JSX 测试”只能称为“源码字符串结构测试”，不能称为“真实 JSX 渲染测试”。

### 8.2 Vitest 配置事实

猫爪执行记录称项目没有 `vitest.config`，但仓库根目录实际存在：

```text
vitest.config.ts
```

读取该文件并判断：

- 真实失败原因是否确实是缺少配置；
- 还是当前配置、测试文件扩展名、组件依赖或 mock 不完整导致；
- 在不改生产配置、不新增依赖的前提下，真实渲染是否确实不可行。

无法证明时必须写“未完成”，不能推测为已完成。

### 8.3 剩余弱断言

逐行搜索：

```powershell
Select-String -Path `
  '__tests__/components/product-pending-payment-flow.test.ts', `
  '__tests__/components/checkout-pending-order-lock.test.ts' `
  -Pattern 'substring\(','\|\|','if\s*\(.*Match','if\s*\(.*Idx\s*===\s*-1','renderToStaticMarkup','React\.createElement'
```

重点检查已发现的残留：

1. `checkout-pending-order-lock.test.ts` 中：

```typescript
const completeCheck = hasPendingBranch.includes('hasCompleteLockedShipping') || ...
```

这是明确的宽松 `A || B` 断言。

2. 多处：

```typescript
source.substring(varIdx, ... + 200)
```

仍属于固定字符窗口。

3. `product-pending-payment-flow.test.ts` 中仍有多处 `substring(...)`；区分：
   - 仅用于从已确认索引切到文件尾，且后续有精确完整块提取；
   - 固定 `+30`、`+200` 等字符窗口；
   - 会导致错误实现也能通过的宽松用法。

猫爪结果报告只承认清理 1 处固定窗口和 1 处宽松 OR，但不能因此忽略其他残留。

### 8.4 判定规则

- 业务代码正确，但真实 JSX 测试缺失、报告表述不准确：至少列为 P1 测试与交付可信度问题。
- 若弱断言能够让错误业务代码通过：列为 P1，提交前必须修复。
- 若仅是不会放宽结论的字符串切片：可以列为 P2，但必须说明理由。
- 不得把源码字符串测试称为真实运行时测试。
