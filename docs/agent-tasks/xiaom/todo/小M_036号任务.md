## 三、业务规则独立核验

### 3.1 共享校验模块

逐项检查：

1. `质量问题`无图片失败。
2. `商品损坏`无图片失败。
3. `未按约定时间发货`无图片无说明成功。
4. `其他`无说明或只有空白说明失败。
5. 图片不是数组、含非字符串、含空字符串、超过 5 张均失败。
6. 成功结果对 reason、description、images 做预期规范化。
7. 未知或恶意输入不会抛出未捕获异常。
8. 导出的集合是否可能被外部意外修改；如存在实际风险，按 P1/P2 说明，不得夸大。

### 3.2 后端退款申请接口

检查 `POST /api/orders/[id]/refund`：

1. 鉴权、订单归属、状态、防重复申请逻辑保持原样。
2. 共享校验发生在 `refundRequest.create` 之前。
3. 校验失败返回 400。
4. 校验失败不创建记录、不写日志、不发通知。
5. 校验成功只使用规范化后的数据写库。
6. pending / approved 继续阻止重复申请。
7. rejected 继续允许再次申请。
8. 退款金额、状态、成功通知和响应没有被改变。

### 3.3 用户退款申请页面

检查：

1. 前端使用同一共享校验模块。
2. `handleSubmit` 在 fetch 前校验，失败时 return。
3. 两种证据型原因显示图片必填标记和明确提示。
4. `其他`显示补充说明必填标记。
5. 切换退款原因不会清空当前申请已上传图片。
6. 页面初始表单为空，不从历史退款申请回填文字或图片。
7. 图片上限、上传、删除和预览路径没有被破坏。

### P1 候选 A：提交按钮禁用状态可能与校验规则不一致

> 提交按钮的禁用状态和提交函数内二次校验必须一致。

当前报告自述按钮仍可能只有：

```tsx

```

请独立判断：

1. 无原因、证据型原因无图片、`其他`无说明时按钮是否仍可点击。
2. 这是否违反明确任务要求和当前页面交互预期。
3. 即使 `handleSubmit` 能拦截，也不能用它替代按钮禁用要求。
