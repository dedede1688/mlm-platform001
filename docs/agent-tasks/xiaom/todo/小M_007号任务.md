### 2. 收益作废资金规则

确认真实代码仍然执行：

```text
earningsAvailable -= amount
earningsVoided += amount
```

确认使用：

```typescript
updateMany({
  where: {
    id,
    earningsAvailable: { gte: amount },
  },
  data: {
    earningsAvailable: { decrement: amount },
    earningsVoided: { increment: amount },
  },
})
```

确认不修改：

- `balance`
- `frozenBalance`
- `consumeBalance`
- `earningsFrozen`
- `earningsPending`

### 3. 可用收益不足状态码

确认：

```text
updateMany.count = 0
→ HTTP 400
→ success = false
→ message = 可用收益不足
```

同时确认：

- 不写 BalanceRecord（资金流水）。
- 不写 OperationLog（操作日志）。
- 不发送通知。
- 其他未知异常继续返回 500。

必须检查是否有自动化测试明确证明“其他未知异常返回 500”。如果代码正确但没有该测试，列为 P1 测试缺口，不要谎报已经测试。

### 4. 通知关联资金流水

确认事务内保存：

```typescript
const balanceRecord = await tx.balanceRecord.create(...)
```

确认事务返回：

```typescript
balanceRecordId: balanceRecord.id
```

确认路由把该编号传给：

```text
notifyEarningsVoid
```

确认通知最终调用 `sendInApp` 时包含：

```typescript
sourceType: 'balance_record'
sourceId: params.balanceRecordId
```

检查 `result.balanceRecordId!`（非空断言）的运行条件：

- 只在 `adjustType === 'earnings_void'` 分支执行。
- 收益作废事务成功时一定创建了资金流水并返回编号。
- 不会把 `undefined` 传给通知。

### 5. 通知失败隔离回归

确认 006 号 P0 修复没有被破坏：

- `notifyEarningsVoid` 整体异常保护仍存在。
- 路由层第二道 `.catch()` 仍存在。
- 通知失败时资金接口仍返回 200 和 `success=true`。
- 不存在未等待异步调用。

### 6. 并发快照和审计记录

确认：

- 条件更新后重新查询数据库真实后值。
- 本次操作前值由真实后值反推。
- BalanceRecord 使用真实快照。
- OperationLog.oldValue 使用真实前值。
- OperationLog.newValue 使用真实后值。
- 通知金额使用真实后值。

### 7. 金额校验

确认继续使用：

```typescript
Number.isFinite(amount)
```

并拒绝字符串、空值、0、负数、`NaN`、`Infinity`、`-Infinity`。

说明测试中 `JSON.stringify(NaN/Infinity)` 会变成 `null`，因此必须同时检查源码，不能只看测试名称。

### 8. 原子更新测试真实性

检查接口测试是否直接断言：

```typescript

  where: {
    id: 'u1',
    earningsAvailable: { gte: 40 },
  },
  data: {
    earningsAvailable: { decrement: 40 },
    earningsVoided: { increment: 40 },
  },
})
```

检查是否直接断言 `updateMany.data` 不含其他资金字段。

### 9. 通知来源测试真实性

确认至少两层测试：

1. 路由测试证明 `BalanceRecord.id` 传入 `notifyEarningsVoid`。
2. 通知服务测试证明 `sendInApp` 收到：

```text

sourceId = BalanceRecord.id
```

不能只检查函数被调用。

## 七、结论口径

出现以下任一情况必须判定不通过：

- 可用收益不足仍返回 500。
- 未知异常错误地返回 400。
- 通知没有关联真实资金流水编号。
- 通知失败仍可能让资金成功接口返回失败。
- 原子资金更新参数不正确。
- 修改了其他资金字段。
- 并发快照不准确。
- 修改数据库结构或使用原始 SQL。

如果核心代码正确、所有现有测试通过，但缺少“未知异常返回 500”的明确自动化测试，可以判定有条件通过，并要求补一条测试后再进入提交阶段。

只有业务逻辑、异常隔离、来源追踪、关键断言和验证命令全部成立，才建议执行生产模板单条同步和提交部署。
