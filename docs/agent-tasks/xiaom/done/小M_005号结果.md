## 2. P0 问题（必须修，否则不通过）

### 🔴 P0-1：notifyEarningsVoid.findUnique 在 try/catch 之外 → 双重作废

**位置**：`src/lib/services/order-notification.service.ts` line 766-769

```typescript
static async notifyEarningsVoid(params: {...}) {
  // ⬇️ 这一行 PRISMA.USER.FINDUNIQUE 在 TRY/CATCH 之外
  const user = await prisma.user.findUnique({
    where: { id: params.userId },
    select: { nickname: true, phone: true },
  })
  if (!user) return
  ...
  await (async () => {
    try {  // ⬇️ try/catch 只能包住这个 IIFE，不能覆盖外层 findUnique
      ...
    } catch (err) { ... }
  })()
}
```

**完整失败链路**（prisma 6 + Vercel + Supabase 真实环境）：

1. 资金事务 `prisma.$transaction(async (tx) => {...})` 已 commit，DB 中 `earningsAvailable: 100 → 60`
2. `route.ts` line 216 `await OrderNotificationService.notifyEarningsVoid({...})` 被调用
3. 进入 `notifyEarningsVoid`，第 1 步 `prisma.user.findUnique` 因 Supabase 连接抖动 / 超时 / Vercel Serverless Function 冷启动后连接未复用等原因失败
4. 异常向上冒泡，**没有顶层 try/catch 保护**（notifyEarningsVoid 整个方法都没有顶层 try/catch，只有内部 IIFE 才有 try/catch）
5. 异常继续冒泡到 `route.ts` 第 216 行的 `await` 调用点
6. `route.ts` line 266 外层 `catch (error)` 捕获，返回 `500 { success: false, message: '数据库错误信息' }`
7. **管理员前端收到 500 错误 → 误以为本次作废失败 → 点击重试**
8. 第二次请求：用户 DB 当前 `earningsAvailable: 60`，`updateMany({ where: { earningsAvailable: { gte: 40 } } })` 中 `60 >= 40` **通过**
9. 第二次 updateMany 成功，DB 中 `earningsAvailable: 60 → 20`，接口返回 200
10. **结果**：用户被扣了 ¥80，但管理员只看到 1 次"成功"响应，资金流水出现 2 条 type=earnings_void 记录

**为什么 updateMany + gte 不能阻止**：updateMany + gte 是"防扣成负数"的并发保护，**不是幂等保护**。第二次请求时用户还有 60 收益，60 >= 40 通过，updateMany 正常执行。

**判定依据**：
- 任务文件第三节第 9 项：明确要求"如果这次用户查询位于 try/catch 之外，那么可能出现资金事务已提交 → 通知查询失败 → 接口返回失败 → 管理员重试 → 同一笔收益被重复作废"——必须列为 P0
- 任务文件第七节第 5 条结论口径："资金已提交后，通知异常可能让接口返回失败并诱发管理员重复操作"——必须判定不通过
- 任务文件**第 179 行明确要求**："不能以'通知内部其他代码有 try/catch'判定安全"——猫爪可能以"IIFE 内部有 try/catch"为由认为安全，但本复审**不被这一理由说服**

**修复动作**（P0 阻塞）：
1. 在 `notifyEarningsVoid` **整个方法外层包 try/catch**，确保 prisma.user.findUnique 失败被吞掉
2. 或：在 route.ts line 216 调用处把 `await OrderNotificationService.notifyEarningsVoid(...)` 包在 try/catch 里，失败仅 log 不抛出
3. **推荐方案 2**（route.ts 现有外层 try/catch 已能 catch，但当前 line 216 在 try 块内 → catch 会让接口 500），需要把 notifyEarningsVoid 调用**移出 route.ts 的 try 块**或者**在 catch 之前先 await notifyEarningsVoid 然后再让 catch 不影响主响应**
4. 完整方案：在 route.ts 的 try/catch 内部，事务 commit 后先把响应数据算好，再单独 try/catch 包裹 notifyEarningsVoid 调用

## 3. P1 问题（不阻塞提交但强烈建议修复）

### 🟡 P1-1：Infinity / -Infinity 校验缺失

**位置**：`src/app/api/admin/users/[id]/balance/route.ts` line 72

```typescript
if (typeof amount !== 'number' || isNaN(amount) || amount === 0) {
  return NextResponse.json(
    { success: false, message: 'amount 必须为非零数字' },
    { status: 400 }
  )
}
```

**问题**：
- `typeof Infinity === 'number'` → 通过
- `isNaN(Infinity) === false` → 通过
- `Infinity === 0 === false` → 通过
- `Infinity <= 0` 在 earnings_void 分支 (line 80) → false（因为 Infinity > 0）→ 通过
- **结果**：`{ amount: 1e308 }` 或 `{ amount: Infinity }`（如果前端被构造 JSON）会通过所有校验，进入 updateMany

**风险**：updateMany `{ decrement: Infinity }` 的 Prisma 行为：
- PostgreSQL 会尝试把 JSONB 解析的 Infinity 转成 float8，**但 PostgreSQL 不支持 Infinity 作为 numeric**（会报 `value out of range` 或截断）
- 最坏情况：用户构造 `{ amount: 1e308 }` 提交，updateMany 实际扣 1e308，让 earningsAvailable 变超大负数（但 gte 仍然能阻止变成负无穷大，因为 gte 1e308 100 < 1e308 时不会通过）

**修复动作**：把 `isNaN(amount)` 替换为 `!Number.isFinite(amount)`，同时拒绝 NaN、Infinity、-Infinity。

### 🟡 P1-2：并发场景下 oldValue 可能过期

**位置**：`src/app/api/admin/users/[id]/balance/route.ts` line 135

```typescript
oldValue: { earningsAvailable: before.earningsAvailable, earningsVoided: before.earningsVoided }
```

**问题**：
- `before` 是 `tx.user.findUnique({ where: { id } })`（line 95），事务**起始**时的快照
- `result.updated` 是 updateMany 之后的 `tx.user.findUnique`（line 131），是**当前事务**的最新值
- 在并发场景下，**两次 findUnique 之间其他事务可能 commit 修改**
- Prisma `$transaction` 默认 READ COMMITTED 隔离级别：第二个事务读到的 `before` 可能是第一个事务 commit 之前的值
- **结果**：oldValue 用 `before`（过期），newValue 用 `result.updated`（真实），两者 delta 不等于 `amount`

**举例**：
- A 事务：read before(100) → updateMany(100→60) → read updated(60) → commit
- B 事务（在 A 提交前开始）：read before(100) → updateMany(60→20，因为 A 已 commit 行锁) → read updated(20)
- B 的 oldValue = `{ earningsAvailable: 100 }`（A 提交前快照）
- B 的 newValue = `{ earningsAvailable: 20 }`（A 提交后真实值）
- B 的 delta = -80，但实际 B 只扣了 40，**不一致**

**真实场景概率**：低（需要 2 个 admin 同时操作同一用户的并发），但代码层面是真实风险。

**修复动作**：把 `oldValue` 改为从 `updated` 之前再 findUnique 一次拿真实"操作前"值，或在 updateMany 之前用 `$queryRaw` 拿行锁（不推荐，违反铁律 5）。
