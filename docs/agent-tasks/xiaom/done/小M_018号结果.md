### 7.3 状态串台风险推演（P0-3）

| 步骤 | 操作 | 状态 |
|------|------|------|
| 1 | 用户点 A "详情" | detailUser=null → 加载 → detailUser=A |
| 2 | A 弹窗输入 reason="A 的原因" | payPwdResetReason="A 的原因" |
| 3 | A 弹窗输入 suffix="1234" | payPwdResetSuffix="1234" |
| 4 | 用户**不点重置按钮**，**不关闭详情** | 状态保留 |
| 5 | 用户点 B "详情" | handleViewDetail(B.id) |
| 5a | setDetailUser(null) | detailUser=null |
| 5b | fetch B 详情 | |
| 5c | setDetailUser(B) | detailUser=B |
| 6 | **B 弹窗打开** | |
| 6a | L1109: `detailUser.hasPaymentPassword` = undefined → 显示"未设置" | 永远"未设置" |
| 6b | L1140: `!detailUser.hasPaymentPassword` = true → 显示"无需重置" | 永远"无需重置" |
| 6c | **重置按钮不显示** | |

**等等**！因为 P0-2 永远显示"未设置"，**重置按钮本来就不显示**，所以**用户根本无法在 B 弹窗触发重置**。

**等等**，但 P0-3 仍然成立 — 如果**未来** P0-2 被修复（详情接口返回 hasPaymentPassword），状态串台就会立刻爆发：
- A 的 reason 文本还在 `payPwdResetReason` 状态
- A 的 suffix 还在 `payPwdResetSuffix` 状态
- 用户在 B 弹窗输入新 reason 时会覆盖 A 的
- 但如果用户**直接**用 A 留下的旧 suffix + 输入新 reason → 触发重置 → **实际重置 B 但日志记录混合的 reason**

**结论**：P0-3 是**潜在 P0**（依赖 P0-2 修复才完全暴露），但**修复时必须同时清空**（任务 7.6 明确要求）。
