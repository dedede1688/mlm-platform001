**当前状态设计**：

### 5.2 严重性判定

**问题本质**：
- **业务规则违反**：013 号业务规则第 2 条明确"同一购物车项目再次确认时必须跳过 POST /api/orders，直接支付原订单"。A 在第 14 步已有待支付订单 O_A，但 `pendingOrder.cartItemId` 已被 B 覆盖（A 在 client 端无法识别"我有未完成订单"）
- **数据库脏数据**：DB 出现两张相同商品的 pending 订单（O_A 和 O_A2）
- **用户体验**：用户在"我的订单"看到两张 A 订单（金额相同、商品相同、状态相同）

**可触发性（不是理论问题）**：
- 用户购物车 ≥ 2 件商品 + 至少 2 件都余额不足 + 用户多次切换 → 100% 触发
- 业务场景：用户想分批支付多件商品，余额不足时分批转入收益 → 频繁切换
- 不需要任何异常操作，正常用户流程

**P0 判定**：
- 业务规则明确违反 ✓
- 数据库脏数据 ✓
- 现实可触发 ✓
- 违反胡子哥在 013 号已确认的业务规则 ✓

**结论**：**P0 - 必须在发布前修复**。

### 5.4 修复建议（供猫爪参考，不在本任务范围）

**最小修复方案**：把 `PendingCartOrder | null` 单值改为 `Record<cartItemId, orderId>` 字典。

```typescript
// 状态改为字典：每个 cartItemId 独立保存自己的 orderId
const [pendingOrders, setPendingOrders] = useState<Record<string, string>>({})
//                                        cartItemId → orderId

// handleBuyNow（变更最小）：
const isSamePendingItem = !!pendingOrders[item.id]
if (!isSamePendingItem) {
  setShortage(0)
}
setShowEarningsTransfer(false)
setCheckoutItem(item)

// handleCheckoutConfirm（复用判断）：
const orderId = pendingOrders[checkoutItem.id]  // 有就复用
if (!orderId) {
  // 创建订单 → setPendingOrders(prev => ({...prev, [checkoutItem.id]: orderId}))
}

// 支付成功后清除：
setPendingOrders(prev => {
  const next = {...prev}
  delete next[checkoutItem.id]
  return next
})
```

**修复影响**：
- `pendingOrder` 字符串全部改为 `pendingOrders`（约 8 处）
- `hasPendingOrder = !!pendingOrders[checkoutItem?.id]` 简化判断
- `PendingCartOrder` interface 可删除
- 测试需更新：`PendingCartOrder`、`pendingOrder.*cartItemId` 正则匹配
- **不变**：业务逻辑（创建/复用/支付）、UI、其它状态、其它测试

**修复后 A→B→A 推演**：
- 第 8 步：setPendingOrders({A: O_A, B: O_B}) ← **两个都保留**
- 第 11 步：isSamePendingItem = !!pendingOrders[A.id] = true → **不调 setShortage(0)**
- 第 14 步：`orderId = pendingOrders[A.id] = O_A` → **直接复用，跳过 POST /api/orders** ✓

**也可以同步把 `shortage` 改字典**（更彻底）：`shortages: Record<cartItemId, number>`，但需要更多改动。建议只改 `pendingOrders`，`shortage` 单值够用（因为只有当前 checkoutItem 才显示）。
