# 智能体任务文件夹协作机制

## 1. 目的

这个目录用于管理小酷、猫爪、小 M 之间的半自动任务交接。

角色说明：

- 小酷：负责理解系统、方案设计、任务拆解、审核和风险控制。
- 猫爪：负责执行代码、运行测试、提交、推送和部署验证；只有任务明确授权时才允许提交或推送。
- 小 M：负责独立只读复审和验收报告；默认不改代码、不提交、不推送。

目标：把散落在聊天里的提示词，升级为可编号、可追踪、可复盘的任务文件和结果文件。

## 2. 核心规则

第一阶段采用“半自动”流程。

小酷创建任务文件；胡子哥手动通知猫爪或小 M 读取任务文件并执行；执行方生成结果文件；小酷读取结果文件并决定下一号任务。

除非工具明确支持“自动监听文件夹”，否则不要假设猫爪或小 M 会自动发现新增文件。

## 3. 目录结构

```text
docs/agent-tasks/
  README.md                         # 协作规则
  templates/                        # 模板
    task-template.md                # 任务模板
    result-template.md              # 结果模板
    review-template.md              # 小酷审核模板
  xiaom/                            # 小 M 任务区
    todo/                           # 待执行
    done/                           # 已完成结果
    archived/                       # 已归档
  catpaw/                           # 猫爪任务区
    todo/                           # 待执行
    done/                           # 已完成结果
    archived/                       # 已归档
  review/                           # 小酷审核结果
```

## 4. 命名规则

文件名优先使用中文，方便胡子哥直接识别。

示例：

```text
docs/agent-tasks/xiaom/todo/小M_001号任务.md
docs/agent-tasks/xiaom/done/小M_001号结果.md
docs/agent-tasks/catpaw/todo/猫爪_001号任务.md
docs/agent-tasks/catpaw/done/猫爪_001号结果.md
docs/agent-tasks/review/小酷_001号审核.md
```

编号规则：

- 使用三位数字：001、002、003。
- 不复用编号。
- 一个任务文件对应一个结果文件。
- 任务废弃时，移动到 `archived（已归档）`，并在文件里说明废弃原因。

## 5. 角色规则

### 小酷

- 创建任务文件。
- 定义任务范围、允许操作、禁止操作和输出格式。
- 审核结果文件。
- 决定下一步是修复、提交、推送、部署，还是继续复审。

### 猫爪

- 只执行分配给猫爪的任务文件。
- 不读取旧任务或无关任务，除非任务文件明确要求。
- 任务文件没有明确允许时，不得 commit（提交）或 push（推送）。
- 完成后必须把结果文件写到 `catpaw/done（猫爪已完成结果目录）`。

### 小 M

- 默认只做只读复审。
- 只读复审任务中，不得改代码、不得 commit（提交）、不得 push（推送）、不得部署。
- 完成后必须把结果文件写到 `xiaom/done（小 M 已完成结果目录）`。

## 6. 任务文件必须包含

每个任务文件必须包含：

1. 任务身份。
2. 背景。
3. 目标。
4. 允许操作。
5. 禁止操作。
6. 文件范围。
7. 检查清单。
8. 验证命令。
9. 输出文件路径。
10. 停止条件。

## 7. 结果文件必须包含

每个结果文件必须包含：

1. 最终结论。
2. 文件清单。
3. 做了什么。
4. 没做什么。
5. 测试和验证结果。
6. P0/P1/P2 风险。
7. git status（代码状态）结果。
8. commit（提交）和 push（推送）状态。
9. 建议下一步。

## 8. 安全规则

- 不要让猫爪和小 M 同时修改同一批文件。
- 不要把任务文件夹机制和业务代码混在一个提交里，除非胡子哥明确批准。
- 如果出现 `AGENTS.md（项目规则文件）`、数据库结构文件、迁移文件或无关模块被修改，必须停止并报告。
- 涉及资金、支付、提现、充值、奖励、订单的任务，提交前必须经过小 M 复审。
- 涉及页面改动时，必须做浏览器验证；如果无法浏览器验证，必须说明原因并做源码级验证。

## 9. 英文使用规则

能用中文就用中文。

必须保留英文时，后面立刻加中文解释。

示例：

- commit（提交）
- push（推送）
- build（构建）
- typecheck（类型检查）
- route（接口路由）
- service（服务层）
- Vercel（部署平台）
- todo（待执行）
- done（已完成）
- archived（已归档）
