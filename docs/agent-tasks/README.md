# 智能体任务文件夹协作机制

## 1. 目的

这个目录用于管理小酷、猫爪、小 M 之间的任务协作。

目标：把散落在聊天里的提示词，升级为可编号、可追踪、可复盘的任务文件和结果文件。所有文件纳入 git（版本控制系统）管理。

## 2. 角色定义

| 角色 | 核心职责 | 权限边界 |
|------|----------|----------|
| **胡子哥**（胡子老师） | 业务决策、授权发布、真实浏览器验收 | 唯一能做业务决策的人 |
| **小酷** | 方案设计 + 代码执行 + 自审 + commit/push + 协调小 M 和猫爪 | 直接写代码并提交，不再只写任务文件 |
| **猫爪** | 辅助执行：代码清理、文档管理、格式化、补测试等机械操作 | 默认禁止 commit/push/deploy，除非任务明确授权 |
| **小 M** | 独立只读复审 | 严格只读，不改代码、不提交、不推送 |

小酷同时承担方案设计和代码执行，消除"方案→执行"之间的信息传递损耗。猫爪不再承担核心业务代码开发。

## 3. 变更分级

所有变更按影响范围分为三级，走不同流程：

### S 级（小修）

- **定义**：改动 < 20 行，不影响业务逻辑，有测试覆盖
- **流程**：小酷直接改 → 跑测试 → commit → push
- **举例**：修按钮 disabled、补测试用例、修 typo（笔误）、改样式
- **不走任务文件，不走小 M 复审**（小 M 可事后抽查）

### A 级（标准）

- **定义**：新功能或改动 > 20 行，影响业务逻辑
- **流程**：写设计文档 → TDD 开发 → 自审 → commit → push → 小 M diff 复审 → 胡子哥验收
- **举例**：退款凭证校验、充值审核流程、通知系统

### P 级（关键）

- **定义**：涉及资金流、权限、数据库结构
- **流程**：写设计文档 → TDD 开发 → 自审 → **小 M 提交前复审** → 胡子哥验收 → commit → push
- **举例**：佣金计算、提现审核、积分解锁、订单支付

### 分级速查

| 维度 | S 级 | A 级 | P 级 |
|------|------|------|------|
| 改动量 | < 20 行 | > 20 行 | 任意 |
| 影响业务逻辑 | 否 | 是 | 是 |
| 涉及资金/权限 | 否 | 否 | 是 |
| 需要任务文件 | 否 | 是 | 是 |
| 需要设计文档 | 否 | 是 | 是 |
| 小 M 复审时机 | 事后抽查 | 提交后 diff | 提交前 |
| 胡子哥验收时机 | 不需要 | 部署后 | 部署前 |
| commit 时机 | 直接 commit | 自审后 commit | 复审通过后 commit |

## 4. 核心规则

### 4.1 协调方式

小酷直接协调猫爪和小 M，不再让胡子哥当传话人。小酷创建任务文件后，直接通知猫爪或小 M 执行。

### 4.2 任务文档纳入 git

所有 `docs/agent-tasks/` 下的文件必须 commit 到版本库，包括任务文件、结果文件、岗位说明、模板。不得留为 untracked（未跟踪）文件。

任务文档和业务代码不要混在同一个 commit（提交）里。

### 4.3 代码提交规则

- S 级和 A 级变更：小酷自审通过后直接 commit + push，不需要等复审
- P 级变更：小 M 复审通过后才 commit + push
- 猫爪：默认禁止 commit/push，除非任务明确授权
- 所有 commit 必须精确 `git add`，禁止 `git add .` 或 `git add -A`

### 4.4 部署与验收解耦

- S 级和 A 级变更：代码 push 后 Vercel（部署平台）自动部署，胡子哥在已部署版本上验收
- P 级变更：胡子哥验收通过后才 push
- 胡子哥暂时没空验收不阻塞后续开发——代码已经在最新版本上

## 5. 目录结构

```text
docs/agent-tasks/
  README.md                         # 协作规则（本文件）
  templates/                        # 模板
    task-template.md                # 任务模板（A/P 级）
    result-template.md              # 结果模板
    review-template.md              # 小酷审核模板
  xiaom/                            # 小 M 任务区
    小M复审岗位说明.md               # 岗位说明
    todo/                           # 待执行
    done/                           # 已完成结果
    archived/                       # 已归档
  catpaw/                           # 猫爪任务区
    猫爪执行岗位说明.md               # 岗位说明
    todo/                           # 待执行
    done/                           # 已完成结果
    archived/                       # 已归档
  review/                           # 小酷审核结果
```

## 6. 命名规则

文件名优先使用中文，方便胡子哥直接识别。

示例：

```text
docs/agent-tasks/xiaom/todo/小M_001号任务.md
docs/agent-tasks/xiaom/done/小M_001号结果.md
docs/agent-tasks/catpaw/todo/猫爪_001号任务.md
docs/agent-tasks/catpaw/done/猫爪_001号结果.md
docs/agent-tasks/review/小酷_001号审核.md
```

编号规则：

- 使用三位数字：001、002、003。
- 不复用编号。
- 一个任务文件对应一个结果文件。
- 任务废弃时，移动到 `archived（已归档）`，并在文件里说明废弃原因。

## 7. 角色规则

### 小酷

- 设计方案、写设计文档和实施计划。
- 直接编写核心业务代码，TDD 流程：先写失败测试 → 确认红灯 → 写实现 → 确认绿灯。
- 自审：跑全量测试 + typecheck（类型检查）+ build（构建）。
- 按变更分级决定 commit 时机。
- 创建猫爪和小 M 的任务文件，定义范围和禁止操作。
- 审核猫爪和小 M 的结果文件。
- 协调整体节奏，直接通知猫爪和小 M，不经过胡子哥传话。

### 猫爪

- 只执行小酷分配的辅助任务。
- 典型任务类型：清理垃圾文件、任务文档入库、补测试用例、CSS/格式调整、依赖版本升级。
- 不承担核心业务代码开发。
- 默认禁止 commit/push/deploy，除非任务明确授权。
- 完成后必须把结果文件写到 `catpaw/done（猫爪已完成结果目录）`。
- 任务完成后停止，等待小酷的下一步指令。

### 小 M

- 独立只读复审，不改代码、不提交、不推送。
- P 级变更：提交前复审——小酷完成代码后通知小 M，小 M 复审通过后小酷才能 commit。
- A 级变更：提交后 diff 复审——小酷 commit + push 后，小 M 对 commit diff（差异）做复审。
- S 级变更：事后抽查——小 M 可随时对最近的 S 级 commit 做 diff 抽查。
- 完成后必须把结果文件写到 `xiaom/done（小 M 已完成结果目录）`。
- 复审重点：资金安全边界、前后端校验一致性、文件范围合规、测试可信度、禁止范围未越界。

## 8. 任务文件要求

### S 级变更

不需要任务文件。小酷直接在 commit message（提交信息）里写清楚改了什么、为什么改。

### A 级和 P 级变更

使用 `templates/task-template.md（任务模板）`，精简为 6 个必填项：

1. 任务目标（3-5 行）
2. 允许修改的文件清单（精确到路径）
3. 禁止操作
4. 业务规则要点
5. 验证命令
6. 完成标准

## 9. 结果文件要求

### 小酷的 A/P 级变更

结果直接写在 commit message 里：
1. 改了什么（文件清单）
2. 为什么改（业务理由）
3. 测试结果（vitest + tsc 输出摘要）
4. 验证命令输出

### 猫爪和小 M 的任务

使用 `templates/result-template.md（结果模板）`，精简为 5 项：
1. 最终结论
2. 修改文件清单（猫爪）或复审文件清单（小 M）
3. 验证结果
4. git status 最终状态（猫爪）
5. 下一步建议

## 10. 安全规则

- 不要让猫爪和小 M 同时修改同一批文件。
- 不要把任务文件夹机制和业务代码混在一个 commit 里。
- 如果出现 `AGENTS.md（项目规则文件）`、数据库结构文件、迁移文件或无关模块被修改，必须停止并报告。
- P 级变更（涉及资金、支付、提现、充值、奖励、订单）提交前必须经过小 M 复审。
- 涉及页面改动时，必须做浏览器验证；如果无法浏览器验证，必须说明原因并做源码级验证。

## 11. 英文使用规则

能用中文就用中文。

必须保留英文时，后面立刻加中文解释。

示例：

- commit（提交）
- push（推送）
- build（构建）
- typecheck（类型检查）
- route（接口路由）
- service（服务层）
- Vercel（部署平台）
- todo（待执行）
- done（已完成）
- archived（已归档）
- diff（差异）
