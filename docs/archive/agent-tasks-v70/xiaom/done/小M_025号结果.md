### 🔴 P1-2：非法数字 `?? 0` 兜底（**反向违反**）

**任务 5.4 原文（必须遵守）**：
> "非法数字返回 null 的目的，是让调用方进入错误状态；?? 0 会把非法数字重新变成'零缺口'；不能让页面显示恢复成功或覆盖会话。"

**猫爪报告 P1-2 第 14 行（实际执行）**：
> "所有 3 个调用方增加 `?? 0` 兜底"

**代码核验**（`src/app/products/[id]/page.tsx`）：

| 调用方 | 行号 | 代码 | 违规行为 |
|--------|------|------|---------|
| `restorePendingPayment` | **L347** | `const shortage = calculatePendingShortage(orderPayAmount, latestUser.balance) ?? 0` | **P0 风险**：null→0 → `setPendingPayment({shortage:0})` + `persistPendingPayment(orderId, 0)` **覆盖原会话** + `setRestoreStatus('restored')` **伪装恢复成功** |
| `handleCheckoutConfirm` (INSUFFICIENT) | **L476** | `calculatePendingShortage(paymentAmount, user.balance) ?? 0` | **P1**：null→0 → 写入 `nextPendingPayment.shortage=0` + `persistPendingPayment(orderId, 0)`，但 verify-payment 会再次失败，所以**实际危害较小** |
| `handleEarningsTransferSuccess` | **L521** | `calculatePendingShortage(pendingPayment.payAmount, latestUser.balance) ?? 0` | **P1**：null→0 → `setPendingPayment({shortage:0})` + `persistPendingPayment(next.orderId, 0)`，**覆盖原会话**为 0 缺口 |

**3 个调用方全部使用 `?? 0` 兜底**，**直接违反任务 5.4 核心要求**。

**真实危害推演**：

```text
场景 A: restorePendingPayment 路径（最严重）
────────────────────────────────────────────
1. 用户余额 = 200, 订单 payAmount = 500
2. 后端 /api/users/me 短暂返回 500（fetchUser 失败）
3. 实际 latestUser 拿到 null → 进入 validation_error → 不写入 shortage=0 → return
4. ✅ P1-1 这里正确处理了

场景 B: 假设 fetchUser 成功但 calculatePendingShortage 返回 null
────────────────────────────────────────────────────
例如：最新修复的 calculatePendingShortage 把 balance=-1 当 null 返回
或者：未来某次重构后 latestUser.balance 类型变了
1. latestUser = { id, balance: -1, ... } （某种异常数据）
2. calculatePendingShortage(500, -1) → null
3. ?? 0 → shortage = 0
4. setPendingPayment({shortage: 0, ...}) → 页面显示"已恢复待支付订单，不会重复下单"
5. persistPendingPayment(orderId, 0) → sessionStorage 被覆盖 shortage=0
6. setRestoreStatus('restored') → 显示绿色"已恢复"提示
7. 用户看到"已恢复"+"差额 0"，但其实支付会失败（因为数据异常）
8. 用户重新点击确认支付 → 走 verify-payment → 500 → 用户困惑
```

**结论**：`restorePendingPayment` 调用方（`:347`）的 `?? 0` 直接让 P1-1 失效，等于**新增了一个 P0 风险**（覆盖原 sessionStorage + 显示恢复成功）。其他两个调用方也违反任务要求，但危害相对小。

**铁律 12**（v46.11/v46.12）：所有 admin 状态变更路由必须自动通知用户——这里虽然不是 admin 路由，但是是 **用户资金状态变更** 路径，非法数字被吞掉会让用户状态错乱，违反铁律 12 的精神。

**任务 5.4 末尾**：
> "如果任一调用方仍用 `?? 0` 直接继续业务流程，列为 P1；如果导致恢复成功或覆盖原会话，可提升为 P0。"

`restorePendingPayment` 调用方**触发恢复成功 + 覆盖原会话**，按任务标准**升级为 P0**。
