### 7.1 风险点源码

### 7.2 `prev[checkoutItem.id]` 何时不存在？

**正常路径分析**：
- `handleEarningsTransferSuccess` 在 `<EarningsTransferModal onSuccess={...}>` 中调用（L628）
- `EarningsTransferModal` 的 `onSuccess` 在 `fetch` 成功后立即调用（L96-99）
- 调用 `onSuccess` 时 `showEarningsTransfer === true`（因为只有 `setShowEarningsTransfer(true)` 后 modal 才打开）
- `setShowEarningsTransfer(true)` 之前必然 `setPendingPayments` 写入 `pendingPayments[checkoutItem.id] = {orderId, shortage}`（L277-282）
- **所以正常路径下 `prev[checkoutItem.id]` 必然存在**，`?? ''` 不可触发

**但代码层面没有守卫**：
- 没有 `if (!current) return prev`
- `?? ''` 显式允许空字符串
- 如果某天状态管理被改动（比如未来 `setPendingPayments({})` 被误加、外部重置、并发竞争），可能触发

### 7.3 触发后会发生什么？

假设 `prev[checkoutItem.id]` 不存在：

1. **写入伪记录**：`pendingPayments[checkoutItem.id] = { orderId: '', shortage: X }`
2. **下次确认**：用户点"重新确认支付" → `handleCheckoutConfirm` L212-214：
   ```typescript

   if (existingPayment) {  // ← 对象存在，truthy！
     orderId = existingPayment.orderId  // ← orderId = '' 空字符串
   }
   ```
3. **跳过创建订单**（因为 existingPayment 存在）→ 走 verify-payment
4. **请求错误 URL**：
   ```typescript
   const verifyRes = await fetch(`/api/orders/${orderId}/verify-payment`, ...)
   // 拼成 '/api/orders//verify-payment'（双斜杠）
   ```
5. **后端返回 500/404**
