# 小M_024号结果：商品详情待支付恢复独立只读复审

## 1. 最终结论

**不通过**。

猫爪 024 号业务实现存在 **2 个 P0/P0 候选**（必须修复）和 **3 个 P1**（建议修复）：

- **🔴 P0 候选 1：待支付状态下 `handleConfirm` 仍校验 `recipientName` 等 state，但 state 在关闭弹窗时已被 200ms 定时器清空，`useEffect` 不从 `lockedShipping` 重新填充。**没有地址簿默认地址的用户在待支付状态下重新确认支付会被 `L194 if (!recipientName.trim())` 拦截，提示"请输入收件人姓名"，永远无法完成支付。**
- **🔴 P0 候选 2：`paymentVerified` 字段被白名单遗漏**——`OrderService.getOrderDetail` 的 select 不含 `paymentVerified`，`[id]/route.ts` GET 构造的 `publicOrder` 不含 `paymentVerified`，但 `dashboard/orders/[id]/page.tsx` L308 `order.paymentVerified && (<span>支付已验证</span>)` 永远 false，**"支付已验证"绿色徽章永远不显示**。既有页面功能被破坏。
- **🟡 P1 候选 3：`restorePendingPayment` 资金读取失败时 `shortage = 0` + `setRestoreStatus('restored')`**，违反"不得显示'已恢复'或让用户误以为无需补余额"原则。
- **🟡 P1 候选 4：`calculatePendingShortage` 缺 NaN/Infinity 校验**，依赖调用方保护。
- **🟡 P1 候选 5：测试部分字符串断言弱**（"任意 hasPendingOrder ?"匹配、"CheckoutLockedShipping 字符串存在"等），不证明真实行为。

### 5.3 根因

`handleConfirm` L194-204 校验的是 **state**（`recipientName / recipientPhone / addressPca / detailAddress`），**不是** `lockedShipping`。

而 `useEffect` L138-167 在弹窗打开时不从 `lockedShipping` 重新填充 state（只从 `existingAddresses` 默认地址填充）。

**修复方向**（仅供 025 号参考）：
- 方案 A：`handleConfirm` 顶部判断 `if (hasPendingOrder && lockedShipping)` → 用 `lockedShipping.recipientName` 等代替 state 校验
- 方案 B：useEffect 打开弹窗时，如果 `hasPendingOrder && pendingOrderShipping` → 用 `pendingOrderShipping` 填充 state
- 方案 C：`lockedShipping` 强制覆盖 state（但 state 仍要保留给 handleConfirm L213 构造 shippingSnapshot）

export function calculatePendingShortage(payAmount: number, balance: number): number {
  return Math.max(0, payAmount - balance)
}

### 7.2 任务 P1 候选 4 三项要求逐条检查
