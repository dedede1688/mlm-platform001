## 三、背景说明

胡子哥发现后台「会员详情」里没有收益账户字段，导致无法给测试用户增加「可用收益」，从而无法继续测试「收益转余额」和「余额不足时引导从收益转入余额」流程。

后端调账接口已经支持：

```text
earnings_add
```

所以本次猫爪任务只应该改后台会员管理前端页面，不应该改后端。

## 四、重点检查项

请逐项检查以下内容：

### 1. 修改范围

确认业务代码是否只修改了：

```text
src/app/admin/users/page.tsx
```

不应修改：

- 后端接口
- 数据库结构
- `prisma/schema.prisma`
- `AGENTS.md`
- 支付逻辑
- 订单逻辑
- 提现逻辑
- 收益转余额后端逻辑

### 2. 会员详情是否新增「收益账户」区域

检查「会员详情」弹窗中是否新增了「收益账户」区域。

必须展示：

| 字段 | 中文展示 |
|---|---|
| `earningsAvailable` | 可用收益 |
| `earningsFrozen` | 冻结收益 |
| `earningsPending` | 待结算收益 |
| `earningsVoided` | 作废收益 |

检查这些字段是否使用 `?? 0` 兜底，避免空值报错。

### 3. 资金调整是否新增「可用收益」

检查资金调整下拉框是否新增：

```text
可用收益
```

并确认提交给后端的类型是：

```text
earnings_add
```

### 4. 是否只允许增加可用收益

检查当类型为 `earnings_add` 时：

- 金额必须大于 0。
- 输入 0 或负数时，前端必须拦截。
- 提示文案应类似：

```text
本次只允许增加可用收益
```

不得出现减少收益、作废收益、余额转回收益等入口。

### 5. 确认弹窗是否适配收益类型

如果页面有二次确认弹窗，请检查：

- 类型中文是否显示为「可用收益」
- 当前值是否取 `earningsAvailable`
- 金额格式是否正常
- 不要错误显示为「余额」或「冻结余额」

### 6. 成功后是否沿用原刷新逻辑

检查调账成功后是否沿用已有刷新逻辑：

- 刷新会员列表
- 如果详情弹窗打开，应能看到最新收益数据

不要为了本任务新造复杂刷新流程。

### 7. 验证命令结果

请检查猫爪是否运行并通过：

```powershell
npx tsc --noEmit -p tsconfig.typecheck.json
npx next build
```

如果你需要自己复核，可以只读运行验证命令，但不要修改代码。
