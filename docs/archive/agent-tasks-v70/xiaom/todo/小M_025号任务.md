## 五、强制复审项

### 5.1 P0：待支付状态必须无条件锁定地址区域

重点核查 `src/components/checkout/CheckoutDialog.tsx`。

任务要求是：

1. `hasPendingOrder=true` 时，无论 `lockedShipping` 是否存在，都不得重新显示可编辑地址表单。
2. `lockedShipping` 完整时，显示只读收货摘要，并允许用户只输入支付密码重新确认。
3. `lockedShipping` 缺失或不完整时，显示只读错误提示并禁止提交。
4. 不得把缺失快照回退成首次下单模式。
5. 不得创建第二个订单或让用户误以为修改了原订单地址。

特别检查当前代码是否仍是：

```tsx
hasPendingOrder && lockedShipping ? lockedSummary : editableForm
```

如果是，则真实含义为：`hasPendingOrder=true` 但 `lockedShipping=null` 时会显示可编辑表单，违反任务要求。

同时检查 `handleConfirm` 是否仍是：

```typescript
if (hasPendingOrder && lockedShipping) {
  // 锁定逻辑
} else {
  // 可编辑字段校验
}
```

如果是，则 `hasPendingOrder=true` 且快照缺失时仍会进入可编辑字段校验和提交路径，也违反要求。

必须独立推演以下场景：

```text
hasPendingOrder=true
pendingOrderShipping=null
submittedShipping=null
```

确认页面显示什么、按钮是否可提交、是否会调用 `onConfirm`。

### 5.2 P0：无默认地址用户重新支付路径

逐步推演：

1. 用户无地址簿默认地址。
2. 首次填写姓名、手机号和地址。
3. 创建订单后余额不足。
4. 关闭弹窗，200ms 后可编辑 state 被清空。
5. 重新打开同一商品。
6. 父组件传入完整 `pendingOrderShipping`。
7. 用户只输入支付密码。
8. `handleConfirm` 是否使用锁定快照。
9. 是否完全跳过隐藏 state 校验。
10. 是否复用原订单编号。

此路径必须能通过。若不能，列为 P0。

### 5.3 P0：`paymentVerified` 兼容字段

检查：

1. `OrderService.getOrderDetail()` select 是否有 `paymentVerified: true`。
2. GET 路由公开对象是否有 `paymentVerified: order.paymentVerified`。
3. true 和 false 是否都返回。
4. 响应仍不得包含：
   - `userId`
   - `user`
   - `passwordHash`
   - `paymentPasswordHash`
5. 不得使用 `{ ...order }`。

### 5.4 P1：非法数字不得用 `?? 0` 吞掉

重点检查 `src/app/products/[id]/page.tsx` 中 `calculatePendingShortage()` 的所有真实调用入口。

猫爪报告写明“三个调用方增加 `?? 0` 兜底”。必须判断这是否违反任务要求：

- 非法数字返回 `null` 的目的，是让调用方进入错误状态；
- `?? 0` 会把非法数字重新变成“零缺口”；
- 不能让页面显示恢复成功或覆盖会话。

至少检查：

1. 恢复待支付订单：计算结果为 null 时，是否进入 `validation_error`，不得 restored。
2. 支付余额不足兜底计算：结果为 null 时，是否停止并提示错误，不得持久化 shortage=0。
3. 收益转入成功后的重新计算：结果为 null 时，是否保留原待支付记录并提示重试，不得写 shortage=0。

如果任一调用方仍用 `?? 0` 直接继续业务流程，列为 P1；如果导致恢复成功或覆盖原会话，可提升为 P0。

### 5.5 P1：资金接口失败处理

检查 `restorePendingPayment`：

1. `fetchUser` 返回 null 时进入 `validation_error`。
2. 不调用 `setPendingPayment`。
3. 不调用 `persistPendingPayment`。
4. 不清除 sessionStorage。
5. 保留重新验证入口。
6. “重新验证”按钮真实调用 `restorePendingPayment(true)`。

### 5.6 P1：测试是否为错误实现放宽

重点审查：

- `__tests__/components/checkout-pending-order-lock.test.ts`
- `__tests__/components/product-pending-payment-flow.test.ts`

必须检查：

1. 测试是否接受 `hasPendingOrder && lockedShipping ? ... : editableForm`。
2. “不显示可编辑地址表单”测试是否只检查 true 分支，而没有检查 `lockedShipping=null` 情况。
3. “快照缺失禁止提交”是否有真实断言，还是只检查字段名存在。
4. 是否仍有 `if (match) { expect(...) }` 静默跳过。
5. `extractBlock` 起点或终点找不到时是否真实失败。
6. 是否存在固定 300/500 字符窗口造成跨函数或漏检。
7. 测试是否真实证明 `restorePendingPayment(true)` 的点击关系。
8. 是否测试三个调用方对 `calculatePendingShortage=null` 的处理，而不只是测试工具函数本身。

测试通过不能替代业务推演。如果测试与任务要求冲突，应指出测试本身有问题。

### 5.7 支付与订单完整链路

按铁律 6 独立核对：

```text
创建订单
→ 保存原订单编号、payAmount、pointsUsed、收货快照
→ verify-payment 返回余额不足
→ 收益转余额或去充值
→ 返回后验证原订单
→ 手动重新输入支付密码
→ 复用原订单编号
→ verify-payment
→ 支付成功
→ 清理 sessionStorage
```

确认：

- 不自动支付；
- 不绕过支付密码；
- 不重复创建订单；
- 不修改支付事务；
- 不使用页面实时价格替代原订单 `payAmount`；
- 恢复原订单 `pointsUsed`。
