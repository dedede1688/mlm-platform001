## 七、P0-2：三个缺口调用入口复审

独立检查 `src/app/products/[id]/page.tsx`。

- 不调用 `persistPendingPayment`；
- 不调用 `clearProductPendingPayment`；

### 7.2 `handleCheckoutConfirm` 余额不足分支

确认：

- 不再有 `calculatePendingShortage(...) ?? 0`；
- 负数、NaN、Infinity、null、空字符串等非法 `data.shortage` 不得被转换成 0；
- 只有真实 number 且有限、非负时才直接采用接口缺口；
- 兜底计算返回 `null` 时进入 `validation_error`；
- 不打开收益转余额弹窗；
- 不自动支付；
- 不清除原订单编号；
- 不覆盖合法的原会话。

### 小酷已发现的重点疑点 B

当前代码先执行：

```typescript
const apiShortage = Number(verifyErr.data?.shortage)
```

再执行 `Number.isFinite(apiShortage)`。请重点验证：

- `Number(null) === 0`；
- `Number('') === 0`；
- `Number(false) === 0`。

如果接口返回这些非法类型，当前实现是否会把非法缺口当作 0。必须给出明确结论和风险等级。

### 7.3 新订单创建后、支付校验前的零缺口持久化

当前代码在创建订单后、调用 `verify-payment` 前执行：

```typescript
shortage: 0

persistPendingPayment(orderId, 0)
```

随后如果余额不足响应的缺口非法，且兜底计算也返回 `null`，代码只进入 `validation_error` 并 return。

必须独立推演并回答：

1. 新订单编号是否保留；
2. sessionStorage 是否留下 `shortage: 0`；
3. 用户点击“重新验证”后会发生什么；
4. 是否可能显示“已恢复”、隐藏真实缺口或允许错误流程继续；
5. 是否违反“非法缺口不得持久化为 0、不得覆盖原会话”；
6. 现有测试是否覆盖这个完整路径。

这是本次复审的最高优先级边界，不得只看 null 分支附近 300 个字符。

### 7.4 `handleEarningsTransferSuccess`

确认：

- `fetchUser` 返回 null 时不猜测缺口；
- 关闭收益弹窗并进入 `validation_error`；
- 保留原 pendingPayment 和原 sessionStorage；
