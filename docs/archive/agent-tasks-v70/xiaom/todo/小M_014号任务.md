## 三、必须检查的 10 项

### 1. 修改范围

确认 014 号本轮新增差异只涉及：

```text
src/app/cart/page.tsx
__tests__/components/cart-earnings-transfer-flow.test.ts
```

说明：013 号整体工作区仍有另外 2 个业务文件处于修改状态，这是前置改动，不要误判为 014 越界。

### 2. 商品切换判断依据

确认 `handleBuyNow` 使用 `pendingOrder?.cartItemId === item.id` 或严格等价逻辑。

如果只依赖 `checkoutItem?.id`，判定不通过，因为关闭结算弹窗后 `checkoutItem` 已经是 `null`。

### 3. 商品 B 不继承商品 A 的余额缺口

确认：

- `pendingOrder.cartItemId` 与新商品 `item.id` 不同时调用 `setShortage(0)`。
- 打开新结算弹窗前调用 `setShowEarningsTransfer(false)`。
- 商品 B 不显示商品 A 的缺口和残留收益转余额弹窗。

### 4. 关闭结算弹窗的状态规则

确认 `handleCheckoutClose`：

- 调用 `setShowEarningsTransfer(false)`。
- 不调用 `setPendingOrder(null)`。
- 不调用 `setShortage(0)`。
- 保留“订单已创建，可在我的订单中继续支付”的提示。

### 5. 同一商品直接重新打开

确认以下状态流成立：

```text
商品 A 余额不足
→ 关闭结算弹窗
→ 直接重新打开商品 A
→ 复用订单 A
→ 原 shortage（余额缺口）仍保留
```

### 6. 多商品往返的重复订单风险（本次重点加查）

不要只检查猫爪任务内已有断言。必须独立推演：

```text
商品 A 创建订单 A，余额不足
→ 打开商品 B
→ 商品 B 创建订单 B，pendingOrder 被覆盖为 B
→ 再打开商品 A
→ 是否还能识别并复用订单 A？
```

当前实现如果只有单个：

```typescript
PendingCartOrder | null
```

而不是按 `cartItemId` 保存多个待支付订单，则可能在 A→B→A 时重复创建订单 A。

请明确判定：

- 这是现实可触发的问题还是理论问题。
- 是否违反 013 号“同一购物车项目再次确认必须复用原订单”的业务规则。
- 风险等级应列为 P0、P1 还是 P2。
- 是否必须在发布前修复。

不要因为 014 号现有测试全部通过就忽略该场景。

### 7. 不自动重新支付

确认测试通过明确起止标记提取了：

```text
const handleEarningsTransferSuccess
到
const handleCheckoutClose
```

并真实断言：

- 不包含 `verify-payment`。
- 不调用 `handleCheckoutConfirm`。
- 仍调用 `fetchUserInfo`。
- 仍提示用户再次确认支付。

### 8. 测试真实性

检查 `extractBlock` 是否会：

- 明确验证开始标记存在。
- 明确验证结束标记位于开始标记之后。
- 返回真实函数体区域。

同时指出剩余使用固定字符窗口的测试是否仍可能产生假阳性。若与本轮核心规则无关，可列 P2；若仍影响核心规则，应列 P1。

### 9. 禁止范围

确认没有修改：

- 后端支付接口和支付事务。
- 订单状态事务。
- 数据库结构或 Prisma schema（数据库模型）。
- 后台页面。
- `AGENTS.md`。

### 10. 提交状态
