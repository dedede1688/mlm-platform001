## 四、必须逐项检查的关键逻辑

### 1. 待支付订单是否真正复用

必须阅读真实分支，不得只看测试名称：

- 第一次 `POST /api/orders` 成功后，是否在调用 `verify-payment` 前保存 `orderId`。
- 同一个 `cartItemId` 再次确认时，是否进入复用分支并跳过 `POST /api/orders`。
- 支付密码错误、余额不足和请求失败后，是否都保留待支付订单编号。
- 支付成功后是否清除待支付订单编号。
- 是否存在快速双击导致重复下单的可能，确认 `CheckoutDialog` 的 `submitting` 是否真实阻止重复点击。

任何可能导致同一商品重复创建订单的问题列为 P0。

### 2. 余额不足识别

确认：

- 优先识别 `INSUFFICIENT_BALANCE`。
- 旧错误文字兼容只作为兜底。
- `shortage` 优先使用接口返回值。
- 本地计算差额时使用非负兜底。
- 正常支付错误不会误触发收益转余额弹窗。

### 3. 默认转入金额

确认真实调用链为：

```text
Math.min(shortage, earningsAvailable)
```

并确认 `EarningsTransferModal` 内部不会把默认金额放大到超过可用收益。

### 4. 禁止自动支付

必须直接检查 `handleEarningsTransferSuccess` 函数体：

- 不得调用 `verify-payment`。
- 不得调用支付处理函数。
- 不得保存或自动复用支付密码发起请求。
- 只允许刷新资金、更新差额和提示用户再次确认。

猫爪新增的源码契约测试只检查弹窗 JSX 片段，可能无法证明回调函数体中没有自动支付；必须人工读代码确认。

### 5. 收益不足和收益为 0

确认：

- 可用收益大于 0 时才自动打开收益转余额弹窗。
- 可用收益为 0 时不打开不可操作弹窗。
- 确认订单弹窗显示准确剩余差额。
- “去充值”跳转 `/dashboard/recharge`。
- 部分转入后仍不足时，差额会正确减少。

### 6. 强制检查状态串台问题

重点模拟以下流程：

```text
商品 A 支付余额不足
→ shortage 保存商品 A 差额
→ 关闭确认订单弹窗
→ 打开商品 B 的确认订单弹窗
```

检查商品 B 是否会错误显示商品 A 的 `shortage`。

特别检查：

- `handleCheckoutClose` 是否清理或隔离 `shortage`。
- `handleBuyNow` 打开不同购物车项目时是否重置旧差额。
- 重新打开同一个待支付订单时是否应保留差额。
- `showEarningsTransfer` 是否可能在确认订单弹窗关闭后残留。

如果不同商品之间会串用旧差额，列为 P1，提交前必须修复。

### 7. 关闭弹窗与重新进入

确认：

- 已创建订单时关闭弹窗会提示“订单已创建，可在我的订单中继续支付”。
- 不取消、不删除订单。
- 关闭后同一页面重新打开同一个购物车项目时，不应重复创建订单。
- 离开页面后本地状态丢失的限制是否已如实记录。

### 8. 支付成功路径

确认原有流程仍保留：

- 推荐奖励未解锁提示。
- 删除购物车项目。
- 更新购物车本地状态。
- 刷新用户信息。
- 跳转订单详情页。

### 9. EarningsTransferModal 兼容性

确认：

- `onSuccess(numAmount)` 真实传入实际金额。
- 余额页、提现页、订单详情页现有调用继续通过类型检查。
- 评估 `onSuccess: (amount?: number) => void` 使用可选参数是否必要；如果实际永远传 number，可列为 P2 类型设计观察，不阻塞。

## 五、测试真实性检查

猫爪报告称新增 23 条源码契约测试。不得因为测试通过就直接判定通过。

必须检查：

1. “已有待支付订单不再创建订单”测试是否真正锁定条件分支，还是只检查出现了 `pendingOrder` 字符串。
2. “不自动支付”测试是否真正读取成功回调函数体，还是只检查 JSX 属性区域。

4. 测试是否可能在关键逻辑被破坏后仍然通过。

测试过弱但源码正确：列为 P1 测试缺口，建议补强后再提交。
