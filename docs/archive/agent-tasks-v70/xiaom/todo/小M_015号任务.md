## 三、必须检查的 12 项

### 1. 修改范围

确认 015 号本轮只继续修改：

```text

```

### 2. 单值状态是否彻底移除

确认真实业务状态已经改为：

```typescript
Record<string, PendingCartPayment>
```

每条记录必须同时包含：

- `orderId`
- `shortage`

确认不再保留单个 `pendingOrder` 或全局标量 `shortage` 作为另一套真实状态。

### 3. 创建订单写入

确认新订单创建成功后：

- 使用当前 `checkoutItem.id` 作为键。
- 写入真实 `orderId`。
- 初始 `shortage` 为 0。
- 使用函数式更新并展开 `...prev`，不会覆盖其他商品记录。
- 保存发生在 `verify-payment`（支付验证）调用之前。

### 4. 已有订单复用

确认 `handleCheckoutConfirm` 在创建订单之前读取：

```typescript
pendingPayments[checkoutItem.id]
```

存在时必须直接复用其 `orderId`，跳过 `POST /api/orders`。

### 5. 余额不足写入

确认余额不足时：

- 只更新当前 `checkoutItem.id`。
- 保存当前真实 `orderId`。
- 保存 `actualShortage`。
- 展开 `...prev` 保留其他商品记录。
- 不会写入 `null`、`undefined` 或空字符串订单编号。

### 6. 支付成功清理

确认支付成功后：

- 只执行 `delete next[checkoutItem.id]` 或严格等价操作。
- 不使用 `setPendingPayments({})` 清空整张表。
- 商品 B 的待支付记录不会因商品 A 支付成功而丢失。

### 7. A→B→A 完整推演

必须独立按真实代码推演：

```text
A 创建 orderA/shortageA
→ B 创建 orderB/shortageB
→ 返回 A
→ A 复用 orderA 并显示 shortageA
→ A 支付成功
→ B 的 orderB/shortageB 仍存在
```

明确判断小 M 014 号发现的重复订单 P0 是否已经闭合。

### 8. 收益转入只更新当前商品

确认 `handleEarningsTransferSuccess`：

- 读取当前商品的旧缺口。
- 只更新当前商品记录。
- 保留当前商品原 `orderId`。
- 保留其他商品记录。
- 不调用 `verify-payment`。
- 不调用 `handleCheckoutConfirm`。
- 刷新资金并提示用户手动再次确认。

### 9. 空订单编号伪记录风险（重点加查）

当前实现疑似包含：

```typescript
orderId: prev[checkoutItem.id]?.orderId ?? ''
```

请独立判断：

1. 在什么状态下 `prev[checkoutItem.id]` 可能不存在。
2. 收益转余额弹窗是否存在在待支付记录缺失时仍触发成功回调的路径。
3. 如果记录不存在，当前代码是否会创建 `{ orderId: '', shortage: ... }` 伪记录。
4. 伪记录是否会让下一次确认跳过创建订单并调用：

```text
/api/orders//verify-payment
```

5. 风险应列为 P0、P1 还是 P2。
6. 是否必须在发布前改成：记录不存在时返回原状态，不创建空订单编号记录。

建议安全写法（仅供复审建议，不要修改）：

```typescript
setPendingPayments(prev => {
  const current = prev[checkoutItem.id]
  if (!current) return prev

  return {
    ...prev,
    [checkoutItem.id]: {
      ...current,
      shortage: remainingShortage,
    },
  }
})
```

### 10. 测试真实性
