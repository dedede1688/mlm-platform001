- 只允许读取文件、检查差异、运行只读验证命令，以及写本任务结果文件。

## 三、复审目标

确认 005 号收益作废功能和 006 号风险修复合并后满足：

```text
earningsAvailable -= amount
earningsVoided += amount
```

同时确认：

- 不会扣成负数。
- 通知失败不会让已成功的资金操作返回失败。
- 并发下流水和日志快照准确。
- 非法金额全部拦截。
- 自动化测试真实覆盖资金写入，而不是只靠 mock（模拟结果）让断言通过。

## 四、强制检查项


注意：未跟踪的新测试文件不会出现在普通 `git diff --name-only` 中，必须结合 `git status --short` 检查。

确认业务和测试改动只有预期的 6 个文件。

以下历史遗留内容不得算入本轮业务范围，也不得建议提交：

- `vercel_ls.txt`。
- 其他临时输出文件。

### 2. P0 通知失败隔离

检查 `notifyEarningsVoid`（收益作废通知）是否满足：

- `prisma.user.findUnique` 位于完整 `try/catch`（异常保护）中。
- `notificationBatch.create` 位于异常保护中。
- `sendInApp` 位于异常保护中。
- 不存在未等待的异步调用。
- 方法发生任何异常时都不向路由抛错。

检查路由层是否有第二道保护：

```typescript
await OrderNotificationService.notifyEarningsVoid(...).catch(...)
```

必须确认：

```text
资金事务成功 + 通知方法主动抛错
→ 接口仍返回 200
→ success = true
```

如果仍可能返回 500，必须列为 P0 并判定不通过。

### 3. 金额校验

确认路由使用：

```typescript
Number.isFinite(amount)
```

并拒绝：

- 字符串。
- `null`。
- `NaN`（非数字）。
- `Infinity`（无穷大）。
- `-Infinity`（负无穷大）。
- 0。
- 负数作废金额。

检查测试时要注明：`JSON.stringify(NaN)` 和 `JSON.stringify(Infinity)` 会变成 `null`。测试名称写“NaN”或“Infinity”并不等于请求真正传到了该特殊数值，因此必须同时依靠源码检查确认 `Number.isFinite` 已存在。

### 4. 资金原子性

确认真实代码仍使用：

```typescript
updateMany({
  where: {
    id,
    earningsAvailable: { gte: amount },
  },
  data: {
    earningsAvailable: { decrement: amount },
    earningsVoided: { increment: amount },
  },
})
```

并确认：

- `count === 0` 时不写流水、不写日志、不发通知。
- 不使用普通 `update` 替代条件更新。
- 不使用 `$queryRaw` 或 `$queryRawUnsafe`。
- 不修改 `balance`。
- 不修改 `frozenBalance`。
- 不修改 `consumeBalance`。
- 不修改 `earningsFrozen`。
- 不修改 `earningsPending`。

### 5. 并发真实快照

确认 `updateMany` 成功后重新查询数据库更新后的用户值。

确认本次操作前的收益值由真实后值反推：

```typescript
oldEarningsAvailable = updated.earningsAvailable + amount
oldEarningsVoided = updated.earningsVoided - amount
```

确认：

- `BalanceRecord`（资金流水）使用数据库真实后值。
- `format4FieldDelta`（四字段变化说明）的前后值准确。
- `OperationLog.oldValue` 使用反推的真实前值。
- `OperationLog.newValue` 使用数据库真实后值。
- 通知参数使用数据库真实后值。

### 6. 资金流水和操作日志

确认资金流水：

- `type = earnings_void`。
- `amount` 为正数。
- `sourceType`、`sourceId` 可以追踪管理员。
- 原因和字段变化说明完整。
- 与资金更新处于同一事务。

确认操作日志：

- 记录管理员和目标用户。
- `oldValue` 有 `earningsAvailable` 和 `earningsVoided`。
- `newValue` 有 `earningsAvailable` 和 `earningsVoided`。
- `logOperation`（操作日志方法）已有内部异常保护，不要把已解决的问题误报为阻塞项。

## 七、结论口径

出现以下任一情况，必须判定不通过：

- 通知失败仍可能让资金成功接口返回失败。
- 资金更新不是原子条件更新。
- 可用收益可能扣成负数。
- 没有同时减少可用收益和增加作废收益。
- 修改了余额、冻结余额、消费余额、冻结收益或待结算收益。
- 并发下流水或日志快照确认错误且会影响审计真实性。
- 使用原始 SQL 或修改数据库结构。

如果业务代码安全、测试通过，但测试没有直接断言关键 `updateMany` 写入参数，可以判定有条件通过，并要求补强测试后再提交。

只有代码逻辑、异常隔离、资金快照和关键测试断言全部成立，才建议进入模板同步、提交和部署阶段。
