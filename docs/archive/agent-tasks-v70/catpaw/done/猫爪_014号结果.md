## 4. `handleBuyNow` 的状态隔离逻辑说明

```typescript
// v014: 商品切换状态隔离 —— 使用 pendingOrder.cartItemId 判断是否同一商品
const isSamePendingItem = pendingOrder?.cartItemId === item.id
if (!isSamePendingItem) {
  // 从商品 A 切换到商品 B：清空 A 的缺口
  setShortage(0)
}
// 无论打开哪个商品：先关闭旧的收益转余额弹窗，避免悬空
setShowEarningsTransfer(false)

setCheckoutItem(item)
```

### 状态流验证

| 场景 | `isSamePendingItem` | `setShortage(0)` | `setShowEarningsTransfer(false)` | 效果 |
|------|---------------------|-------------------|-----------------------------------|------|
| 重新打开同一待支付商品 A | `true` | 不调用 | 调用 | 保留原 `shortage`，复用订单 A |
| 从商品 A 切换到商品 B | `false` | 调用 | 调用 | 清空 A 的缺口，B 不显示旧缺口 |
| 首次打开商品（无 pendingOrder） | `false` | 调用 | 调用 | `shortage` 本来就是 0，无副作用 |

关键点：使用 `pendingOrder?.cartItemId` 而非 `checkoutItem?.id`，因为关闭弹窗后 `checkoutItem` 已为 `null`，但 `pendingOrder` 仍保留。

## 5. `handleCheckoutClose` 为什么保留 `pendingOrder` 和 `shortage`

```typescript
const handleCheckoutClose = () => {
  if (pendingOrder) {
    toast.info('订单已创建，可在我的订单中继续支付')
  }
  // v014: 关闭收益转余额弹窗，但保留 pendingOrder 和 shortage
  setShowEarningsTransfer(false)
  setCheckoutItem(null)
}
```

### 设计理由

- **保留 `pendingOrder`**：已创建的待支付订单在数据库中仍然存在。如果清除 `pendingOrder`，用户重新打开同一商品时会创建第二张订单，违反"复用原待支付订单"的业务规则。
- **保留 `shortage`**：用户重新打开同一商品时应看到原来的余额缺口，而不是突然消失。如果清除 `shortage`，余额不足提示区和按钮文案会错误地回到初始状态。
- **关闭 `showEarningsTransfer`**：收益转余额弹窗是附属弹窗，关闭结算弹窗时必须一起关闭，避免悬空不可操作的弹窗。
- **不调用取消订单接口**：订单保留在数据库中，用户可从"我的订单"继续支付。

### 4 条断言

| # | 断言 | 说明 |
|---|------|------|
| 1 | `block` 不包含 `verify-payment` | 回调函数不会调用支付验证接口 |
| 2 | `block` 不包含 `handleCheckoutConfirm` | 回调函数不会触发下单/支付流程 |
| 3 | `block` 包含 `fetchUserInfo` | 回调函数仍刷新用户资金信息 |
| 4 | `block` 包含 `余额已补足，请再次确认支付` | 回调函数保留提示用户手动确认的消息 |

### 替换的旧测试

旧测试（第 4 组）用 `<EarningsTransferModal` JSX 标记后 500 字符窗口检查 `onSuccess` 区域不含 `verify-payment`。这种方式只能检查 JSX 属性赋值区域，无法约束 `handleEarningsTransferSuccess` 函数体的实际内容。新测试直接提取函数体，精确约束回调行为。
