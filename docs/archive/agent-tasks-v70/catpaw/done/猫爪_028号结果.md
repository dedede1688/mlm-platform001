## 关键设计决策

1. **方案 A（预校验）**：在调用 `POST /api/orders` 前先调 `calculatePendingShortage(finalPrice, user.balance)`，null 时阻止创建订单 → 防止"订单创建成功但缺口计算失败"的 P0 场景
2. **hasCompleteLockedShipping 统一变量**：将 3 处重复的 `lockedShipping?.recipientName.trim() && ...` 抽为单一布尔变量，handleConfirm/JSX/disabled 三处统一引用 → 空白字符串字段不再被 truthy 检查误判为"有值"
3. **orderId 不丢失**：`if (!orderId)` 块之后不再有 return null 丢弃有效 orderId；`initialShortage` 使用预校验值兜底，不再硬编码 0
