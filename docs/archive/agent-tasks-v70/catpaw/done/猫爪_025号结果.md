## 修复详情

### P0-1：待支付订单重新确认时，被已隐藏且已清空的收货字段拦截

**根因**：`handleConfirm` 无条件校验 `recipientName`/`recipientPhone`/`addressPca` state，但待支付模式下这些字段已隐藏且关闭弹窗时被清空。

**修复**（`CheckoutDialog.tsx:192-249`）：
- 待支付模式（`hasPendingOrder && lockedShipping`）：校验 `lockedShipping` 三字段完整性，不校验可编辑 state
- 非待支付模式：保持原有校验逻辑
- `shippingSnapshot` 计算：`hasPendingOrder && lockedShipping ? lockedShipping : buildEditableShippingSnapshot()`
- `lockedShipping` 不完整时：toast 错误"收货信息不完整，请联系客服"，不退回可编辑表单

**红灯证据**：6 个 P0-1 测试全部失败（`handleConfirm` 不含 `lockedShipping` 引用）
**绿灯证据**：13/13 测试通过

### P0-2：订单详情公开白名单遗漏 `paymentVerified` 字段

**根因**：`getOrderDetail` select 缺少 `paymentVerified: true`，GET 路由 `publicOrder` 缺少 `paymentVerified: order.paymentVerified`。

**修复**：
- `order.service.ts:244`：select 增加 `paymentVerified: true`
- `route.ts:55`：publicOrder 增加 `paymentVerified: order.paymentVerified`
- `false` 值也正确返回（不省略）

**红灯证据**：3 个 P0-2 测试失败
**绿灯证据**：28/28 测试通过

### P1-1：`restorePendingPayment` 资金读取失败时伪装恢复成功

**根因**：`latestUser ? calculatePendingShortage(...) : 0`，fetchUser 失败时 shortage=0 + setRestoreStatus('restored')，伪装恢复成功并覆盖原 sessionStorage。

**修复**（`page.tsx:341-345`）：
- `fetchUser` 返回 null 时：进入 `validation_error`，保留原 sessionStorage
- 设置可读错误文案"资金信息读取失败，请重新验证"
- 不调用 `persistPendingPayment` 覆盖原会话

**红灯证据**：4 个 P1-1 测试失败
**绿灯证据**：27/27 测试通过

### P1-2：`calculatePendingShortage` 缺少 NaN/Infinity/负数校验

**根因**：函数只做 `Math.max(0, payAmount - balance)`，NaN/Infinity/负数输入产生不可预测结果。

**修复**（`pending-payment-session.ts:89-93`）：
- 返回类型改为 `number | null`
- `!Number.isFinite(payAmount) || !Number.isFinite(balance)` → 返回 `null`
- `payAmount < 0 || balance < 0` → 返回 `null`
- 合法输入 → `Math.max(0, payAmount - balance)`
- 所有 3 个调用方增加 `?? 0` 兜底

**红灯证据**：9 个 P1-2 测试失败
**绿灯证据**：36/36 测试通过
