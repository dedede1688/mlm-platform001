## 四、是否新增或补齐 `earnings_void`

**补齐。** `earnings_void` 类型在 `VALID_TYPES` 中已存在，但原逻辑只增 `earningsVoided` 不减 `earningsAvailable`（等于只记账不扣款）。本次补齐为双字段联动：
- `earningsAvailable -= amount`（减少可用收益）
- `earningsVoided += amount`（增加作废收益）

## 五、前端是否新增「作废收益」选项

**是。** 资金调整下拉框现有 4 个选项：
1. 余额
2. 冻结余额
3. 可用收益（增加）
4. 作废收益 ← 新增

## 六、`earningsAvailable` 是否减少

**是。** 使用 Prisma `updateMany` + `{ decrement: amount }` 原子操作减少。

## 七、`earningsVoided` 是否增加

**是。** 使用 Prisma `updateMany` + `{ increment: amount }` 原子操作增加。

## 八、可用收益不足是否拦截

**是。** 双重拦截：
1. **前端拦截**：`amount > detailUser.earningsAvailable` 时提示「可用收益不足，当前仅剩 ¥xx.xx」
2. **后端并发保护**：`updateMany({ where: { id, earningsAvailable: { gte: amount } } })`，`count === 0` 时抛 `可用收益不足`

## 二十、技术实现说明

### 后端核心逻辑（balance/route.ts）

```typescript
// earnings_void 特殊处理：减可用收益 + 增作废收益
if (adjustType === 'earnings_void') {
  // 并发保护：只有 earningsAvailable >= amount 才能更新
  const voidResult = await tx.user.updateMany({
    where: { id, earningsAvailable: { gte: amount } },
    data: {
      earningsAvailable: { decrement: amount },
      earningsVoided: { increment: amount },
    },
  })
  if (voidResult.count === 0) throw new Error('可用收益不足')
  // ... 写 BalanceRecord、返回结果
}
```

### 为什么不复用 `balance_change` 通知模板

1. `balance_change` 模板在 seed.ts 中不存在（是预存死代码），所有 `notifyBalanceChange` 调用实际上都在静默跳过
2. 任务要求通知内容包含「剩余可用收益」「累计作废收益」，`balance_change` 模板变量不匹配
3. 新建专用 `earnings_voided` 模板更清晰，variables 匹配业务语义

### 前端大额确认弹窗

作废收益 ≥ 1000 元时弹二次确认，显示：
- 作废金额
- 当前可用收益
- 作废后可用收益
- 作废后累计作废
