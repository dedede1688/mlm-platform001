## 4. 待支付订单编号如何保存和复用

### 保存机制

```typescript
interface PendingCartOrder {
  cartItemId: string
  orderId: string
}

const [pendingOrder, setPendingOrder] = useState<PendingCartOrder | null>(null)
```

在 `handleCheckoutConfirm` 中，创建订单成功后**立即**调用 `setPendingOrder({ cartItemId: checkoutItem.id, orderId })`，此调用在 `verify-payment` fetch 之前执行。

### 复用机制

```typescript
if (pendingOrder && pendingOrder.cartItemId === checkoutItem.id) {
  orderId = pendingOrder.orderId  // 复用，跳过 POST /api/orders
} else {
  // 调用 POST /api/orders 创建新订单
  // 创建成功后 setPendingOrder(...)
}
```

同一个 `cartItemId` 再次确认时，直接使用已保存的 `orderId`，跳过创建订单接口。

### 清除机制

支付成功后 `setPendingOrder(null)` 清除状态。

## 5. 如何保证第二次确认不重复创建订单

- `pendingOrder` 状态保存了 `cartItemId → orderId` 映射。
- `handleCheckoutConfirm` 入口处检查 `pendingOrder?.cartItemId === checkoutItem.id`，匹配则跳过 `POST /api/orders`。
- 支付密码错误、余额不足、网络失败都不会清除 `pendingOrder`，下一次确认仍然复用同一 `orderId`。
- 只有支付成功后才清除 `pendingOrder`。

## 6. 如何识别 INSUFFICIENT_BALANCE

```typescript
if (verifyErr.code === 'INSUFFICIENT_BALANCE' || (verifyErr.error && verifyErr.error.includes('余额不足'))) {
  // 余额不足处理
}
```

- 优先检查结构化错误码 `data.code === 'INSUFFICIENT_BALANCE'`。
- 兼容旧后端：错误文字包含"余额不足"。
- `shortage` 优先使用接口返回 `data.data?.shortage`，未返回时使用 `Math.max(0, payAmount - currentBalance)`。

initialAmount={Math.min(shortage, userInfo?.earningsAvailable ?? 0)}

在 `EarningsTransferModal` 的 `initialAmount` prop 中传入 `Math.min(shortage, earningsAvailable)`，即余额缺口和可用收益取较小值。弹窗内部已有 `Math.min(initialAmount, earningsAvailable)` 兜底逻辑。

## 8. 收益不足或为 0 时的界面行为

### 可用收益 > 0 但不足以补齐

- 打开 `EarningsTransferModal` 收益转余额弹窗。
- 转入成功后计算 `remainingShortage = Math.max(0, shortage - transferredAmount)`。
- 剩余差额 > 0 时提示："收益已转入，支付仍差 ¥X，请充值后继续支付"。
- `CheckoutDialog` 内余额不足提示区显示"购物余额不足，还差 ¥X" + "去充值"按钮。

### 可用收益为 0

- 不打开 `EarningsTransferModal`（条件守卫 `earningsAvailable > 0`）。
- `CheckoutDialog` 内余额不足提示区显示"购物余额不足，还差 ¥X" + "去充值"按钮。
- 提示用户"购物余额不足，请充值后继续支付"。

## 9. 转入成功后是否自动支付

**否**。`handleEarningsTransferSuccess` 回调只做两件事：

1. 调用 `fetchUserInfo(token)` 刷新用户资金信息。
2. 计算 `remainingShortage` 并更新 `shortage` 状态 + toast 提示。

回调内不调用 `verify-payment`，不自动支付。用户必须再次主动点击"重新确认支付"按钮。

## 10. 关闭弹窗时如何处理已创建订单

```typescript
const handleCheckoutClose = () => {
  if (pendingOrder) {
    toast.info('订单已创建，可在我的订单中继续支付')
  }
  setCheckoutItem(null)
}
```

- 显示提示"订单已创建，可在我的订单中继续支付"。
- 不调用取消订单接口。
- 不删除订单。
- 不自动清理数据库中的订单。
- **限制说明**：关闭弹窗后 `pendingOrder` 状态保留在内存中，但如果用户离开购物车页面再回来，`pendingOrder` 会丢失（React 状态不持久化）。用户应从"我的订单"继续支付。

## 11. 支付成功路径是否保持不变

是。支付成功后仍执行：

1. 删除对应购物车项目（`DELETE /api/cart/${checkoutItem.id}`）。
2. 刷新购物车本地状态（`setCartItems` 过滤）。
3. 刷新积分 Map（`setPointsMap` 删除对应 key）。
4. 清除待支付订单状态（`setPendingOrder(null)` + `setShortage(0)`）。
5. 刷新用户资金信息（`fetchUserInfo(token)`）。
6. 显示购买成功（`toast.success('购买成功！')`）。
7. 跳转订单详情页（`router.push('/dashboard/orders/${orderId}')`）。
8. 推荐奖未解锁提示保持不变。

## 19. P0/P1/P2 风险

### P0 - 无

### P1 - 需关注

1. **pendingOrder 状态不持久化**：用户关闭弹窗后如果离开购物车页面再回来，`pendingOrder` 状态会丢失，已创建的待支付订单无法在购物车内找回。用户需从"我的订单"继续支付。已在关闭弹窗时 toast 提示。

### P2 - 低风险

1. **shortage 状态在 CheckoutDialog 和 EarningsTransferModal 之间同步**：收益转入成功后 `shortage` 状态更新，CheckoutDialog 的余额不足提示区会同步更新。如果用户在收益转入弹窗打开期间快速操作 CheckoutDialog，可能有短暂状态不一致，但不影响业务正确性。
2. **浏览器验证未完成**：因无测试账号凭据，完整浏览器验证链路未走通。已用源码契约测试 + 类型检查 + lint 三重验证替代。
