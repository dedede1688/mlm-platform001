## 快速发布步骤

### 1. 确认范围

运行：

```powershell
git diff --name-only
git diff --check
```

业务差异必须恰好是上述 15 个文件。

### 2. 唯一一次完整测试与覆盖率门槛

为避免普通测试、覆盖率测试重复运行，本地只运行一次：

```powershell
npx vitest run --coverage --coverage.thresholds.statements=70
```

必须记录：

- 测试文件数量。
- 测试数量。
- 失败数量。
- statements（语句覆盖率）。
- 70% 覆盖率门槛是否通过。

测试或覆盖率失败立即停止，不得提交。

### 3. 唯一一次构建

运行：

```powershell
npx next build
```

构建失败立即停止，不得提交。

### 4. 精确暂存

逐个暂存上述 15 个文件，禁止使用 `git add .`。

暂存后运行：

```powershell
git diff --cached --name-only
git diff --cached --stat
git status --short
```

确认暂存区恰好 15 个文件，并且没有任务文件、旧文档或 `vercel_ls.txt`。

### 5. 提交

提交说明：

```text
chore: fix lint blocking errors
```

### 6. 推送和铁律 1 验证

运行：

```powershell
git push origin main
git log origin/main --oneline -1
```

必须确认本地 HEAD 与 origin/main 相同。

### 7. 远程自动检查

等待 GitHub Actions（自动检查）的 `test`（测试任务）完成。

必须确认以下步骤都实际执行且成功，不能只看总状态：

```text
Lint（代码规范检查）
TypeScript type check（类型检查）
Run tests（普通测试）
Run tests with coverage（覆盖率测试）
Check coverage threshold（覆盖率门槛）
```

任何一步失败或被跳过，都不能报告“远程检查全部通过”。

如果 `gh` 命令不可用，可以通过 GitHub 网页或 GitHub API（接口）只读查询检查状态。

### 8. 部署平台验证

确认 Vercel（部署平台）：

- 最新提交编号等于本次提交编号。
- Production（生产环境）。
- Ready（已就绪）或 success（成功）。

### 9. 最终状态

运行：

```powershell
git status --short
```

15 个目标文件必须从状态列表消失。旧文档和本地任务文件保持未提交状态。
