## 三、缺口一：右上角关闭按钮测试不得假通过

当前测试存在类似写法：

```typescript
if (closeBtnMatch) {
  expect(closeBtnMatch[0]).toContain('closeDetailModal')
}
```

匹配失败时会跳过断言并假通过。

必须改成：

```typescript
expect(closeBtnMatch).not.toBeNull()

```

## 六、缺口四：真实验证登录密码哈希不会泄露

当前 `detail.test.ts` 的 `FULL_USER` 不包含 `passwordHash`，所以“响应不含 passwordHash”可能只是因为测试数据本身没有该字段。

### 6.1 测试数据必须包含敏感字段

为该安全场景准备数据库原始记录，必须真实包含：

```typescript
passwordHash: 'login_password_hash_should_never_leave_db'
paymentPasswordHash: 'payment_password_hash_should_never_leave_db'
```

### 6.2 mock 必须模拟 Prisma select

不要让 `mockResolvedValue` 无视 `select` 后把完整数据库记录直接返回给路由。为该场景实现一个最小的顶层字段投影辅助函数，例如：

```typescript
function applyTopLevelSelect(record: Record<string, unknown>, select: Record<string, unknown>) {
  return Object.fromEntries(
    Object.keys(select).map(key => [key, record[key]])
  )
}
```

嵌套关系字段可以直接取测试记录中的现有值，本任务只要求真实模拟顶层字段白名单。

该安全测试必须形成完整证据链：

1. 原始数据库测试记录真实包含 `passwordHash`。
2. 路由调用 Prisma 时的 `select` 不包含 `passwordHash`。
3. mock 根据真实 `select` 投影返回对象，因此路由实际拿不到 `passwordHash`。
4. 最终响应序列化文本不包含 `passwordHash`。
5. 原始记录包含 `paymentPasswordHash`。
6. `select` 包含 `paymentPasswordHash`，供路由计算状态。
7. 最终响应不包含 `paymentPasswordHash`，但包含正确的 `hasPaymentPassword: true`。

不要通过让路由查询完整用户再手动删除登录密码哈希来修测试。业务代码的 Prisma 字段白名单才是第一道安全边界。
