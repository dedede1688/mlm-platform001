## 五、正确的数据模型

不要只把 `orderId` 改成字典；余额缺口也必须随商品保存，否则 A→B→A 时虽然能找回订单 A，但会丢失 A 的缺口。

将单值状态：

```typescript
interface PendingCartOrder {
  cartItemId: string
  orderId: string
}

const [pendingOrder, setPendingOrder] = useState<PendingCartOrder | null>(null)
const [shortage, setShortage] = useState(0)
```

改成按 `cartItemId`（购物车项目编号）索引的状态表。推荐结构：

```typescript
interface PendingCartPayment {
  orderId: string
  shortage: number
}

const [pendingPayments, setPendingPayments] = useState<Record<string, PendingCartPayment>>({})
```

每个购物车项目分别保存：

- 已创建的待支付订单编号 `orderId`。
- 该订单最近一次支付返回的余额缺口 `shortage`。

禁止继续保留单个 `pendingOrder` + 单个 `shortage` 作为真实业务状态，避免两套状态不同步。

## 七、实现要求

### 7.1 当前商品待支付记录

在组件渲染期间，根据 `checkoutItem?.id` 取得当前记录，例如：

```typescript
const currentPendingPayment = checkoutItem
  ? pendingPayments[checkoutItem.id]
  : undefined

const currentShortage = currentPendingPayment?.shortage ?? 0
```

命名可调整，但语义必须一致。

### 7.2 打开商品结算弹窗

`handleBuyNow` 不再清除全局 `shortage`，因为缺口已按商品隔离。

每次打开商品时仍必须：

```typescript
setShowEarningsTransfer(false)
setCheckoutItem(item)
```

打开 A 自动读取 A 的记录；打开 B 自动读取 B 的记录。

### 7.3 创建并保存订单

`handleCheckoutConfirm` 必须先读取：

```typescript
pendingPayments[checkoutItem.id]
```

如果存在，复用其中的 `orderId`，跳过 `POST /api/orders`。

如果不存在，创建订单成功后使用函数式更新，保留其他商品记录：

```typescript
setPendingPayments(prev => ({
  ...prev,
  [checkoutItem.id]: {
    orderId,
    shortage: 0,
  },
}))
```

### 7.4 余额不足

收到 `INSUFFICIENT_BALANCE`（余额不足错误码）后，只更新当前商品记录：

```typescript
setPendingPayments(prev => ({
  ...prev,
  [checkoutItem.id]: {
    orderId,
    shortage: actualShortage,
  },
}))
```

必须保留其他商品记录。

### 7.5 收益转入成功

`handleEarningsTransferSuccess` 只更新当前 `checkoutItem.id` 对应的 `shortage`。

- 补足：该商品 `shortage` 更新为 0。
- 未补足：该商品 `shortage` 更新为剩余差额。
- 不自动调用支付。
- 不修改其他商品的订单编号或缺口。

函数开始时若 `checkoutItem` 为空，应安全返回或只刷新资金，不能误改其他商品。

### 7.6 支付成功

支付成功后只删除当前商品对应记录：

```typescript
setPendingPayments(prev => {
  const next = { ...prev }
  delete next[checkoutItem.id]
  return next
})
```

不能使用：

```typescript
setPendingPayments({})
```

否则会删除其他商品的待支付订单关联。

### 7.7 关闭弹窗

`handleCheckoutClose`：

- 关闭收益转余额弹窗。
- 当前商品已有待支付记录时显示“订单已创建，可在我的订单中继续支付”。
- 不删除任何 `pendingPayments` 记录。
- 不修改任何商品的 `shortage`。

### 7.8 页面参数

传给 `CheckoutDialog` 和 `EarningsTransferModal` 的数据必须来自当前商品记录：

- `shortage={currentShortage}`。
- `hasPendingOrder={!!currentPendingPayment}`。
- `initialAmount={Math.min(currentShortage, earningsAvailable)}`。

## 八、业务状态必须满足

```text
1. 打开 A，创建 orderA，余额不足 shortageA。
2. 关闭 A，不删除 A 记录。
3. 打开 B，不显示 shortageA。
4. 创建 orderB，余额不足 shortageB，不覆盖 A。
5. 关闭 B，再打开 A，读取 orderA 和 shortageA。
6. 再次确认 A，跳过创建订单接口，直接验证 orderA。
7. A 支付成功，只删除 A；B 的 orderB/shortageB 仍保留。
8. 收益转入成功后不自动支付。
```
