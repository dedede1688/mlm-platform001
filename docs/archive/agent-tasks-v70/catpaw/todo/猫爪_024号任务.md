## 四、业务目标

### P0：关闭订单详情敏感字段泄露

现有 `OrderService.getOrderDetail()` 使用 `user: true`，可能使订单详情接口返回完整用户对象和密码哈希。

必须：

- 服务层改用显式 `select（字段白名单）`；
- 不查询完整 `user`；
- GET 接口使用显式公开响应映射；
- 响应禁止包含 `userId`、`user`、`passwordHash`、`paymentPasswordHash`；
- 保持用户订单详情页和支付页所需字段完整。

### P1：商品详情余额不足引导

必须：

- 商品详情余额不足时识别 `INSUFFICIENT_BALANCE（余额不足错误码）`；
- 有收益时打开收益转余额弹窗；
- 无收益或收益不足时显示缺口和“去充值”；
- 收益转入后不自动支付；
- 用户必须手动重新输入支付密码确认。

### P1：待支付订单恢复和防重复

必须：

- 首次创建后立即保存原订单编号；
- 再次支付复用原订单，不再创建；
- 充值往返使用 `sessionStorage（当前标签页会话存储）`；
- 恢复时用服务端订单接口验证所有权、状态、商品和数量；
- 恢复原订单的 `pointsUsed（已使用积分）` 和 `payAmount（应付金额）`；
- 用原订单 `payAmount` 减最新购物余额重新计算缺口，不得用页面实时 `finalPrice（当前计算价）` 覆盖原订单金额；
- 网络或 500 时保留记录、禁止新建、提供“重新验证”；
- 明确无效时清除记录；
- 支付成功后清除记录。

### P1：共用结算弹窗锁定收货信息

必须：

- 待支付订单存在时隐藏可编辑地址表单；
- 显示首次提交或服务端恢复的收货人、手机号和完整地址只读摘要；
- 支付密码仍可输入；
- 购物车获得相同锁定行为，但不得修改购物车待支付字典。
