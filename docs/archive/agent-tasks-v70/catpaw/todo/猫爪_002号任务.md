## 2. 背景

收益转余额第 1 包和第 2 包已经上线：

- 第 1 包：后端已支持 `POST /api/user/earnings-transfer`（用户收益转余额接口）。
- 第 2 包：用户端已有 `EarningsTransferModal`（收益转余额弹窗）。
- 第 3 包第 1 步：支付接口 `POST /api/orders/[id]/verify-payment`（订单支付验证接口）在余额不足时，已经返回结构化错误：

```json
{
  "success": false,
  "code": "INSUFFICIENT_BALANCE",
  "error": "可用余额不足",
  "data": {
    "balance": 200,
    "payAmount": 500,
    "shortage": 300
  }
}
```

本任务只做第 3 包第 2 步：在一个真实支付入口做试点。试点入口选定为：

```text
src/app/dashboard/orders/[id]/page.tsx
```

也就是用户端订单详情页的“去支付”按钮。

## 3. 目标

完成以下目标：

1. 订单详情页支付时，如果接口返回 `code: "INSUFFICIENT_BALANCE"`（余额不足错误码），不要只弹普通错误提示。
2. 页面应弹出 `EarningsTransferModal`（收益转余额弹窗），引导用户从可用收益转入购物余额。
3. 订单详情页需要读取用户当前资金信息：`balance`（购物余额）和 `earningsAvailable`（可用收益）。
4. 转入成功后刷新用户资金信息，并提示用户重新点击支付。
5. 本任务只做订单详情页试点，不扩展到订单列表、购物车、商品详情页、支付页。

## 7. 实现要求

### 7.1 订单详情页读取资金信息

在 `src/app/dashboard/orders/[id]/page.tsx` 中读取 `/api/users/me`，拿到：

- `balance`（购物余额）
- `earningsAvailable`（可用收益）

读取方式要和项目现有用户端页面保持一致，必须带：

```text
Authorization: Bearer token
```

说明：`Authorization` 是鉴权请求头，`Bearer token` 是登录令牌格式。

### 7.2 捕获余额不足错误

订单详情页调用：

```text
POST /api/orders/[id]/verify-payment
```

如果响应里有：

```text
code === "INSUFFICIENT_BALANCE"
```

则打开收益转余额弹窗。

不要只用字符串判断“可用余额不足”。可以保留兜底字符串判断，但主判断必须使用 `code`（错误码）。

### 7.3 弹窗金额建议

如果需要增强 `EarningsTransferModal`（收益转余额弹窗），可以增加可选参数：

```typescript
initialAmount?: number
```

用途：打开弹窗时，默认填入建议转入金额。

建议金额规则：

```text
建议转入金额 = min(shortage, earningsAvailable)
```

其中：

- `shortage` 是支付接口返回的差额。
- `earningsAvailable` 是当前可用收益。

如果不增加 `initialAmount`，也可以让用户手动输入金额，但结果报告必须说明。

### 7.4 转入成功后的行为

转入成功后：

1. 刷新订单详情页里的用户资金信息。
2. 不要自动再次支付。
3. 给用户提示：“余额已补充，请重新点击支付”。

不自动再次支付的原因：支付涉及支付密码，不能绕过用户确认。

### 7.5 用户收益不足的情况

如果 `earningsAvailable <= 0`，仍然可以提示余额不足，但按钮或弹窗里要明确显示“暂无可转入收益”。

不要伪造余额，不要让前端绕过后端余额校验。
