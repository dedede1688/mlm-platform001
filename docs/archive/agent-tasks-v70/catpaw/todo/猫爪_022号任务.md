本任务只修复这个前端校验和错误提示缺口，必须采用 TDD（测试驱动开发）：先补失败测试并记录红灯，再修改最小业务代码让测试通过。

## 二、根因和目标

### 2.1 已确认根因

当前后端接口：

```text
POST /api/admin/users/[id]/payment-password/reset
```

已经使用：

```typescript
const actualSuffix = targetUser.phone.slice(-4)
if (phoneSuffix !== actualSuffix) {
  return errorResponse('手机号后 4 位不匹配', 400)
}
```

所以后端没有真正重置错误号码对应的用户。

前端 `handleResetPaymentPassword` 目前只验证：

```typescript
/^\d{4}$/.test(payPwdResetSuffix)
```

没有验证输入值是否等于：

```typescript
detailUser.phone.slice(-4)
```

同时，接口失败返回字段为 `error`，但前端主要读取 `data.message`，导致真实错误可能退化为通用“重置失败”。

### 2.2 修复目标

输入错误后四位时必须：

1. 页面立即显示“手机号后 4 位不匹配，请核对后重试”。
2. “重置支付密码”按钮保持禁用。
3. 不打开二次确认弹窗。
4. 不调用重置接口。
5. 不进入“处理中...”状态。
6. 后端校验继续保留，作为最终安全边界。

输入正确后四位时，原有重置流程必须保持正常。

## 五、最小业务修复

只修改：

```text
src/app/admin/users/page.tsx
```

### 5.1 真实后四位校验

使用当前详情会员手机号：

```typescript
const actualPhoneSuffix = detailUser.phone.slice(-4)
```

输入值建议统一使用：

```typescript
const normalizedSuffix = payPwdResetSuffix.trim()
```

校验规则：

```typescript
/^\d{4}$/.test(normalizedSuffix) && normalizedSuffix === actualPhoneSuffix
```

### 5.2 `handleResetPaymentPassword` 必须拦截

保留现有权限、原因长度、4 位数字格式校验，并在打开确认弹窗前新增相等校验：

```typescript
if (normalizedSuffix !== actualPhoneSuffix) {
  showMessage('error', '手机号后 4 位不匹配，请核对后重试')
  return
}
```

错误后四位不得执行：

```typescript
setShowPayPwdConfirm(true)
```

### 5.3 `doResetPaymentPassword` 再次校验

二次确认执行函数必须在：

```typescript
setSavingPayPwdReset(true)
fetch(...)
```

之前再次比较真实后四位。

如果不匹配：

- 显示同一条错误提示。
- 关闭或保持关闭二次确认弹窗。
- 立即 return。
- 不进入处理中状态。
- 不调用接口。

这是前端第二道保护；后端现有校验是第三道保护，不得删除。

### 5.4 输入框即时提示

当输入长度为 4 且与真实后四位不匹配时，在输入框下显示红色提示：

```text
手机号后 4 位不匹配，请核对后重试
```

输入正确或未满 4 位时，不显示该“不匹配”提示。

### 5.5 按钮禁用条件

“重置支付密码”按钮只有同时满足以下条件才能启用：

1. 当前不是处理中。
2. 原因去空格后不少于 5 个字。
3. 后四位是 4 位数字。
4. 后四位等于 `detailUser.phone.slice(-4)`。

不能只依赖点击处理函数拦截；按钮状态也必须即时反馈。

### 5.6 后端错误信息正确显示

接口失败分支改为：

```typescript
showMessage('error', data.error || data.message || '重置失败')
```

确保即使后端兜底拦截，也能向管理员显示“手机号后 4 位不匹配”，而不是笼统的“重置失败”。
