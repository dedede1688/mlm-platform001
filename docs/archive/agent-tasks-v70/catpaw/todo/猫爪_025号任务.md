## 六、P0-1：修复待支付订单隐藏字段拦截

### 6.1 现有问题

`CheckoutDialog.tsx` 在 `hasPendingOrder=true` 时显示锁定收货摘要，但 `handleConfirm` 仍校验：

- `recipientName`
- `recipientPhone`
- `addressPca`
- `detailAddress`

这些可编辑 state 在关闭弹窗后会被定时清空。没有地址簿默认地址的用户再次打开待支付订单，只能看到锁定摘要，却会被提示“请输入收件人姓名”或“请选择完整地址”，无法继续支付。

### 6.2 必须实现

1. `hasPendingOrder=true` 时，必须使用 `lockedShipping` 作为本次确认的收货数据。
2. 待支付模式不得校验已经隐藏的可编辑 state。
3. 待支付模式下：
   - `lockedShipping` 存在且三个字段有效，允许只输入支付密码重新确认；
   - `lockedShipping` 缺失或字段不完整，禁止提交，并显示只读错误提示；
   - 不得退回可编辑地址表单；
   - 不得创建第二个订单。
4. 非待支付模式保持原有地址校验、地址簿保存和创建订单逻辑不变。
5. `shippingSnapshot` 和传给 `onConfirm` 的收货字段，待支付模式必须来自 `lockedShipping`，首次下单模式来自可编辑 state。
6. `onSaveAddress` 在待支付模式仍不得调用。
7. 支付密码输入框必须继续显示。

推荐最小实现结构：

```typescript
const lockedShipping = pendingOrderShipping ?? submittedShipping

const handleConfirm = async () => {
  const shippingSnapshot = hasPendingOrder
    ? lockedShipping
    : buildEditableShippingSnapshot()

  if (!shippingSnapshot) {
    // 待支付模式显示锁定信息缺失；首次模式维持原有逐项提示
    return
  }

  // 支付密码校验保持不变
  // onConfirm 使用 shippingSnapshot
}
```

不要求机械照抄，但必须满足上述行为。

### 6.3 必须先补的测试

强化 `__tests__/components/checkout-pending-order-lock.test.ts`，至少证明：

1. `hasPendingOrder=true` 时渲染分支无条件隐藏可编辑地址表单，不得是 `hasPendingOrder && lockedShipping ? ... : editable`。
2. `handleConfirm` 在待支付分支使用 `lockedShipping`。
3. 待支付分支不检查 `recipientName.trim()`、`addressPca`、`detailAddress` 等隐藏 state。
4. `lockedShipping` 缺失时不渲染可编辑表单，并禁止调用 `onConfirm`。
5. 无默认地址、首次提交、关闭、重新打开、只输入支付密码的契约路径可通过。
6. 非待支付模式仍保留原地址校验。

测试必须提取真实函数块或真实 JSX 分支；提取失败必须显式失败。禁止只用：

## 九、P1-2：计算缺口的有限数字校验

### 9.1 必须实现

`calculatePendingShortage(payAmount, balance)` 必须在函数内部防御非法数字。

建议返回类型改为：

```typescript
number | null
```

规则：

- `payAmount` 或 `balance` 不是有限数字：返回 `null`；
- `payAmount < 0` 或 `balance < 0`：返回 `null`；
- 合法输入：返回 `Math.max(0, payAmount - balance)`。

如果采用抛错方式，也必须让所有调用方明确处理，不能让 `NaN` 进入 state 或会话。优先采用 `number | null`，改动更可控。

所有调用方必须显式处理 `null`：

- 恢复流程进入 `validation_error`；
- 支付余额不足流程优先使用接口返回的合法 `shortage`，兜底计算非法时进入错误状态；
- 收益转入成功后的重新计算不得写入非法缺口。
