# 猫爪_013号任务：购物车余额不足时引导收益转余额

## 一、任务性质

## 二、业务背景

当前购物车确认下单流程是：

1. 先调用 `POST /api/orders` 创建订单。
2. 再调用 `POST /api/orders/[id]/verify-payment` 验证支付密码并支付。
3. 如果支付失败，页面返回 `null`，用户再次点击确认时会重新创建订单。

这会导致两个问题：

1. 余额不足后没有收益转余额入口。
2. 支付失败后再次确认可能重复创建订单。

订单详情页已经实现余额不足时打开 `EarningsTransferModal`（收益转余额弹窗），本任务要把同样能力接入购物车确认下单，但保持购物车弹窗内连续操作。

## 三、胡子老师已确认的业务规则

1. 第一次点击确认时创建订单，并立即保存订单编号。
2. 后续再次确认时，如果已有当前购物车项目对应的待支付订单编号，必须跳过 `POST /api/orders`，直接支付原订单。
3. 余额不足时识别结构化错误码 `INSUFFICIENT_BALANCE`。
4. 有可用收益时打开收益转余额弹窗。
5. 默认转入金额为：

```text
min(shortage, earningsAvailable)
```

即余额缺口和可用收益取较小值。

6. 收益转入成功后不允许自动支付，必须由用户再次主动点击确认。
7. 可用收益不足以补齐时，显示剩余差额，并提供“去充值”入口。
8. 可用收益为 0 时，不打开不可操作的收益转余额弹窗，直接显示余额缺口和“去充值”入口。
9. 用户关闭确认订单弹窗时，如果订单已经创建，保留待支付订单，并提示：

```text
订单已创建，可在我的订单中继续支付
```

10. 不自动取消订单，不删除待支付订单，不自动复用支付密码发起支付。

## 六、实现要求

### 1. 扩充购物车用户资金信息

在 `src/app/cart/page.tsx` 的 `UserInfo` 中增加：

```typescript
balance: number
earningsAvailable: number
```

继续复用现有 `GET /api/users/me`，不要新增接口。

读取时必须使用数值兜底：

```typescript
balance: Number(data.data.balance) || 0
earningsAvailable: Number(data.data.earningsAvailable) || 0
```

### 2. 保存并复用待支付订单

购物车页面新增当前结算项目对应的待支付订单状态。推荐结构：

```typescript
interface PendingCartOrder {
  cartItemId: string
  orderId: string
}
```

必须满足：

- 第一次创建订单成功后，在调用 `verify-payment` 之前就保存 `orderId`。
- 同一个 `cartItemId` 再次确认时，直接复用该 `orderId`。
- 只有没有可复用订单编号时才调用 `POST /api/orders`。
- 支付密码错误、余额不足或网络失败都不能导致下一次确认重新创建订单。
- 支付成功后清除待支付订单状态。

### 3. 识别余额不足

支付失败时按以下优先级识别：

```typescript
data.code === 'INSUFFICIENT_BALANCE'
```

允许保留旧后端兼容兜底：错误文字包含“余额不足”。

识别后保存：

```typescript
shortage
```

优先使用接口返回：

```typescript
data.data?.shortage
```

只有接口未返回时才使用：

```typescript
Math.max(0, payAmount - currentBalance)
```

### 4. 打开收益转余额弹窗

当 `earningsAvailable > 0` 时：

- 保持 `CheckoutDialog`（确认订单弹窗）打开。
- 在其上方打开 `EarningsTransferModal`（收益转余额弹窗）。
- `initialAmount` 必须是 `Math.min(shortage, earningsAvailable)`。
- 弹窗层级必须高于确认订单弹窗。

当 `earningsAvailable <= 0` 时：

- 不打开收益转余额弹窗。
- 在确认订单弹窗内显示余额不足提示和“去充值”按钮。

### 5. 收益转入成功后的处理

将 `EarningsTransferModal` 的成功回调调整为返回实际转入金额：

```typescript
onSuccess: (amount: number) => void
```

成功时调用：

```typescript
onSuccess(numAmount)
```

现有余额页、提现页和订单详情页使用无参数回调，必须继续兼容且通过类型检查。

购物车收到转入金额后：

1. 重新读取 `/api/users/me` 资金数据。
2. 计算：

```typescript
remainingShortage = Math.max(0, shortage - transferredAmount)
```

3. 如果剩余差额为 0，提示：

```text
余额已补足，请再次确认支付
```

4. 如果仍有差额，提示：

```text
收益已转入，支付仍差 ¥X，请充值后继续支付
```

5. 不得在成功回调中调用 `verify-payment`，不得自动支付。

### 6. 确认订单弹窗提示区

为 `CheckoutDialog` 增加可选参数，名称可按现有代码风格确定，但必须支持：

- 当前剩余差额。
- 当前可用收益。
- 是否已经存在待支付订单。
- 打开收益转余额弹窗的回调。
- 去充值的回调。

弹窗内余额不足提示区必须展示：

```text
购物余额不足，还差 ¥X
```

行为要求：

- 有收益时显示“收益转入余额”。
- 无收益或转入后仍不足时显示“去充值”。
- 已存在待支付订单时，底部主按钮文字改为“重新确认支付”。
- 未创建订单时仍显示“确认下单”。

“去充值”跳转路径：

```text
/dashboard/recharge
```

跳转前保留数据库中的待支付订单。

### 7. 关闭弹窗

关闭确认订单弹窗时：

- 如果当前已有待支付订单，显示提示：

```text
订单已创建，可在我的订单中继续支付
```

- 不调用取消订单接口。
- 不删除订单。
- 不自动清理数据库中的订单。
- 可以关闭本地弹窗状态，但结果报告必须说明重新进入购物车不会自动找回该订单，用户应从“我的订单”继续支付。

### 8. 支付成功路径保持不变

支付成功后仍然执行：

- 删除对应购物车项目。
- 刷新购物车本地状态。
- 刷新用户资金信息。
- 显示购买成功。
- 跳转订单详情页。

不得破坏推荐奖励未解锁提示。
