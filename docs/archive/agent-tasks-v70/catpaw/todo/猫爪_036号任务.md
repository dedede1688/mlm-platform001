## 一、任务目标

完成“退款申请凭证优化”的第一包，只处理用户提交退款申请的链路：

1. `质量问题`：至少上传 1 张图片，补充说明可选。
2. `商品损坏`：至少上传 1 张图片，补充说明可选。
3. `未按约定时间发货`：图片可选，补充说明可选。
4. `其他`：图片可选，补充说明必填，去除首尾空格后不得为空。
5. 前端提示和后端校验必须一致。
6. 后端必须作为最终安全边界，不能只依赖前端按钮禁用。
7. 被拒绝后再次申请时，必须使用全新的空表单，不继承上一次的图片或说明。

本包完成后停止，等待小 M 036 号只读复审。后台审核历史展示属于下一包，不在本任务范围内。

## 六、业务规则

### 6.1 共享校验模块

新建 `src/lib/refunds/refund-validation.ts`，提供以下导出：

```typescript
REFUND_REASONS_REQUIRING_IMAGES
refundReasonRequiresImages(reason)
refundReasonRequiresDescription(reason)
validateRefundApplication(input)
```

校验输入至少包含：

```typescript
{
  reason: string
  description?: unknown
  images?: unknown
}
```

校验结果必须能区分成功和失败。成功结果必须返回清理后的：

- `reason`
- `description`：去除首尾空格
- `images`：合法字符串数组

失败结果必须返回稳定、可测试的中文错误信息。

### 6.2 图片规则

- `images` 必须是数组；未提供时按空数组处理。
- 每项必须是非空字符串。
- 最多 5 张。
- `质量问题`、`商品损坏` 至少 1 张。
- 不允许因为前端漏传或恶意直调接口而绕过后端校验。

建议错误文案：

```text
质量问题或商品损坏至少需要上传 1 张凭证图片
退款凭证图片最多上传 5 张
退款凭证图片格式不正确
选择“其他”时必须填写补充说明
```

如现有项目已有同义文案，可保持一致，但前后端和测试必须使用同一套文案。

### 6.3 后端退款接口

在 `POST /api/orders/[id]/refund` 中：

1. 保留现有鉴权、订单归属、状态和防重复退款逻辑。
2. 在写入退款申请前调用共享校验模块。
3. 校验失败返回 400。
4. 校验失败时不得创建退款申请。
5. 校验失败时不得发送通知。
6. 校验成功时只写入清理后的说明和图片。
7. 不修改现有退款金额、状态机或通知成功链路。

### 6.4 用户退款申请页面

在 `src/app/dashboard/orders/[id]/refund/page.tsx` 中：

1. 提交前调用同一共享校验模块。
2. 根据退款原因动态显示必填提示。
3. `质量问题`、`商品损坏`明确显示“至少上传 1 张图片”。
4. `其他`明确显示“补充说明必填”。
5. 提交按钮的禁用状态和提交函数内二次校验必须一致。
6. 切换退款原因时保留用户在本次申请中已经上传的图片，不静默删除。
7. 页面初始化和重新申请时不得从历史退款记录回填图片或说明。
8. 图片上限继续保持 5 张。
9. 保留当前上传、删除、预览和提交成功流程。

## 七、测试驱动开发顺序

严格执行“先写失败测试，再写实现”的流程。

### Task 1：共享校验模块

1. 新建 `__tests__/lib/refund-validation.test.ts`。
2. 覆盖以下场景：
   - 质量问题无图片失败。
   - 商品损坏无图片失败。
   - 未按约定时间发货无图片成功。
   - 其他无说明失败。
   - 其他只有空格说明失败。
   - 合法说明被 trim。
   - 非数组图片失败。
   - 图片数组含空字符串失败。
   - 超过 5 张失败。
   - 合法 1 至 5 张成功。
3. 先运行测试，记录因模块不存在或逻辑缺失产生的失败证据。
4. 再实现共享校验模块，直到测试通过。

### Task 2：后端接口校验

1. 先扩展 `__tests__/api/orders/refund-route.test.ts`。
2. 至少新增：
   - 质量问题无图片返回 400，且不写数据库、不发通知。
   - 商品损坏无图片返回 400，且不写数据库、不发通知。
   - 其他无说明返回 400。
   - 未按约定时间发货无图片可以成功。
   - 质量问题有 1 张图片可以成功。
   - 图片格式错误返回 400。
   - 超过 5 张返回 400。
3. 保留并继续通过 pending / approved 防重复、rejected 允许再次申请等已有测试。
4. 对按调用次数排队的模拟数据使用 `vi.resetAllMocks()` 或等效方式，避免前一测试残留的 `mockResolvedValueOnce` 污染后一测试。
5. 先运行测试记录失败证据，再接入共享校验模块。

### Task 3：用户页面动态校验

1. 新建 `__tests__/components/refund-application-page.test.ts`。
2. 测试必须限制在真实函数块或 JSX 结构块内，使用括号/花括号平衡提取；禁止固定 200/300/500 字符窗口。
3. 禁止 `if (match)` 静默跳过。
4. 禁止 `A || B` 宽松断言。
5. 覆盖：
   - 页面导入共享校验模块。
   - 提交函数在 fetch 前调用共享校验。
   - 校验失败时 return，不发请求。
   - 质量问题/商品损坏显示图片必填提示。
   - 其他显示说明必填提示。
   - 重新申请不读取历史退款图片或说明作为默认值。
   - 切换原因不会清空本次已选图片。
6. 先运行测试记录失败证据，再修改页面。
