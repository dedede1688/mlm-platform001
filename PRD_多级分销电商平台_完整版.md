## 1. 项目概述

### 1.1 项目信息

| 项目 | 内容 |
|-----|------|
| **项目名称** | 多级分销电商平台 |
| **项目代号** | MLC-Platform |
| **目标用户** | 个人消费者、会员、经销商、各级管理者 |
| **核心价值** | 通过会员制和推荐奖励机制，实现用户裂变和销售增长 |

### 1.2 项目背景

本平台采用**三三复制**的多级分销模式，结合会员等级制度和多重奖励机制，激励用户主动推广和复购。核心特点：

- **双轨关系**：直推关系（永久绑定）+ 安置关系（动态滑落）
- **多重奖励**：推荐奖 + 品牌管理奖 + 分红奖
- **积分锁仓**：长期激励，促进持续消费
- **等级晋升**：清晰的升级路径，持续激励

### 1.3 核心指标

| 指标 | 目标 |
|-----|------|
| 商品数量 | 10个（可扩展）|
| 会员等级 | 8级 |
| 分销层级 | 3层合规 |
| 奖励比例 | 推荐奖20% + 品牌管理奖20% + 分红5% |

### 2.2 会员系统

#### 2.2.1 会员等级

| 等级 | 身份 | 升级条件 | 核心权益 |
|-----|------|---------|---------|
| 0 | 游客 | 无需注册 | 浏览、购物（零售价）|
| 1 | 会员 | 注册即可 | 购物（会员价）、推荐奖20% |
| 2 | 经销商 | 购买指定产品10件 | 同上 + 品牌管理奖 + 购物积分 |
| 3 | 主任 | 直推3个经销商 或 直推销售额达5万 | 同上 + 分红资格 |
| 4 | 经理 | 直推6个经销商 或 直推销售额达10万 | 同上 |
| 5 | 总监 | 直推9个经销商 或 直推销售额达20万 | 同上 |
| 6 | 总裁 | 直推15个经销商 或 直推销售额达50万 | 同上 |
| 7 | 董事 | 直推30个经销商 或 直推销售额达100万 | 同上 |

#### 2.2.2 升级条件说明

**经销商升级**：
- 累计购买指定产品10件
- 达到后自动升级
- 获得一次性积分奖励

**主任及以上升级**：
- 条件A：直推经销商数量达标
- 条件B：直推销售额达标
- 满足任一条件即可升级

#### 2.2.3 等级降级

- 系统支持手动降级（后台操作）
- 降级后分红按新等级计算
- 已发放奖励不追回

### 2.3 推荐关系系统（三三复制）

#### 2.3.1 核心概念

| 概念 | 说明 |
|-----|------|
| **直推关系** | 直接推荐的用户，永久绑定，不可变更 |
| **安置关系** | 滑落后的位置关系，动态调整 |
| **三三复制** | 每个会员最多3个直推位置，第4个自动滑落 |

#### 2.3.2 结构示例

```
                    A
        ┌───────────┼───────────┐
        B           C           D
    ┌───┼───┐   ┌───┼───┐   ┌───┼───┐
    E   F   G   H   I   J   K   L   M
```

**滑落规则**：
- A推E → 滑到B-1
- A推F → 滑到B-2
- A推G → 滑到B-3
- A推H → 滑到C-1
- 以此类推（从上到下，从左到右）

#### 2.3.3 关系表设计

| 用户 | 直推人 | 安置位置 | 层级 |
|-----|--------|---------|------|
| B | A | A-1 | 1 |
| C | A | A-2 | 1 |
| D | A | A-3 | 1 |
| E | A | B-1 | 2 |
| F | A | B-2 | 2 |
| G | A | B-3 | 2 |
| H | A | C-1 | 2 |

### 2.4 奖励系统

#### 2.4.1 推荐奖（20%）

| 项目 | 说明 |
|-----|------|
| **获得资格** | 所有会员（注册即可）|
| **计算基数** | 被推荐人消费金额 |
| **归属规则** | 永远给直推人，与滑落位置无关 |
| **发放时机** | 订单支付成功后实时到账 |
| **示例** | E消费500元，A（直推人）获得100元 |

#### 2.4.2 品牌管理奖（20%）

| 项目 | 说明 |
|-----|------|
| **获得资格** | 经销商及以上级别 |
| **计算基数** | 安置链下级消费金额 |
| **归属规则** | 按安置链向上轮询，给最近的经销商 |
| **层级解锁** | 直推经销商数决定可拿层数 |
| **发放时机** | 订单支付成功后实时到账 |

**层级解锁规则**：

| 直推经销商数 | 解锁层数 |
|------------|---------|
| 0 | 0层（无品牌管理奖）|
| 1 | 2层 |
| 2 | 4层 |
| 3+ | 10层（封顶）|

**轮询示例**：

```
E消费第1单500元：
- 推荐奖：A获得100元（直推人）
- 品牌管理奖：B获得100元（安置上一层，经销商）

E消费第2单500元：
- 推荐奖：A获得100元
- 品牌管理奖：A获得100元（安置上二层）

E消费第3单500元：
- 推荐奖：A获得100元
- 品牌管理奖：A的上一层获得100元（安置上三层）
```

> **实现说明（与上方规则的差异记录）**：当前代码实现（`src/lib/services/reward.service.ts` 的 `computeMaxLayers`）与上方 PRD 表格存在差异，属已知产品/实现偏差，按决策暂不修改代码，仅在此记录：
> - PRD 规定"会员无资格"，但 `computeMaxLayers` 对会员（MEMBER）返回 10 层；实际发放时 `findBrandBonusRecipients` 会跳过会员本身（仅向上轮询经销商），故会员不会实际获得品牌管理奖，但遍历深度按 10 层计算。
> - PRD 规定"直推经销商数 0 → 0 层"，但 `computeMaxLayers` 对 0 个直推经销商的经销商返回 2 层。
> - 后续如需严格对齐 PRD，再行调整 `computeMaxLayers` 并补充对应测试。

#### 2.4.3 分红奖（5%）

| 项目 | 说明 |
|-----|------|
| **获得资格** | 主任及以上级别 |
| **计算基数** | 每笔订单金额的5% |
| **分红池** | 当日所有订单的5%总和 |
| **分配规则** | 按等级均分，高级别可叠加低级别分红 |
| **结算周期** | 按天快照等级，每周统一发放 |
| **明细展示** | 每笔订单的分红金额实时显示 |

**分红计算示例**：

```
某日销售1个500元产品，分红池=25元

当前等级人数：
- 董事：1人
- 总裁：3人
- 总监：2人
- 经理：4人
- 主任：6人

计算：
- 董事分红：25÷1=25元/人
- 总裁分红：25÷(1+3)=6.25元/人
- 总监分红：25÷(1+3+2)=4.17元/人
- 经理分红：25÷(1+3+2+4)=2.5元/人
- 主任分红：25÷(1+3+2+4+6)=1.56元/人

累积分红（高级别拿低级别）：
- 董事：25+6.25+4.17+2.5+1.56=39.48元
- 总裁：6.25+4.17+2.5+1.56=14.48元
- 以此类推
```

### 2.5 积分系统

#### 2.5.1 积分获得

| 项目 | 说明 |
|-----|------|
| **获得条件** | 累计购买指定产品10件 |
| **获得方式** | 一次性获得（10件×单价）|
| **示例** | 10件×500元=5000积分 |

#### 2.5.2 积分解锁

| 项目 | 说明 |
|-----|------|
| **解锁比例** | 每天解锁1%（后台可配置）|
| **解锁天数** | 连续100天释放完毕（后台可配置）|
| **解锁计算** | 第N天解锁=总积分×1% |
| **示例** | 5000积分，每天解锁50积分 |

**解锁时间表**：

| 天数 | 累计解锁 | 剩余锁定 |
|-----|---------|---------|
| 第1天 | 50 | 4950 |
| 第10天 | 500 | 4500 |
| 第50天 | 2500 | 2500 |
| 第100天 | 5000 | 0 |

#### 2.5.3 积分使用

| 项目 | 说明 |
|-----|------|
| **兑换比例** | 1积分=1元 |
| **使用方式** | 购物时按比例抵扣 |
| **抵扣比例** | 每个商品可设置（如50%）|
| **示例** | 100元商品，最多可用50积分 |

#### 2.5.4 积分转赠

| 项目 | 说明 |
|-----|------|
| **转赠条件** | 仅限已解锁积分 |
| **转赠对象** | 仅限会员（游客不可接收）|
| **转赠方式** | 输入对方会员ID或手机号 |
| **到账时间** | 实时到账 |

#### 2.5.5 积分有效期

| 项目 | 说明 |
|-----|------|
| **锁定期间** | 未解锁积分，退货时全部作废 |
| **解锁后** | 永久有效 |
| **退货处理** | 中途退货，全部积分作废，已使用积分从余额扣除 |

### 2.6 订单系统

#### 2.6.1 订单流程


#### 2.6.3 支付方式

| 项目 | 说明 |
|-----|------|
| **当前阶段** | 模拟支付（测试环境）|

| 项目 | 说明 |

## 6. 边界情况处理

### 6.1 推荐关系边界

#### 场景1：用户A推荐B，B滑落到C下，A再推荐D

**处理**：D继续按滑落规则找空位（从上到下，从左到右）

#### 场景2：用户A推荐B，B未激活（未购买升级产品）

**处理**：B仍占据位置，但不享受经销商权益（无品牌管理奖）

#### 场景3：安置链上无经销商

**处理**：品牌管理奖沉淀到平台

### 6.2 奖励计算边界

#### 场景1：分红时某等级人数为0

**处理**：该等级分红沉淀到平台

#### 场景2：品牌管理奖轮询超过10层

**处理**：超过10层后停止轮询，剩余部分沉淀到平台

#### 场景3：用户同时满足多个等级条件

**处理**：按最高等级计算

### 6.3 等级晋升边界

#### 场景1：用户一天内从会员升级到董事

**处理**：允许，分红按最终等级计算

#### 场景2：用户降级后再次升级

**处理**：重新计算，分红按新等级计算

### 6.4 积分边界

#### 场景1：解锁期间退货

**处理**：全部积分作废，已使用积分从余额扣除

#### 场景2：转赠后对方未注册

**处理**：转赠失败，积分退回

#### 场景3：转赠后对方注销

**处理**：转赠失败，积分退回

### 6.5 订单边界

#### 场景1：支付超时

**处理**：订单自动取消，库存释放

#### 场景2：退款时奖励已发放

**处理**：从余额扣除或标记负数，下期结算抵扣

#### 场景3：用户注销

**处理**：保留数据，标记为已注销，不可登录

#### 场景4：并发支付

**处理**：数据库事务 + 乐观锁，防止重复发放奖励

### 6.6 财务边界

#### 场景1：余额为负数

**处理**：不允许提现，需充值或等待新奖励抵扣

#### 场景2：分红池为0

**处理**：当日无分红，不报错

#### 场景3：提现申请超过余额

**处理**：前端校验，不允许提交

## 8. 风险与合规

### 8.1 法律风险

#### 风险1：传销合规

**问题**：多级分销模式可能涉及传销

**应对措施**：
- 分销层级控制在3层以内
- 奖励必须来自实际销售，非入门费
- 不收取高额入门费
- 上线前咨询法律意见

#### 风险2：税务合规

**问题**：奖励发放涉及个人所得税

**应对措施**：
- 记录所有奖励发放明细
- 年度提供用户收益报告
- 提醒用户自行申报个税

### 8.2 技术风险

#### 风险1：并发问题

**问题**：高并发时奖励计算可能出错

**应对措施**：
- 数据库事务 + 乐观锁
- 关键操作加分布式锁
- 异步处理奖励发放

#### 风险2：数据一致性

**问题**：分布式系统数据不一致

**应对措施**：
- 关键数据使用数据库事务
- 定期对账检查
- 异常数据自动修复

#### 风险3：系统安全

**问题**：SQL注入、XSS攻击

**应对措施**：
- 使用ORM防止SQL注入
- 输入验证和转义
- HTTPS加密传输

### 8.3 业务风险

#### 风险1：早期用户收益过高

**问题**：顶层用户收益远高于底层

**应对措施**：
- 设置分红上限
- 动态调整分红比例
- 引入新人激励机制

#### 风险2：用户流失

**问题**：底层用户看不到收益，流失率高

**应对措施**：
- 降低会员门槛
- 增加非分销收益（如积分返利）
- 优化用户体验

### 9.2 计算公式汇总

```
推荐奖 = 订单金额 × 20%

品牌管理奖 = 订单金额 × 20%
（按安置链向上轮询，给最近的经销商）

分红池 = 当日所有订单金额 × 5%

某等级分红 = 分红池 × 5% ÷ 该等级人数
（高级别可叠加低级别分红）

每日解锁积分 = 总积分 × 1%
```
